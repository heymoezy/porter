# Examples — wireframe-specialist

## Representative requests

### 1. New onboarding flow
“Wireframe a self-serve onboarding flow for a B2B analytics product that connects data sources and gets users to a first dashboard.”

Expected shape:
- identify the user goal, sequence, and key setup risks
- map screens from entry through activation
- show branching for skipped setup, failed connection, and success
- annotate what product questions are still unresolved

### 2. Workflow comparison
“Should approvals happen in a table, a side panel, or a step-by-step flow?”

Expected shape:
- define the decision criteria
- show 2–3 structural options in low-fi terms
- compare density, speed, learnability, and mobile behavior
- recommend one option with explicit tradeoffs

### 3. Dashboard state map
“List the wireframes and states we need for an operations dashboard before design starts.”

Expected shape:
- screen inventory
- state inventory for empty, loading, populated, no-access, and error states
- notes on navigation, filters, and detail drill-downs
- missing requirements called out directly

### 4. Feature handoff skeleton
“Turn this rough product spec into a wireflow engineering can react to.”

Expected shape:
- user task summary
- step-by-step wireflow with decisions and transitions
- per-screen purpose and primary action
- concise annotations for validation, dependencies, and bounce-back paths

## Output expectation
A strong answer should:
- reduce ambiguity about what exists, what happens next, and what edge cases matter
- stay low-noise and appropriately low fidelity
- expose structural gaps before visual design or implementation begins
- give the next team a clean blueprint rather than a vague concept sketch
