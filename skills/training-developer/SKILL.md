# training-developer

Design training that changes performance on the job.

This skill owns learning-design judgment: how to define the learner, identify the behavior gap, sequence instruction, build realistic practice, measure competence, and reinforce retention after the event.

## Scope

Use this skill for:
- onboarding programs and role-based enablement
- instructor-led workshops and cohort-based training
- self-paced modules and blended learning paths
- facilitator guides, lesson plans, and slide/story flow
- labs, simulations, exercises, and practice scenarios
- assessment design: quizzes, performance checks, rubrics
- train-the-trainer packs and certification-readiness paths
- reinforcement plans: office hours, job aids, refreshers, follow-up practice
- converting SME knowledge into teachable curriculum
- mapping learning objectives to business or operational outcomes

## Do not use this skill for

Do not use this skill for:
- generic explainer content, documentation, or marketing copy with no instructional outcome
- pure HR policy drafting where learning design is not the core problem
- academic course theory when the real ask is operational enablement
- product messaging or sales collateral unless it must become training
- assessment-only requests where psychometric rigor or credential governance is the main issue

## Routing rules

Route to **training-developer** when the main challenge is deciding:
- what people must be able to do after the training
- how to sequence content and practice for durable skill building
- which format fits the audience, constraints, and stakes
- how to assess readiness, not just attendance
- how to turn scattered expertise into a repeatable learning program

Do **not** route here just because the output will be educational.
If the real issue is documentation, tutorials, or change communication, use the skill centered on that artifact.

## Inputs to gather

Before designing the program, identify:
- learner roles, baseline knowledge, and common failure modes
- business goal or performance gap the training must close
- success criteria on the job, not only in the classroom
- delivery model: live, async, blended, cohort, lab, coaching
- duration, budget, facilitator capacity, and tooling constraints
- required depth: awareness, working fluency, certification, or expert practice
- practice environment availability and realism
- assessment stakes: low-risk knowledge check vs high-risk operational signoff
- reinforcement needs after launch

If the request does not define a real performance outcome, say the training goal is underspecified.

## Output expectations

Return outputs such as:
- curriculum maps and learning-path architectures
- session-by-session lesson plans
- facilitator guides and learner agendas
- exercise packs, labs, scenarios, and answer keys
- assessment plans with observable pass criteria
- reinforcement and rollout plans
- stakeholder recommendations on scope, format, and risks

Prefer run-ready training assets over abstract pedagogy.

## Working method

### 1. Start from job performance
Define what success looks like in real work.
Translate vague goals like “understand the platform” into observable behaviors.

### 2. Write measurable learning objectives
Use verbs tied to evidence: diagnose, configure, escalate, explain, troubleshoot, evaluate.
Avoid objectives that only say “know” or “understand.”

### 3. Choose the right training architecture
Match the format to the need:
- awareness -> concise overview, examples, and reference aids
- skill building -> guided practice, feedback, repetition
- high-stakes execution -> simulations, checklists, performance validation

### 4. Sequence for cognitive load
Move from context to demonstration to guided practice to independent performance.
Keep each module focused on one meaningful capability cluster.

### 5. Build practice into the core
Include realistic scenarios, labs, role plays, call reviews, or decision exercises.
Training without practice is usually documentation in disguise.

### 6. Assess transfer, not attendance
Define how readiness will be verified.
Use quizzes only for recall; use performance tasks for operational competence.

### 7. Reinforce after delivery
Add job aids, refreshers, follow-up prompts, coaching loops, or office hours so the skill survives first exposure.

## Heuristics

Prefer:
- fewer, sharper objectives
- role-specific scenarios over generic examples
- practice before polish-heavy content
- facilitator notes that make delivery reliable
- short modules with strong transitions and checkpoints
- assessment criteria tied to visible behavior

Avoid:
- content dumping with no learner action
- overstuffed agendas that leave no practice time
- objectives that cannot be observed or measured
- one-size-fits-all design for very different roles
- quizzes standing in for real competence
- forgetting post-training reinforcement

## Adjacent skill boundaries

- **training-developer** designs learning programs and skill-building systems
- **tutorial-creator** owns single task walkthroughs and step-by-step procedural teaching
- **documentation-writer** owns reference docs, not behavior-change curriculum
- **onboarding-specialist** may own broader experience design; this skill owns the learning architecture within it
- **course-creator** may overlap on educational packaging; use this skill when operational capability and enablement are the real center

## Quick routing examples

Use **training-developer** for:
- building a 30-day onboarding program for support engineers
- designing a Terraform workshop with labs and readiness checks
- turning SME interviews into a sales-enablement certification path
- creating train-the-trainer packs for regional facilitators

Do **not** use **training-developer** for:
- rewriting a setup guide into a quickstart tutorial
- creating policy copy that employees must acknowledge
- drafting long-form documentation with no instructional sequencing

## Quality bar

A strong result should:
- tie training design to a real performance gap
- define measurable learner outcomes
- include realistic practice and usable assessment
- match format and sequencing to the audience and stakes
- account for facilitation, rollout, and reinforcement
- produce assets someone could run with minimal extra interpretation
