# Documentation Writer — Example Output Shapes

Use these as patterns for documentation deliverables.

## Example 1 — README rewrite

**Input:**
Rewrite our repo README so a new engineer can run the project locally in under 15 minutes.

**Good output shape:**
- audience and repo purpose
- prerequisites and environment requirements
- quick-start steps in order
- expected success signals
- first three likely failure points and fixes
- links to deeper docs

## Example 2 — Partner API documentation

**Input:**
Document our webhook endpoints for integration partners.

**Good output shape:**
- endpoint purpose and authentication model
- request schema and example payloads
- response behavior and retries
- idempotency, sequencing, and timeout notes
- error cases and operational limits
- testing or sandbox guidance

## Example 3 — Incident runbook

**Input:**
Create a runbook for failed nightly sync jobs.

**Good output shape:**
- trigger conditions
- immediate containment steps
- diagnostics in priority order
- likely causes and how to confirm them
- recovery steps and rollback path
- escalation criteria and follow-up

## Example 4 — Migration guide

**Input:**
Document how customers should move from v1 to v2 of our API.

**Good output shape:**
- who is affected
- breaking changes summary
- before / after examples
- migration sequence
- validation steps
- rollback guidance and deprecation timeline

## Example 5 — Documentation audit

**Input:**
Audit our internal docs and tell us what to fix first.

**Good output shape:**
- current-state diagnosis
- duplicated, stale, and missing content
- top reader journeys that fail today
- prioritized remediation plan
- suggested information architecture
- ownership and maintenance recommendations
