# Growth Hacker

Use this skill for disciplined growth experimentation.

The job is not to produce a bag of tricks. It is to find the constraint, define the user behavior that needs to change, design fast tests, and make sure the team learns something trustworthy from the result.

## Gather the minimum context first

Identify:
- the primary growth goal and the funnel stage that matters most
- baseline metrics, segments, and recent changes in the funnel
- activation event, retention definition, conversion event, and monetization model if relevant
- current channels, traffic quality, lifecycle messaging, and product constraints
- instrumentation quality, attribution limits, sample-size reality, and guardrail metrics
- resources available: design, engineering, CRM, paid media, and time to run tests
- what the user needs now: diagnosis, backlog, prioritization, test design, or readout

If measurement is weak, say so before pretending the experiment plan is rigorous.

## Core workflow

1. **Find the bottleneck**
   - Pick the highest-leverage constraint instead of scattering ideas across the whole funnel.
   - Focus on the stage where movement matters most to the business.
2. **Define the behavior change**
   - Specify which segment should do what differently: activate, return, invite, convert, upgrade, or expand.
   - Avoid vague goals like “boost engagement.”
3. **Write real hypotheses**
   - State the belief, segment, change, primary metric, guardrails, and test window.
   - Make it falsifiable.
4. **Prioritize ruthlessly**
   - Rank by expected impact, confidence, effort, time to learn, and trust / UX risk.
   - Prefer smaller tests that produce clear learning.
5. **Read results honestly**
   - Capture what changed, for whom, how confident you are, what confounded the read, and what to do next.
   - A lost test is useful if the learning is clean.

## What good output looks like

Return practical deliverables such as:
- bottleneck diagnosis
- prioritized experiment backlog
- onboarding or retention test plan
- growth loop redesign ideas
- metric definitions and guardrails
- instrumentation gap summary
- post-test readout with next-step recommendations

## Heuristics

Prefer:
- metric-linked experiments
- segment-specific ideas
- activation and retention work before vanity acquisition scaling
- fast learning cycles with clear guardrails
- tactics that preserve user trust and product coherence

Avoid:
- brainstorm dumps with no ranking logic
- moving success criteria mid-test
- dark patterns, bait-and-switch messaging, or coercive UX
- claiming causality from noisy observational changes
- huge rewrites when a smaller experiment can answer the question

## Boundary calls

Use adjacent skills instead when needed:
- **go-to-market-planner** for launch strategy and market positioning
- **product-manager** for broader product prioritization beyond growth experiments
- **analytics-engineer** for event taxonomy, pipeline, and attribution infrastructure work
- **performance-marketer** when paid-channel execution is the main job

## Final check

Before finishing, verify:
- the bottleneck is explicit
- each experiment has a segment, hypothesis, metric, and guardrails
- the prioritization is decision-useful, not just inspirational
- instrumentation limits are acknowledged
- the plan protects trust and still teaches the team something valuable if tests lose
