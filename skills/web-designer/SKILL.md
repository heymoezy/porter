# Web Designer

Design pages that feel intentional, readable, and worth acting on. Solve hierarchy, flow, and trust before surface polish.

## Focus

Use this skill to:
- design new marketing, product, commerce, dashboard, or editorial web pages
- critique existing layouts and diagnose why they feel weak, cluttered, or low-trust
- restructure section order, page flow, and CTA emphasis
- tune typography, spacing, density, contrast, and rhythm for the web
- define responsive behavior across desktop, tablet, and mobile
- give buildable direction to designers or frontend engineers

## Do not use this skill for

- frontend implementation or code changes
- brand identity creation without concrete web-page application
- illustration, icon, or bespoke asset production
- accessibility-only review when compliance or assistive-use quality is the main goal
- vague taste validation with no user task, audience, or business objective

## Inputs to gather

Capture as many of these as possible:
- page type and primary job: convert, explain, compare, retain, or help complete a task
- target audience and their likely scan behavior
- primary CTA, secondary CTA, and actions that should be de-emphasized
- required content blocks, proof elements, and constraints
- existing brand system, screenshots, wireframes, or components
- traffic context: paid landing page, organic homepage, logged-in product surface, etc.
- device realities: mobile-first, dense desktop workflow, localization growth, accessibility constraints

## Output expectations

Return practical artifacts such as:
- page diagnosis with the biggest hierarchy failures first
- recommended section order and narrative flow
- layout direction for hero, proof, feature, pricing, dashboard, or content sections
- responsive behavior notes by breakpoint
- simplification and de-emphasis guidance
- concise rationale tied to trust, clarity, conversion, or task success

## Working method

### 1. Define the page promise

State what the page must help the user do or believe:
- understand the offer
- trust the product
- choose an option
- complete a workflow
- find information fast

If the page promise is fuzzy, the layout will drift.

### 2. Fix information hierarchy before styling

Decide:
- what gets first attention
- what earns second attention
- what can wait
- what should be removed entirely

Use size, spacing, contrast, grouping, and section order to make priority obvious. Do not rely on decoration to rescue weak hierarchy.

### 3. Design section sequencing as an argument

Each section should answer one question at the right moment:
- what is this?
- why should I care?
- can I trust it?
- how does it work?
- what should I do next?

Marketing pages need narrative momentum. Product pages need task momentum. Dashboards need decision momentum.

### 4. Control density deliberately

Dense pages are not automatically bad; unmanaged density is. For each region, decide whether it should feel:
- focused and spacious
- comparative and scannable
- data-dense but structured
- supportive and secondary

Use chunking, labels, visual grouping, and progressive disclosure before adding more chrome.

### 5. Design the responsive version, not a shrunk version

For every major section, specify:
- what stays visible early on mobile
- what stacks, collapses, or converts to cards
- what can be trimmed or deferred
- how comparison tables, nav, filters, and side panels adapt

If the mobile structure is an afterthought, the design is incomplete.

### 6. Use proof and action with discipline

Make proof earn trust:
- customer logos, metrics, screenshots, testimonials, certifications, or examples
- place proof near hesitation points, not randomly

Make CTAs disciplined:
- one primary action per region
- clear visual dominance for the main action
- secondary actions that support without competing

### 7. Hand off buildable guidance

Recommendations should be specific enough that someone can implement them without guessing:
- section order
- component pattern choices
- spacing/hierarchy rules
- state and content edge cases
- breakpoint behavior
- what to simplify, merge, or remove

## Quality bar

A strong result should:
- improve page comprehension within a quick scan
- make the primary action and narrative path obvious
- reduce clutter and equal-weight noise
- account for real content lengths and responsive behavior
- feel cohesive, modern, and restrained rather than trendy-for-trendy's-sake
- give a designer or engineer a clear next move
