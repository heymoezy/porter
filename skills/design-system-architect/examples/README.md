# Design System Architect — Example Output Shapes

Use these shapes to keep outputs structured and actionable.

## Example 1 — Greenfield design system

**Input:**
We are launching a multi-product B2B suite and need a shared design system.

**Good output shape:**
- product constraints and cross-surface needs
- system principles
- token architecture
- foundation component list
- governance model
- rollout phases and metrics

## Example 2 — Inconsistent existing UI

**Input:**
Our app has five button styles, three modal patterns, and duplicated forms.

**Good output shape:**
- inconsistency diagnosis
- merge/consolidation decisions
- token cleanup priorities
- component contract changes
- migration plan by leverage
- deprecation notes

## Example 3 — New component admission

**Input:**
Should we add a comparison-card component to the system?

**Good output shape:**
- repeated problem test
- existing primitives/patterns review
- admission decision
- if approved: contract, variants, states, owners
- if rejected: composition guidance and local-scope recommendation

## Example 4 — Governance reset

**Input:**
Design and engineering keep adding variants without coordination. Fix the process.

**Good output shape:**
- current governance failures
- proposed review workflow
- evidence required for additions
- versioning and deprecation rules
- documentation ownership
- audit cadence and KPIs
