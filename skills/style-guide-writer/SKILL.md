# Style Guide Writer

Turn “our voice” from vibes into enforceable writing decisions.

## Use this skill to

- create or rewrite voice-and-tone guides
- define editorial standards, grammar preferences, and usage rules
- build terminology systems, word lists, naming guidance, and banned-language lists
- write microcopy standards for product surfaces and lifecycle messaging
- audit existing guides for vagueness, contradiction, and adoption failure
- design style systems reviewers can use consistently across teams and channels

## Do not use this skill for

- one-off copywriting or campaign execution
- brand strategy work when language standards are not yet defined enough
- translation / localization operations
- legal drafting style manuals as the core deliverable

## Inputs to gather

Collect the material that reveals the writing system:

- brand positioning or messaging baseline
- examples of content that feels on-brand and off-brand
- audience segments, channels, and risk-sensitive moments
- required terms, prohibited claims, compliance or legal constraints
- writer workflow, reviewer workflow, and where inconsistency currently appears
- whether the guide must cover marketing, product, support, docs, exec comms, or all of them

If reference material is weak, say so. Do not fake brand specificity from adjectives alone.

## Deliverables

Return one or more of these:

- full style guide or condensed working guide
- voice and tone framework with flex rules by context
- editorial standards and usage rules
- terminology / naming / capitalization guide
- microcopy standards for UI states and lifecycle moments
- examples library with before/after rewrites

## Working method

### 1. Define what the guide must control

Find the failure mode: tone drift, terminology chaos, overlong UX copy, inconsistent punctuation, compliance slippage, or channel mismatch. The guide should fix a real pattern, not perform brand theater.

### 2. Separate principles from rules

Principles explain intent. Rules govern execution. Both are necessary. A guide full of principles is vague; a guide full of rules with no rationale becomes brittle and ignored.

### 3. Distinguish constants from flex points

Some standards should stay fixed across all channels; others should flex by context. Make that line explicit so writers know what may change and what may not.

### 4. Use examples as operating logic

Examples teach faster than adjectives. Show preferred wording, banned wording, before/after rewrites, and channel-specific contrasts. Examples should make review decisions obvious.

### 5. Design for adoption

Make the guide scannable, prioritized, and usable in review. If a new writer or editor cannot apply it quickly, the guide is too abstract.

### 6. Remove contradiction and loopholes

Check whether rules conflict, whether examples actually obey the rules, and whether reviewers would make the same call from the guide.

## Heuristics

- Prefer explicit “say / avoid / why” patterns over abstract traits.
- Limit top-level voice principles to a memorable set.
- Include banned patterns, not just positive guidance.
- Separate marketing tone, product microcopy, support language, and high-stakes comms when they genuinely differ.
- Keep terminology governance strict where product meaning or compliance depends on it.
- Favor short rules with strong examples over sprawling prose.
- If two rules conflict in practice, define precedence.

## Failure modes to catch

- the guide describes voice with adjectives but no usable examples
- every channel is forced into one identical tone
- terminology rules ignore product architecture or support realities
- microcopy advice is too broad to review against
- examples contradict the stated principles
- the guide is too long, too vague, or too precious to adopt

## Review questions

- Could a new writer use this without live coaching?
- Can an editor review copy against it consistently?
- Are universal rules separated from channel-specific adaptations?
- Are required terms, banned terms, and naming logic explicit?
- Do examples show real contrast, not cosmetic variation?
- Would following this actually reduce inconsistency?

## Adjacent skill boundaries

- Use **copywriter** when the main task is writing the asset itself.
- Use **brand-designer** or strategy-oriented brand skills when positioning and identity are still unsettled.
- Use **ux-writer** when interface copy execution is the main work rather than system design.
- Use **documentation-writer** when the primary need is technical documentation structure.
- Use **localization-specialist** when multilingual adaptation is the core problem.
- Use **editorial-planner** when governance is less important than content-program planning.

## Quality bar

A strong result is:

- specific enough to enforce
- easy enough to adopt
- rich in examples
- consistent across contexts without flattening them
- distinctive to the brand, product, or organization
