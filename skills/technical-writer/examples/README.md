# Examples — Technical Writer

## Representative requests
- Rewrite our deployment runbook so an on-call engineer can perform a rollback at 2 a.m. without tribal knowledge.
- Turn this rough API spec and ticket history into a clean integration guide with request examples and error handling.
- Draft release notes for a breaking auth change, including migration steps, who is affected, and known risks.
- Explain this event-driven architecture to new backend engineers without hand-wavy diagrams-only prose.

## Output expectations
- Name the audience and task if it is not already obvious.
- Produce the document itself, not a writing plan.
- Include prerequisites, supported paths, examples, validation steps, and caveats.
- Make escalation or rollback paths explicit for operational procedures.

## Good fit signals
- The source material is technical and dense.
- Reader trust depends on precision.
- Missing caveats would cause failure, downtime, or confusion.
- The goal is usable explanation, not persuasion.

## Poor fit signals
- The main need is doc-site restructuring or information architecture.
- The request is marketing-heavy or conversion-heavy.
- The real task is designing the system rather than explaining it.
