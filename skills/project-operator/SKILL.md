# Project Operator

Make the work move. Strong project operation is not performative project management. It is clean ownership, fast blocker visibility, useful meeting outputs, disciplined follow-through, and just enough structure to keep momentum real.

## Scope

Use this skill for:
- execution cadence design
- owner and action tracking
- blocker/risk logging and follow-through
- meeting output conversion into decisions and tasks
- launch coordination and readiness management
- turning a project plan into an operating rhythm

## Do not use this skill for

Do not use this skill for:
- initial strategy or product-shaping work when the problem is still undefined
- abstract architecture planning as the main task
- generic notes with no owners, dates, or decisions
- process-heavy overhead that slows small teams down

## Inputs to gather

Before operationalizing the work, identify:
- current plan or scope baseline
- decision owners and working team
- milestones or delivery dates
- main blockers, dependencies, and escalation paths
- communication surfaces: standup, weekly review, launch room, async updates
- what 'done' means for the next interval

## Output expectations

Return outputs such as:
- delivery operating plan
- action tracker with owners and due dates
- meeting readout with decisions and follow-ups
- blocker log and escalation recommendations
- launch readiness checklist / runbook
- weekly status format that drives action

## Working method

### 1. Start from the current plan, not from a blank process template

Project operation should fit the project size and urgency. Apply the minimum useful structure needed to maintain momentum and accountability.

### 2. Make ownership impossible to misunderstand

Every important item should have:
- single owner
- due date or review date
- status / dependency note
- escalation path if blocked

Shared ownership is often disguised abandonment.

### 3. Convert meetings into movement

A useful readout captures:
- decisions made
- open questions
- actions with owners
- blockers needing escalation
- next checkpoint

If a meeting output cannot change behavior, it was just transcription.

### 4. Track blockers separately from normal tasks

Blockers deserve their own view because they distort timelines and attention. Note:
- blocker description
- impact
- owner
- dependency
- escalation trigger

### 5. Protect cadence quality

Use a rhythm that matches the work:
- daily for urgent delivery coordination
- weekly for milestone tracking
- milestone-based for sparse initiatives

Too much ceremony kills speed. Too little visibility creates surprise.

### 6. End every output with next moves

The output should always answer:
- who does what next
- what is at risk
- what decision is waiting
- when the team regroups

## Adjacent skill boundaries

- **project-architect**: shapes the project structure; this skill runs the structure once defined
- **product-manager**: decides product priorities and scope; this skill keeps execution aligned to those decisions
- **operations-manager**: broader business operations; this skill focuses on project delivery motion
- **program-manager**: may coordinate portfolios; this skill is more hands-on with a project’s day-to-day execution system

## Quality bar

A strong result should:
- create visible ownership and next actions
- surface blockers before they become schedule surprises
- keep updates concise and decision-oriented
- match process weight to project complexity
- improve delivery reliability without adding theater
