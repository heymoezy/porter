# Porter v0.12.70 Draft — Locations & First-Time Mesh Onboarding Plan

Date: 2026-02-26
Status: Draft (queued; do not implement until current Claude run completes)
Owner: Lobster + User review

---

## 1) Context

Current Locations UX assumes Tailscale is already connected and peers are discoverable.
For first-time users, this creates a trust and usability gap: empty device lists with unclear next steps.

This plan defines a comprehensive, implementation-ready Locations upgrade while preserving the current layout the user likes.

---

## 2) Product Goals

1. Preserve existing Locations structure and visual familiarity.
2. Add clear, explicit network readiness states.
3. Provide actionable first-time onboarding when mesh is not ready.
4. Prevent dead-end flows and ambiguous empty states.
5. Keep local filesystem usage available even when mesh is unavailable.

---

## 3) Scope for v0.12.70 (post-Claude)

### A. Devices Header Enhancements
- Add manual **Refresh** button above device list.
- Show compact **Last updated** timestamp (e.g., "Updated 12s ago").
- Show transient in-flight indicator during refresh.

### B. Mesh Readiness State Layer
Add a state badge near Devices header with one of:
- `Mesh Connected`
- `Mesh Not Connected`
- `Mesh Service Down`
- `Mesh Status Unknown`

### C. First-Time Onboarding Empty State (No Tailscale)
When no mesh is detected, replace raw empty list with guided checklist:
1. Install/start Tailscale on this host
2. Authenticate and join tailnet
3. Install/start Tailscale on target device
4. Return and press Refresh

Include primary action: **Open Setup Steps**
Include secondary action: **Continue Local Only**

### D. Local-Only Fallback Mode
If mesh unavailable:
- Keep local node and local mounts fully usable.
- Show info label: `Local-only mode active`.
- Do not block core local browsing workflows.

### E. Error/Status Clarity
Use deterministic status copy:
- No peers found vs daemon down vs auth required must be distinguishable.
- All failures include next-step action text.

---

## 4) UX State Matrix (Locations)

### State 1 — Healthy Mesh + peers
- Badge: `Mesh Connected`
- Device list visible
- Refresh button active
- Last updated visible

### State 2 — Tailscale daemon down / unavailable
- Badge: `Mesh Service Down`
- Show setup checklist + restart guidance
- Keep local-only mode active

### State 3 — Tailscale running but not authenticated/joined
- Badge: `Mesh Not Connected`
- Show auth/join checklist
- Keep local-only mode active

### State 4 — Authenticated but no peers online
- Badge: `Mesh Connected (No online peers)`
- Show "No remote peers currently online" helper text
- Keep local-only mode active

### State 5 — API/inspection error
- Badge: `Mesh Status Unknown`
- Show retry + diagnostic action
- Keep local-only mode active

## 5) Copy Deck (Draft)

### Header status messages
- Connected: "Mesh connected. Remote devices available."
- Not connected: "Mesh not connected yet. Complete setup to discover devices."
- Service down: "Mesh service is not running on this host."
- Unknown: "Couldn’t verify mesh status. Retry or check diagnostics."

### First-time panel title
"Set up secure device connectivity"

### First-time panel body
"Porter uses your private mesh network to discover and connect remote devices. Complete these steps, then refresh."

### Buttons
- Primary: "Open Setup Steps"
- Secondary: "Continue Local Only"
- Utility: "Refresh"

## 6) Technical Plan (implementation outline)

### Backend requirements
- Reuse existing Tailscale status APIs where available.
- Normalize response into one UI-oriented object:
  - `mesh_state`
  - `daemon_running`
  - `authenticated`
  - `peer_count_total`
  - `peer_count_online`
  - `last_checked_at`
  - `error_code` / `error_message`

### Frontend requirements
- Add lightweight status presenter for mesh state.
- Add refresh trigger calling existing status + node reload path.
- Add empty-state renderer for onboarding checklist.
- Preserve current Locations card layout and ordering.

---

## 7) Acceptance Criteria

1. First-time user without Tailscale sees guided setup, not ambiguous empty UI.
2. User can still browse local mounts when mesh is unavailable.
3. Device area includes manual refresh and last-updated text.
4. Mesh states are distinct and map to specific next actions.
5. No regressions to existing Locations visual layout.
