# Plan 38-02: Deep Execution & Tool Output Compression

**Phase:** 38 — Adaptive Agent Context
**Requirements:** ACX-03, ACX-04
**Depends on:** Plan 38-01 (directive scoring provides context metadata)

## Goal

Porter agents sustain 50+ turn conversations without context collapse. Verbose tool call results are automatically compressed before being stored in conversation history. Older conversation turns are summarized when context pressure rises, preserving key facts while reclaiming token budget.

## Current State

- `session-registry.ts` tracks `tokens_used` / `token_budget` per session
- At 95% context pressure, `rotateSession()` closes the session and opens a fresh one — this is a **hard cutoff that loses all conversation context**
- No mid-session compression exists
- Tool call results are stored verbatim in conversation history
- `__DISPATCH_META__` token already threads metadata through SSE streams

## Design

### Two-Layer Compression

#### Layer 1: Tool Output Compression (per-turn, synchronous)

Before a tool call result is appended to conversation history, check its size:
- If result ≤ 500 tokens (estimated): store verbatim
- If result > 500 tokens: compress via LLM summarization call

Compression call:
- Model: cheapest available (Haiku or local Ollama)
- System prompt: "Summarize this tool output preserving: key data values, error messages, status codes, file paths, and actionable items. Be concise."
- Max output: 200 tokens
- Fire-and-forget fallback: if compression call fails, store original (truncated to 1000 tokens)

Storage pattern:
```json
{
  "type": "tool_result",
  "compressed": true,
  "summary": "...",
  "original_tokens": 2400,
  "compressed_tokens": 180,
  "full_result_ref": "dispatch_log_id:abc123"
}
```

Full results are always preserved in `bridge_dispatch_log` for recall.

#### Layer 2: Conversation Compression (threshold-triggered, async)

When context pressure crosses 70% (configurable):
1. Identify turns older than the most recent 10
2. Group old turns into chunks of 5
3. Summarize each chunk: "These 5 turns discussed [topic]. Key outcomes: [list]."
4. Replace original turns with summaries in the working context
5. Original turns remain in the database (never deleted)

Trigger points:
- **70% context**: compress tool outputs from turns > 10 ago
- **85% context**: compress all turns > 5 ago (more aggressive)
- **95% context**: existing rotation behavior (unchanged, last resort)

### Context Summarizer Service

New service: `context-compressor.ts`

```typescript
export interface CompressionResult {
  originalTokens: number;
  compressedTokens: number;
  savedTokens: number;
  compressionEvents: number;
}

export async function compressToolOutput(
  result: string,
  threshold?: number
): Promise<{ summary: string; compressed: boolean }>

export async function compressConversationHistory(
  turns: ConversationTurn[],
  keepRecent: number,
  targetTokenBudget: number
): Promise<{ compressed: ConversationTurn[]; stats: CompressionResult }>
```

### Recall Mechanism

When an agent needs the full tool output that was compressed:
- System prompt includes: "If you need the full output of a previous tool call, reference its dispatch_id."
- Backend can serve full results from `bridge_dispatch_log` on demand
- This is passive — no new API endpoint needed initially; the data is already in the log

## Tasks

### Task 1: Context compressor service
**Files:** `backend/src/services/context-compressor.ts` (new, ~150 lines)
- `compressToolOutput(result, threshold)` — LLM summarization with fallback truncation
- `compressConversationHistory(turns, keepRecent, targetBudget)` — chunk-and-summarize
- `estimateTokens(text)` — reuse 4-chars-per-token heuristic from memory-injection
- Uses cheapest available model via Bridge internal dispatch (Haiku preferred, Ollama fallback)
- Fire-and-forget: compression failures never block dispatch

### Task 2: Tool output compression hook
**Files:** `backend/src/services/bridge/routing-engine.ts`
- After stream completion in `dispatchStream()`, before storing result in conversation history
- Check result token estimate; if > threshold, call `compressToolOutput()`
- Store compression metadata alongside the compressed result
- Add `compression_stats JSONB` to dispatch log entry (or extend existing metadata)

### Task 3: Conversation compression trigger
**Files:** `backend/src/services/session-registry.ts`
- Modify `upsertSession()`: when context_pct crosses 70% or 85%, emit compression event
- New method: `triggerCompression(sessionId, aggressiveness: 'mild' | 'aggressive')`
- Compression runs async — does not block the current dispatch
- Emits SSE `bridge:compression` event for admin visibility

### Task 4: Migration — compression tracking columns
**Files:** `backend/src/db/migrate-acx-v2.ts`
- Add to `bridge_dispatch_log`: `compression_stats JSONB` (nullable)
  - Shape: `{ tool_outputs_compressed: number, tokens_saved: number, compression_model: string }`
- Add to `session_registry`: `compression_events INTEGER DEFAULT 0`, `tokens_reclaimed INTEGER DEFAULT 0`

### Task 5: Configuration
**Files:** `backend/src/services/context-compressor.ts`
- Environment variables:
  - `PORTER_COMPRESS_THRESHOLD` — tool output token threshold (default 500)
  - `PORTER_COMPRESS_MILD_PCT` — context % for mild compression (default 0.70)
  - `PORTER_COMPRESS_AGGRESSIVE_PCT` — context % for aggressive compression (default 0.85)
  - `PORTER_COMPRESS_KEEP_RECENT` — turns to keep verbatim (default 10)
  - `PORTER_COMPRESS_MODEL` — preferred compression model (default 'haiku')

## Verification

1. Send a dispatch that returns a 3000-token tool result → result compressed to ~200 tokens in conversation history, full result in dispatch log
2. Run a 30-turn conversation → at turn ~20 (70% context), older turns get compressed; conversation continues coherently
3. Run a 50-turn conversation → multiple compression events fire, agent maintains coherent responses about topics from turn 5
4. Compression failure (model unavailable) → original result stored truncated, no error propagated
5. `session_registry` shows compression_events count and tokens_reclaimed
6. Admin SSE receives `bridge:compression` events
