# Plan 38-01: Context-Aware Directive Injection

**Phase:** 38 — Adaptive Agent Context
**Requirements:** ACX-01, ACX-02
**Depends on:** Phase 35 (evolution proposals use directive context), Phase 36 (quality scoring patterns reusable)

## Goal

`memory-injection.ts` selects only task-relevant directives instead of bulk-injecting all active directives. Agents with skills get directives matched to those skills. Agents without context get the current behavior (all directives, priority-ordered).

## Current State

`buildMemoryContext()` Tier 2 (lines ~70-95 of memory-injection.ts):
- Queries `SELECT content, priority FROM directives WHERE status='active' AND (scope='workspace' OR project_id=$1) ORDER BY priority ASC`
- Iterates results, appending each directive until token budget exhausted
- No filtering by task type, skill context, or conversation topic

## Design

### Directive Scoring

Reuse the pure-function keyword-matching pattern from `skill-selector.ts`:

```typescript
function scoreDirective(
  taskWords: string[],
  skillTags: string[],
  directive: { content: string; priority: number; tags?: string[] }
): number
```

Scoring factors:
- **Priority bonus**: `(10 - directive.priority)` — high-priority directives always score well
- **Task keyword match**: +2 per matched word in directive content (same as skill scoring)
- **Skill tag match**: +3 per matched tag — if the active skills have tags that appear in directive tags
- **Always-inject flag**: directives with `priority <= 2` bypass scoring entirely (critical directives)

### Directive Tags

Add optional `tags TEXT[]` column to `directives` table. Tags like `['coding', 'security', 'api', 'testing', 'deployment']` allow skill→directive affinity matching. Existing directives get tags via a one-time migration based on content keyword analysis.

### Injection Flow (Modified Tier 2)

```
1. Query all active directives (same query)
2. Partition: always_inject (priority ≤ 2) vs scoreable
3. Score scoreable directives against: taskText words + active skill tags
4. Sort scored by score DESC
5. Inject always_inject first, then scored until budget exhausted
6. Log: { total_directives, injected_count, skipped_count } in dispatch metadata
```

### Fallback

If no taskText or no skills context provided → fall back to current behavior (all directives, priority-ordered). This ensures backward compatibility for all existing dispatch paths.

## Verification

1. Dispatch an agent with coding skills on a coding task → only coding-relevant directives injected (< 50% of total)
2. Dispatch same agent on a security question → different directive subset selected
3. Dispatch with no task context → all directives injected (backward compat)
4. Priority ≤ 2 directives always injected regardless of scoring
5. `bridge_dispatch_log` shows directive selection stats
