# Phase 13.05: PostgreSQL Migration - Context

**Gathered:** 2026-03-23
**Status:** Ready for planning

<domain>
## Phase Boundary

Migrate Porter's entire data layer from SQLite/better-sqlite3 to PostgreSQL 16 + pgvector. All Drizzle schemas ported, all raw SQL converted, FTS5 replaced with tsvector/GIN, all 13 migrations consolidated into one PostgreSQL-native migration, existing data migrated with zero loss, boot sequence and config updated. Zero SQLite imports remain after completion.

</domain>

<decisions>
## Implementation Decisions

### PostgreSQL Environment
- Docker container with PG 16 + pgvector image via new `docker-compose.yml` at project root
- Hard-cut to PostgreSQL only — remove ALL SQLite code, no dual-driver support
- Connection via `DATABASE_URL` env var (standard connection string: `postgresql://porter:password@localhost:5432/porter`)
- Persistent Docker volume for PG data

### Migration Execution
- One-shot migration: stop Porter, run standalone script, switch config, restart
- Standalone migration script at `scripts/migrate-sqlite-to-pg.ts` — not part of normal boot
- No rollback plan — commit fully to PostgreSQL, delete SQLite after verified migration
- Use `drizzle-orm/node-postgres` (pg) as the PostgreSQL driver, replacing better-sqlite3

### Data Validation
- Row counts for all 35+ tables compared between SQLite and PG
- Spot-check 5-10 records per table (verify field values match)
- FTS verification: 5-10 known search terms against both SQLite FTS5 and PG tsvector, compare result sets
- Console summary table output: table name | sqlite rows | pg rows | status (green/red)

### Schema Consolidation
- Rewrite `backend/src/db/schema.ts` from scratch using `pgTable()` — clean break from SQLite dialect
- Single consolidated PG-native migration file replacing all 13 migrate-XX.ts files
- Delete old migrate-04.ts through migrate-13.ts after migration verified
- JSON columns (`metadata`, `config`, `trigger_data`, etc.) become `jsonb` — enables native PG JSON queries
- All timestamp columns become `timestamptz` — replace `strftime`/`unixepoch` with `NOW()` defaults
- FTS5 virtual tables → tsvector columns + GIN indexes on `messages` and `concepts` tables
- FTS5 sync triggers → PostgreSQL trigger functions for tsvector auto-update

### Claude's Discretion
- Exact Docker Compose configuration (ports, volumes, healthcheck)
- pgvector extension setup (CREATE EXTENSION) — for future Memory V3 use
- Connection pool configuration (pool size, idle timeout)
- Exact tsvector column placement and trigger function implementation
- Order of table creation in consolidated migration
- How to handle PRAGMA table_info() patterns (replace with information_schema queries or remove entirely)
- Sequence/serial strategy for auto-increment columns

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- Drizzle ORM already in use — pgTable() is a drop-in replacement for sqliteTable() with type changes
- All route handlers use `db` import from client.ts — single connection point to swap
- Drizzle query builder used in most routes — many queries won't need raw SQL changes

### Established Patterns
- All migrations check `schema_migrations` table for idempotency — same pattern works in PG
- FTS5 triggers follow insert/delete/update sync pattern — PG triggers use same logic with different syntax
- JSON stored as TEXT everywhere — jsonb conversion is a type change in schema, queries need operator swap (json_extract → ->/#>>)
- Unix epoch integers for timestamps — timestamptz conversion requires both schema change and query updates (strftime → NOW())

### Integration Points
- `backend/src/db/client.ts` — THE connection point. Replace Database(path) with Pool(connectionString)
- `backend/src/index.ts` lines 137-147 — Migration runner on startup (needs PG migration call)
- `backend/src/services/scheduler.ts` — Heaviest raw SQL user (15+ queries)
- All `sqlite.prepare()` calls across 55 files — need conversion to parameterized PG queries
- `backend/src/plugins/auth.ts` — Session validation queries

### Migration Complexity Hotspots
- migrate-10.ts — Legacy table rename + UUID backfill (complex, but one-time PG schema handles it)
- migrate-11.ts — FTS5 virtual tables + 8 junction tables + 3 triggers (biggest rewrite)
- migrate-13.ts — FTS5 + CHECK constraints + trigger-based sync

</code_context>
