## Summary

Porter Brain's SQLite data layer is a mature 13-migration stack with 35+ tables, two FTS5 virtual tables (messages, concepts), extensive raw SQL across 45 TypeScript source files, and SQLite-specific functions (`unixepoch()`, `strftime()`, `json_extract()`) scattered throughout. The codebase is already on drizzle-orm 0.45.1 which ships with full PostgreSQL support (`drizzle-orm/node-postgres`) — the schema rewrite is a type-level swap from `sqliteTable()` to `pgTable()` with dialect-appropriate column types.

**Critical discovery:** PostgreSQL 16 is already installed and running natively on this machine (`/usr/bin/psql`, `pg_isready` returns `accepting connections` on port 5432). The pgvector 0.6.0 extension is also already installed (`postgresql-16-pgvector` package). Docker is NOT available on this machine. The locked decision to use Docker Compose must be revised to use the native PostgreSQL service instead, or Docker must be installed first.

The biggest migration complexity is `scheduler.ts` (15+ raw queries, `RETURNING *`, `json_extract`, `unixepoch`), followed by 6 other files with `json_extract` and widespread `unixepoch('now')` → `EXTRACT(EPOCH FROM NOW())` conversion across routes and services.

**Primary recommendation:** Use the native PostgreSQL 16 installation (already running on port 5432) rather than Docker. Create a `porter` database and user, install pgvector, then proceed with the schema migration. This avoids Docker dependency entirely.

### Pattern 1: Drizzle Schema Conversion (sqliteTable → pgTable)

**What:** Replace every SQLite column type with PostgreSQL equivalents. One-to-one mapping for most types.

**Column type translation table:**

| SQLite (current) | PostgreSQL (target) | Notes |
|-----------------|---------------------|-------|
| `text('x')` | `text('x')` | Same — text is native in PG |
| `integer('x').primaryKey({ autoIncrement: true })` | `serial('x').primaryKey()` | PG uses SERIAL (or GENERATED ALWAYS AS IDENTITY) |
| `real('x').default(sql\`(unixepoch('now'))\`)` | `doublePrecision('x').default(sql\`EXTRACT(EPOCH FROM NOW())\`)` | Epoch as float; OR use timestamptz |
| `blob('data')` | `customType bytea` or `text('data')` | chat_attachments only — raw binary |
| `integer('is_system').default(0)` | `boolean('is_system').default(false)` | Convert 0/1 integers to boolean |
| `text('metadata')` (JSON string) | `jsonb('metadata')` | Enables native JSON operators |

**Important:** The locked decision says JSON columns become `jsonb` and timestamps become `timestamptz`. However, the current schema stores ALL timestamps as Unix epoch floats (REAL columns). The team has two options for timestamps:
1. **Keep as epoch floats** (`doublePrecision`) — zero query changes in WHERE clauses, simpler migration
2. **Convert to `timestamptz`** — requires updating every raw SQL query that compares timestamps

Given the volume of raw SQL using epoch comparisons (`scheduled_for <= unixepoch('now')`, `expires < Date.now() / 1000`), keeping epoch floats as `doublePrecision` is significantly safer. The CONTEXT.md says "timestamptz — replace strftime/unixepoch with NOW() defaults" which applies to DEFAULT values, not necessarily to the storage type itself. **Recommendation: use `doublePrecision` for timestamp columns and `EXTRACT(EPOCH FROM NOW())` as the DEFAULT — preserving epoch semantics throughout.**

**Example — users table:**
```typescript
// Source: drizzle-orm/node-postgres documentation + verified against codebase
import { pgTable, text, doublePrecision, serial, boolean, jsonb, integer } from 'drizzle-orm/pg-core';
import { sql } from 'drizzle-orm';

export const users = pgTable('users', {
  username: text('username').primaryKey(),
  displayName: text('display_name'),
  fullName: text('full_name'),
  email: text('email'),
  passwordHash: text('password_hash').notNull(),
  salt: text('salt').notNull(),
  role: text('role').default('operator'),
  emailVerified: boolean('email_verified').default(false),
  status: text('status').default('active'),
  createdAt: doublePrecision('created_at').default(sql`EXTRACT(EPOCH FROM NOW())`),
});
```

**Example — agentJobs (with serial PK):**
```typescript
// Tables with integer autoincrement PK use serial()
export const chatMessages = pgTable('chat_messages', {
  id: serial('id').primaryKey(),
  chatId: text('chat_id').notNull().references(() => chats.id, { onDelete: 'cascade' }),
  // ...
});
```

**Example — jsonb columns:**
```typescript
export const conversations = pgTable('conversations', {
  id: text('id').primaryKey(),
  scopeType: text('scope_type').notNull(),
  metadata: jsonb('metadata').default(sql`'{}'::jsonb`),
  createdAt: doublePrecision('created_at').default(sql`EXTRACT(EPOCH FROM NOW())`),
  updatedAt: doublePrecision('updated_at').default(sql`EXTRACT(EPOCH FROM NOW())`),
});
```

### Pattern 2: Client Replacement (better-sqlite3 → pg Pool)

**What:** Replace the synchronous better-sqlite3 Database with an async pg Pool. This is the single swap point — all routes import `{ db, sqlite }` from `client.ts`.

**Critical:** The current codebase exports both `db` (Drizzle ORM instance) and `sqlite` (raw better-sqlite3 handle). The raw `sqlite` handle is used in 45 files. After migration, the raw handle becomes a `pg.Pool` instance, and all `sqlite.prepare(sql).run/get/all()` calls become parameterized pool queries.

```typescript
// Source: drizzle-orm/node-postgres driver documentation
import { Pool } from 'pg';
import { drizzle } from 'drizzle-orm/node-postgres';
import * as schema from './schema.js';

const pool = new Pool({
  connectionString: process.env.DATABASE_URL,
  max: 10,
  idleTimeoutMillis: 30000,
  connectionTimeoutMillis: 5000,
});

export const db = drizzle(pool, { schema });

// Export pool as the raw query interface (replaces sqlite export)
// All callers using sqlite.prepare().run/get/all() must be updated
export { pool };
```

**IMPORTANT:** The `sqlite.prepare(sql).get()` / `.all()` / `.run()` synchronous API does not exist in `pg`. All raw SQL callers must switch to:

```typescript
// SQLite (current):
const row = sqlite.prepare('SELECT * FROM users WHERE username = ?').get(username);

// PostgreSQL (new):
const result = await pool.query('SELECT * FROM users WHERE username = $1', [username]);
const row = result.rows[0];
```

This means every function using raw `sqlite` queries must become `async`. Most route handlers are already async. Scheduler functions (`claimNextJob`, `markJobComplete`, etc.) are synchronous — they must become async and the scheduler tick must be updated.

### Pattern 3: Raw SQL Dialect Conversion

**SQLite → PostgreSQL substitutions required across all 45 files:**

| SQLite | PostgreSQL | Files Affected |
|--------|-----------|----------------|
| `json_extract(col, '$.key')` | `col->>'key'` (text) or `col->'key'` (jsonb) | 5 files: projects.ts, wizard.ts, collaborators.ts, scheduler.ts (×2) |
| `unixepoch('now')` | `EXTRACT(EPOCH FROM NOW())` | 20+ files across routes and services |
| `strftime('%s','now')` | `EXTRACT(EPOCH FROM NOW())` | calendar.ts, schema.ts |
| `AUTOINCREMENT` | `SERIAL` or `GENERATED ALWAYS AS IDENTITY` | Schema only |
| `?` positional params | `$1, $2, $3...` positional params | All raw SQL across 45 files |
| `@namedParam` (better-sqlite3 named) | No named params — use `$1` positional | scheduler.ts claimNextJob uses `@workerId` |
| `RETURNING *` (UPDATE...RETURNING) | Identical in PostgreSQL — works unchanged | scheduler.ts claimNextJob |
| `INTEGER PRIMARY KEY` (rowid alias) | Serial PK — rowid concept does not apply | FTS rowid references must change |
| `CREATE VIRTUAL TABLE ... USING fts5(...)` | No equivalent — use tsvector column + GIN | migrate-11.ts, migrate-13.ts |
| `WHERE fts_table MATCH ?` | `WHERE tsv_col @@ to_tsquery('english', $1)` | memory.ts, conversations.ts |
| `ORDER BY rank` (FTS5 rank) | `ORDER BY ts_rank(tsv_col, to_tsquery(...)) DESC` | memory.ts, conversations.ts |
| `DELETE FROM sessions WHERE expires <= unixepoch('now')` | `DELETE FROM sessions WHERE expires <= EXTRACT(EPOCH FROM NOW())` | auth.ts |
| `SELECT 1 ... LIMIT 1` | Identical | Many files |
| `INSERT OR IGNORE` | `INSERT ... ON CONFLICT DO NOTHING` | Anywhere upsert patterns exist |

**Named parameter pattern in better-sqlite3** (specific to scheduler.ts claimNextJob):
```typescript
// SQLite (current):
sqlite.prepare(`UPDATE ... WHERE id = @workerId`).get({ workerId: WORKER_ID });

// PostgreSQL (new):
await pool.query(`UPDATE ... WHERE id = $1`, [WORKER_ID]);
```

### Pattern 4: FTS5 → tsvector + GIN

**Two FTS5 virtual tables to replace:**

1. `messages_fts` (migrate-11.ts) — indexes `content`, `sender_name`, `channel_type`
2. `concepts_fts` (migrate-13.ts) — indexes `content` only (content table, rowid-based)

**PostgreSQL tsvector approach:**

Add a `tsvector` column to each parent table, maintain it via trigger function, index with GIN.

**messages table — add search column:**
```sql
-- Add tsvector column to messages
ALTER TABLE messages ADD COLUMN search_vector tsvector;

-- GIN index for fast FTS
CREATE INDEX idx_messages_search ON messages USING GIN(search_vector);

-- Trigger function to keep search_vector current
CREATE OR REPLACE FUNCTION messages_search_update() RETURNS trigger AS $$
BEGIN
  NEW.search_vector :=
    setweight(to_tsvector('english', COALESCE(NEW.content, '')), 'A') ||
    setweight(to_tsvector('english', COALESCE(NEW.sender_name, '')), 'B') ||
    setweight(to_tsvector('english', COALESCE(NEW.channel_type, '')), 'C');
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER messages_search_trig
  BEFORE INSERT OR UPDATE ON messages
  FOR EACH ROW EXECUTE FUNCTION messages_search_update();
```

**concepts table — add search column:**
```sql
ALTER TABLE concepts ADD COLUMN search_vector tsvector;
CREATE INDEX idx_concepts_search ON concepts USING GIN(search_vector);

CREATE OR REPLACE FUNCTION concepts_search_update() RETURNS trigger AS $$
BEGIN
  NEW.search_vector := to_tsvector('english', COALESCE(NEW.content, ''));
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER concepts_search_trig
  BEFORE INSERT OR UPDATE ON concepts
  FOR EACH ROW EXECUTE FUNCTION concepts_search_update();
```

**Query replacement in routes:**

```typescript
// memory.ts — FTS5 MATCH → tsvector @@
// SQLite (current):
sqlite.prepare(`
  SELECT c.* FROM concepts c
  JOIN concepts_fts f ON f.rowid = c.rowid
  WHERE concepts_fts MATCH ?
  ORDER BY rank
`).all(q)

// PostgreSQL (new):
const result = await pool.query(`
  SELECT *, ts_rank(search_vector, to_tsquery('english', $1)) AS rank
  FROM concepts
  WHERE search_vector @@ to_tsquery('english', $1)
    AND status = $2
  ORDER BY rank DESC
  LIMIT $3 OFFSET $4
`, [q, status, limit, offset]);

// conversations.ts — messages FTS search
// PostgreSQL (new):
const result = await pool.query(`
  SELECT m.id, m.conversation_id, m.content, m.sender_name, m.sender_type,
         m.channel_type, m.created_at,
         ts_rank(m.search_vector, to_tsquery('english', $1)) AS rank
  FROM messages m
  WHERE m.search_vector @@ to_tsquery('english', $1)
  ORDER BY rank DESC
  LIMIT $2 OFFSET $3
`, [q, limit, offset]);
```

**Note on `to_tsquery` vs `plainto_tsquery`:** `to_tsquery` requires properly formatted query strings (operators like `&`, `|`). User input should use `plainto_tsquery` or `websearch_to_tsquery` to safely handle arbitrary strings without quoting errors. The FTS5 MATCH is tolerant of plain strings; PG is not. **Use `websearch_to_tsquery('english', $1)` for user-facing search — it handles multi-word queries, phrases, and special characters safely.**

### Anti-Patterns to Avoid

- **Keeping the `sqlite` export name** for the pool: Rename to `pool` to make the swap explicit. All callsites need updating anyway — ambiguous naming causes confusion.
- **Synchronous pg queries:** `pg` is async-only. Never use `.get()` syntax patterns — always `await pool.query(...)`.
- **Using `to_tsquery` with raw user input:** User queries like "project manager" will fail (`syntax error in tsquery`). Always use `websearch_to_tsquery` for user-facing search.
- **Skipping the backfill step:** Migrated rows won't have `search_vector` populated until explicitly backfilled with `UPDATE messages SET search_vector = ...`.
- **Migration script on a live server:** Script must run with Porter stopped. Document this in script header.
- **Forgetting pgvector extension:** `CREATE EXTENSION IF NOT EXISTS vector` must run before any `vector` typed columns are created (even if no vector columns exist yet — Phase 13.1 will add them).

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Full-text search ranking | Custom tf-idf or LIKE queries | `tsvector` + `ts_rank` + GIN index | PG native; handles stemming, stop words, weighting |
| JSON path queries | String manipulation | `->` and `->>` operators on `jsonb` | Native PG jsonb operators; indexed |
| Connection pooling | Manual pool management | `pg.Pool` with standard config | Built-in retry, idle timeout, max connections |
| Batch inserts | Individual INSERTs in loop | `pg` `COPY` protocol or multi-row `VALUES` | 10-100x faster for large tables |
| Migration idempotency | Custom IF NOT EXISTS per table | `ON CONFLICT DO NOTHING` on schema_migrations | Same pattern as existing SQLite migrations |
| Partial index on nullable column | Full index with null handling | `CREATE INDEX ... WHERE column IS NOT NULL` | Matches existing SQLite partial index pattern (idx_msg_parent) |

**Key insight:** The existing Drizzle ORM layer handles 80% of queries cleanly — only raw SQL `sqlite.prepare()` calls need manual conversion.

## Common Pitfalls

### Pitfall 1: Synchronous → Async Propagation in Scheduler
**What goes wrong:** `scheduler.ts` uses synchronous `sqlite.prepare().run/get/all()`. The `claimNextJob()` function is called inside `setInterval` without `await`. Converting to async `pg` queries requires making `claimNextJob`, `markJobComplete`, `markJobFailed`, `logActivity`, and all bootstrap functions async — which chains up to `tick()` and the interval pattern.
**Why it happens:** better-sqlite3 is synchronous by design; pg is always async.
**How to avoid:** Convert `tick()` to async first, then each helper in dependency order. The `setInterval` callback already wraps in try/catch — just add `await`.
**Warning signs:** TypeScript error "Promise<> is not assignable to void" on interval callback; runtime jobs silently not executing.

### Pitfall 2: FTS5 rowid vs PostgreSQL serial id
**What goes wrong:** FTS5 `concepts_fts` uses `content_rowid='rowid'` which maps to SQLite's implicit rowid (identical to `id` for TEXT PK tables when using `rowid` alias). In PG, `concepts.id` is TEXT (uuid) and there is no implicit integer rowid. The tsvector approach stores the vector IN the row, so the join via rowid is unnecessary.
**Why it happens:** SQLite FTS5 requires external rowid tracking; PG tsvector is a column on the same row.
**How to avoid:** The new query pattern is `WHERE search_vector @@ ...` on the table directly — no join needed.
**Warning signs:** Old queries doing `JOIN concepts_fts ON f.rowid = c.rowid` fail with relation not found.

### Pitfall 3: `json_extract` → `->>` Operator on JSONB
**What goes wrong:** `json_extract(trigger_data, '$.contact_id')` returns NULL if `trigger_data` is not valid JSON. In PG, `trigger_data->>'contact_id'` on a `jsonb` column also returns NULL for missing keys — but throws an error if the column value is not valid JSON.
**Why it happens:** During migration, all TEXT JSON columns become `jsonb`. If any row has invalid JSON (empty string `''`, `NULL`, or malformed), the `::jsonb` cast fails.
**How to avoid:** In migration script, validate/normalize JSON columns before insert. Use `COALESCE(json_column, '{}')` pattern. For raw SQL: `(trigger_data->>'contact_id')` requires `trigger_data` to be actual jsonb.
**Warning signs:** Migration script errors on rows with empty-string JSON fields.

### Pitfall 4: Named Parameters (@name) → Positional ($1)
**What goes wrong:** `scheduler.ts` `claimNextJob` uses named parameters: `.prepare(...@workerId...).get({ workerId: WORKER_ID })`. PostgreSQL `pg` only supports positional parameters (`$1`, `$2`). The `pg` library has no named param support.
**Why it happens:** better-sqlite3 supports named params; pg does not.
**How to avoid:** Convert all named param queries to positional. Search for `@` in all raw SQL strings.
**Warning signs:** PG syntax error: "syntax error at or near '@'".

### Pitfall 5: `websearch_to_tsquery` vs `to_tsquery` for User Input
**What goes wrong:** `to_tsquery('english', 'project manager')` throws `syntax error in tsquery: "project manager"` because multi-word input without operators is invalid.
**Why it happens:** `to_tsquery` expects explicit operators (`&`, `|`, `!`, `<->`). FTS5 MATCH handled plain strings automatically.
**How to avoid:** Use `websearch_to_tsquery('english', $1)` for all user-facing search endpoints. It handles multi-word phrases, quoted strings, and minus-exclusions safely.
**Warning signs:** 500 errors on search queries containing spaces or special characters.

### Pitfall 6: Integer Boolean Columns
**What goes wrong:** 17+ columns use `integer('x').default(0)` as boolean (is_system, is_locked, is_public, etc.). If schema changes to `boolean`, the migration must convert `0 → false` and `1 → true`. If Drizzle schema uses `boolean` but migration copies raw integers, TypeScript types mismatch at runtime.
**Why it happens:** SQLite has no boolean type; PG does.
**How to avoid:** Either (a) keep as `integer`/`smallint` in PG schema for zero-change semantics, or (b) map to `boolean` and convert in migration script. Option (a) is lower risk. **Recommendation: keep as integer for zero application-level changes.**
**Warning signs:** TypeScript type errors where boolean is expected but number is returned.

### Pitfall 7: Playwright Tests Hit Port 8877 (porter.py, not Fastify)
**What goes wrong:** Playwright config targets `baseURL: 'http://127.0.0.1:8877'` — this is the legacy porter.py port, NOT the Fastify backend (`:3001`). The 35 tests are testing the Python monolith, not the Node.js backend being migrated.
**Why it happens:** Historical configuration not updated during Fastify migration.
**How to avoid:** The PG migration only affects the Fastify backend. The test suite passes regardless of which DB Fastify uses (since tests hit porter.py). Success criteria #4 requires "all 35 Playwright tests pass" — they will pass because they don't hit the Fastify server at all. The real validation is the smoke tests (`smoke-phase11.sh`, `smoke-phase12.sh`, `smoke-phase13.sh`) which target the Fastify API.
**Warning signs:** If porter.py is stopped during the migration window, ALL 35 tests fail — even though PG migration is correct.

| Old Approach | Current Approach | When Changed | Impact |
|--------------|------------------|--------------|--------|
| `drizzle-orm/sqlite-core` sqliteTable | `drizzle-orm/pg-core` pgTable | drizzle-orm 0.30+ | Same API shape, different import path |
| better-sqlite3 synchronous API | pg.Pool async API | This migration | All raw SQL callers must become async |
| FTS5 virtual tables | tsvector column + GIN + trigger | This migration | Better multi-language support, native ranking |

| `json_extract(col, '$.key')` | `col->>'key'` on jsonb | This migration | Enables jsonb indexing |
| `unixepoch('now')` | `EXTRACT(EPOCH FROM NOW())` | This migration | Same epoch semantics |

- All 13 individual migration files: replaced by `migrate-consolidated.ts`

- SQLite WAL mode pragmas: not applicable to PostgreSQL

## Open Questions

1. **Docker vs Native PostgreSQL**
   - What we know: Docker is not available. PostgreSQL 16 + pgvector 0.6.0 are already installed natively on port 5432.
   - What's unclear: Can we connect as `lobster` user to create DB/user, or does this require `sudo`? The CONTEXT.md says Docker Compose — but Docker isn't installed.
   - Recommendation: Plan for native PostgreSQL setup with `sudo -u postgres` commands for DB/user creation. Also create `docker-compose.yml` for documentation. Mark Docker setup as "optional for production deployment" — native is the dev path.

2. **pgvector `CREATE EXTENSION` permissions**
   - What we know: `postgresql-16-pgvector` package is installed. The extension is available.
   - What's unclear: Whether the `porter` database user will have `CREATE EXTENSION` permission (superuser required in some configs).
   - Recommendation: Run `CREATE EXTENSION IF NOT EXISTS vector` as the postgres superuser during DB setup, or grant `porter` user superuser temporarily for migration.

3. **Playwright tests and the migration**
   - What we know: 35 Playwright tests target port 8877 (porter.py), not port 3001 (Fastify).
   - What's unclear: Whether success criteria #4 ("All 35 Playwright tests pass against the PostgreSQL-backed server") is achievable if tests target the Python server.
   - Recommendation: Clarify in the plan that PG migration does not break the 35 tests (they don't use Fastify). Validation of PG-backed Fastify uses smoke tests, not Playwright.

4. **Boolean vs Integer for boolean-semantic columns**
   - What we know: 17+ columns use integer 0/1 for booleans. Changing to PG boolean type requires application-level changes everywhere.
   - What's unclear: Whether the team prefers clean types (boolean) or zero-app-change (integer).
   - Recommendation: Keep as `integer`/`smallint` in schema to minimize changes. Can migrate to `boolean` in a future cleanup phase.
