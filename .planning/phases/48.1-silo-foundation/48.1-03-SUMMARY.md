## Decisions Made

- **Endpoint mounted at `/api/v1/intellect/silo-command`, not `/api/v1/silo-command`:** keeps it inside the intellect route group prefix where the consumer (`/context`) and the detector live. Co-location simplifies reasoning about read-write coupling on `session_silo_overrides`.
- **Two distinct UPSERTs** (one for clear, one for set) instead of a parameterized single UPSERT: the clear path doesn't need `siloName`, and writing `silo_id = $2` with `$2 = NULL` in the set path would require a runtime null-vs-string check that two simple queries avoid.
- **Hook fail-closed on backend failure:** the alternative (fall through to /correction or to model) was rejected because a `/silo software` line passed to the model would (a) confuse the system about whether the silo had been applied and (b) potentially be misinterpreted as a magic command by the model. Blocking with a clear "backend unreachable" message is the only honest UX.
- **Silos validated against `enabled = TRUE`:** matches the cache load in `silo-detector.ts` (which also filters by `enabled = TRUE`) so disabled silos cannot be overridden into. Validator query is cheap (PK lookup with a partial filter).
- **No source commit for the hook:** the hook file is part of the user's global Claude Code configuration, not the Porter repo. Committing it would require either symlinks or a script step — out of scope for this plan. The hook content is reproduced verbatim in this SUMMARY for re-deployment if needed.
