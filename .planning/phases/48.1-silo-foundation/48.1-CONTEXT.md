<decisions>
## Implementation Decisions (all locked)

### Schema
- **Table name:** `silos`
- **Columns:** `id TEXT PK`, `display_name TEXT`, `description TEXT`, `prompt_path TEXT`, `cadence_seconds INT DEFAULT 604800`, `default_model TEXT DEFAULT 'claude-sonnet-4-6'`, `detect_rules JSONB DEFAULT '{}'`, `enabled BOOLEAN DEFAULT true`, `created_at TIMESTAMPTZ DEFAULT now()`
- **Seed row:** `id='software'`, `display_name='Software Development'`, `prompt_path='backend/src/services/intellect/dream-prompts/software.md'` (file will not exist yet — that's a 48.3 deliverable; column carries the future path), `cadence_seconds=604800`, `default_model='claude-sonnet-4-6'`, `detect_rules={"project_types":["website","app","api","library"],"file_globs":["*.ts","*.tsx","*.py","*.go","*.rs","*.sql","*.sh"],"cwd_markers":["package.json","pyproject.toml","Cargo.toml","go.mod",".git"]}`
- **Migration file:** `backend/src/db/migrate-silos-v1.ts` following existing migrate-*-v1 pattern (idempotent, runs at startup)
- **Drizzle schema file:** update `backend/src/db/schema.ts` (or wherever the canonical Drizzle table definitions live — discover during research)

### Silo Detection (server-side, at session-start hook)
- **Input:** `cwd` from the session-start hook payload, plus optional `silo_override` from a session-scoped record
- **Algorithm:**
  1. If `silo_override` is set for this session → return that.
  2. Look up project by cwd via existing project-detection (Phase 47 substrate has this — discover the exact lookup in research).
  3. If project has `project_type` ∈ `detect_rules.project_types` for any enabled silo → that silo.
  4. If cwd contains any file matching `detect_rules.cwd_markers` (presence check, not content) → check matching silos.
  5. Multiple silo matches → return all (multi-silo session). Empty → return null.
- **No ML, no model dispatch.** Pure deterministic checks. The expensive synthesis-time re-classification lives in 48.3.

### Context API Injection
- **Endpoint touched:** `GET /api/v1/intellect/context?project=X` in `backend/src/routes/v1/intellect.ts`
- **New section in the returned markdown context:**
  ```
  ## Silo: <display_name> — Operating Rules
  - <directive 1>
  - <directive 2>
  ```
- **Query:** `SELECT id, content, priority FROM directives WHERE status='active' AND scope='silo' AND scope_id = ANY($silos) ORDER BY priority DESC LIMIT 20`
- **Placement:** silo section appears AFTER global directives and BEFORE project directives in the output. Order matters because tiered injection treats earlier sections as higher-trust context.
- **Null silo:** section omitted entirely. No empty header.

### `/silo` CLI Command
- **Type:** A slash command handled by Porter's existing CLI-input pipeline (already in place for `/clear`, `/help`, etc — discover entry point in research).
- **Persistence:** Per-session override in a new lightweight table `session_silo_overrides (session_id TEXT PK, silo_id TEXT, set_at TIMESTAMPTZ)`. TTL: 24h via cleanup job. Or: piggyback on existing `session_registry` table if it has a free column — research finds out.
- **Commands:**
  - `/silo software` — force software silo
  - `/silo none` — clear override, fall back to detection
  - `/silo` (no arg) — print current effective silo and source (override vs detected)

### Seed Directives — Already in DB
The 5 standing software-silo directives are already in the `directives` table (priority 95, scope=`silo`, scope_id=`software`, source_type=`moe-direct`). **DO NOT re-seed in this phase.** Plans must verify they exist; if a plan seeds them again it's wrong.

### Seed Immutability
A new check on the directives endpoint (or via a DB CHECK / trigger): any UPDATE or DELETE on a row where `source_type='moe-direct'` is rejected unless a special bypass flag is passed (admin API only). Implementation choice: PostgreSQL trigger is cleanest, mirroring the existing `personas_template_born_check` pattern (see Phase 6.8.0 — `backend/src/db/migrate-born-check-v1.ts`).

### Parallelism (Moe's standing rule)
This phase's plans are mostly sequential (schema → API consumes schema → hook consumes API). But within a plan, independent file edits should run in parallel. Plan authors must mark non-competing tasks `parallel: true` in the wave config. Where research is needed (e.g., locating Drizzle schema entry, locating CLI command dispatch), spawn parallel Explore agents per Moe's rule.

<specifics>
## Specific Ideas

- **Multi-silo handling:** When detection returns multiple silos (rare, but a Porter project doing legal review of Porter code might match both `software` and `legal`), the context endpoint should inject **all** matched silos' directives — each in its own `## Silo: <name>` section. They never blend. Plans must support N silos cleanly.
- **Performance:** silo detection runs on every session start. The lookup is O(silos in registry × few checks). Cache silos in-process at startup; reload on registry change via an event. Don't query Postgres on every hook.
- **Backward compatibility:** existing CLI sessions without a silo override see the same behavior as before EXCEPT for code projects, which start getting software-silo directives injected. This is intended — it's the whole point. If something breaks, it's a verification finding, not a regression.
- **Verification command for success criterion #3:** Plans should spell out the exact verification: `cd /home/lobster/projects/Porter && curl -s "http://127.0.0.1:3001/api/v1/intellect/context?project=Porter" | grep -c "Silo: Software"` should return `1`.

</specifics>
