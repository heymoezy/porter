### Investigation 5: `/silo` CLI Command Dispatch

**Finding: No existing slash command pipeline exists.**

Claude CLI's `UserPromptSubmit` hook (`~/.claude/hooks/porter-user-prompt.js`) is the ONLY entry point for intercepting user-typed text before it reaches the model. It receives a JSON payload via stdin with:
```json
{ "prompt": "...", "session_id": "...", ... }
```

The hook currently: checks `prompt.length < 12`, extracts project from cwd, POSTs to `/api/v1/intellect/correction`. It is fire-and-forget and never writes to stdout (by design, to avoid polluting the prompt pipeline).

**The `/silo` command mechanism:**
- Claude does NOT have a native slash command system (unlike `/clear`, `/help` which are Claude CLI built-ins handled by the CLI itself, not hooks).
- `/clear` and `/help` are not hook-dispatched — they are Claude CLI internal commands.
- A `/silo` command must be intercepted in the `UserPromptSubmit` hook by checking `if (prompt.startsWith('/silo'))`.
- The hook can return `{ "decision": "block" }` to prevent the prompt from reaching the model, OR it can output text that gets injected as additional context.

**Confirmed mechanism:** The `PostToolUse` hook in `gsd-context-monitor.js` (line 144-151) outputs JSON with `{ hookSpecificOutput: { hookEventName: "PostToolUse", additionalContext: "..." } }` to inject content. The `UserPromptSubmit` hook equivalent is `{ decision: "block" }` to suppress the message from going to the model, plus stdout text for any response to the user.

**Recommended implementation:** Extend `porter-user-prompt.js` to detect `prompt.startsWith('/silo')`, call a new backend endpoint `POST /api/v1/intellect/silo-command` with `{ sessionId, command: '/silo software' }`, and either:
  - Return `{ "decision": "block" }` to suppress the raw command from the model AND print a confirmation to stdout (which gets injected as context)
  - OR let it pass through so the model sees it and can respond naturally

The cleaner UX is to block it (it's a meta-command, not a prompt) and echo a confirmation as `additionalContext`. This mirrors how Claude CLI built-in commands work.

**Planning Blocker #1:** The `porter-user-prompt.js` hook currently never outputs anything to stdout and is fire-and-forget. For `/silo` to work as a blocking interceptor, the hook needs a new code path that: (a) detects the `/silo` prefix, (b) calls the backend synchronously (awaiting the response), (c) outputs a `{ decision: "block", ... }` JSON or `additionalContext` JSON, and (d) exits. This is new infrastructure — the hook currently has no `await`-based blocking flow.

### Investigation 7: Project Type Lookup for Silo Detection

**`projects` table has `type TEXT DEFAULT 'custom'` and `fs_path TEXT` columns.**

Live data shows: `type` values include `'manual'`, `'custom'` — not yet `'website'`, `'app'`, `'api'`, `'library'` (the types in `detect_rules.project_types`). The Porter project itself does not have an entry with `type='app'` or `fs_path` pointing to its root.

**Detection algorithm for Phase 48.1:**
1. Try project lookup: `SELECT type FROM projects WHERE fs_path IS NOT NULL AND $cwd LIKE fs_path || '%'` — if match and `type IN detect_rules.project_types` → silo match.
2. Fallback cwd_markers: `fs.existsSync(path.join(cwd, marker))` for each marker in `detect_rules.cwd_markers`.
3. Multi-match: return all matching silo IDs.
4. No match: return `[]` (null silo).

**Critical note:** The Porter project at `/home/lobster/projects/Porter` does not have an `fs_path` entry in `projects`. Step 2 (cwd_markers) is the primary detection path for the success criterion test `cd /home/lobster/projects/Porter && curl ...`. The `package.json` in Porter root will match the `cwd_markers` check. This is the expected fallback and is correct.

**`/home/lobster/projects/Funds`** — no `projects` row likely exists, no `package.json` or other code markers. Will return null. This is the expected outcome for success criterion #4.

### Investigation 8: Migration Patterns — Idempotency

**Canonical pattern from `migrate-born-check-v1.ts`:**
1. Check `schema_migrations` for the migration ID: `SELECT 1 FROM schema_migrations WHERE id = 'born_check_v1'`
2. If found → `COMMIT; return;`
3. All DDL inside `BEGIN` / `COMMIT` transaction
4. Functions use `CREATE OR REPLACE FUNCTION`
5. Triggers use `DROP TRIGGER IF EXISTS` before `CREATE TRIGGER`
6. Tables use `CREATE TABLE IF NOT EXISTS`
7. `INSERT INTO schema_migrations (id, applied_at) VALUES ('silos_v1', EXTRACT(EPOCH FROM NOW()))`
8. On error: `ROLLBACK; throw err;`
9. Console log: `[migrate-silos-v1] complete`

**`migrate-intellect-v1.ts` pattern** adds `ALTER TABLE ... ADD COLUMN IF NOT EXISTS` and `CREATE INDEX IF NOT EXISTS` — idempotent for incremental additions.

**The silos migration** creates a new table (no alter needed), inserts a seed row, and creates the immutability trigger. Pattern:
- `CREATE TABLE IF NOT EXISTS silos (...)`
- `INSERT INTO silos ... ON CONFLICT (id) DO NOTHING` (idempotent seed)
- `CREATE OR REPLACE FUNCTION protect_moe_direct_directives() RETURNS trigger AS ... `
- `DROP TRIGGER IF EXISTS moe_direct_immutable ON directives`
- `CREATE TRIGGER moe_direct_immutable BEFORE UPDATE OR DELETE ON directives FOR EACH ROW ...`

### Investigation 9: Seed Immutability Mechanism

**Option A — PostgreSQL trigger:**
- Mirrors the existing `personas_template_born_check` trigger (`migrate-born-check-v1.ts`)
- Function: `IF OLD.source_type = 'moe-direct' THEN RAISE EXCEPTION '...' END IF; RETURN NEW/NULL;`
- Pros: DB-layer guarantee, catches all paths (API, direct psql, pruner, promoter)
- Cons: Blocks legitimate admin override via DB — needs a bypass pattern

**Existing threat surface confirmed:** `memory-pruner.ts` line 180 does `DELETE FROM directives WHERE status = 'superseded'` — `moe-direct` rows have `status='active'`, not `'superseded'`, so they are NOT at risk from the pruner today. However, the dedup logic at line ~100-165 could mark a `moe-direct` row as `status='superseded'` if it finds a near-duplicate (similarity >= 0.85). This is the real risk path.

**Option B — Application-layer check:**
- Add `if (row.source_type === 'moe-direct') throw new Error(...)` in the mutation endpoints
- Pros: Easier to override (admin flag)
- Cons: Doesn't protect against direct DB operations or future services not implementing the check

**Recommendation:** Option A (trigger). The spec explicitly says "mirroring the existing `personas_template_born_check` pattern." The trigger RAISE EXCEPTION prevents accidental deletion/modification by any process. For intentional admin overrides, a separate `DELETE FROM directives WHERE source_type='moe-direct' AND id=$1` via `psql` is always available to Moe.

**Trigger placement:** In the `migrate-silos-v1.ts` migration file — same file that creates the `silos` table, keeping all Phase 48.1 DDL together.

## Planning Blockers

### Blocker 1: `/silo` slash command — no existing pipeline

**Status:** Infrastructure gap. The `porter-user-prompt.js` hook is fire-and-forget and never writes to stdout. A blocking interception path must be added.

**Scaffolding needed:**
1. Extend `porter-user-prompt.js` to detect `/silo` prefix and execute a synchronous HTTP request to the backend, then output `{ "decision": "block", ... }` JSON.
2. New endpoint `POST /api/v1/intellect/silo-command` in `intellect.ts` (or a new `silo.ts` file).
3. New table `session_silo_overrides` in the migration.
4. New query in `GET /context` to check override for the given `session_id`.

This is not a fundamental blocker — it is known, scoped work. Plans must allocate a wave for this.

### Blocker 2: `session_id` missing from context endpoint call chain

**Status:** The `porter-session-start.js` hook does not currently read stdin or pass `session_id`. The context endpoint needs `session_id` to look up silo overrides. The `SessionStart` hook payload includes `session_id` in stdin JSON — the hook must be modified to read stdin first and pass it as a query param.

**Scaffolding needed:** Modify `porter-session-start.js` to read stdin JSON (like other hooks), extract `session_id`, and pass it to `/api/v1/intellect/context?project=X&cwd=Y&session_id=Z`.

### Blocker 3: Porter project not in `projects` table with `fs_path`

**Status:** The primary success criterion test (`cwd=/home/lobster/projects/Porter`) relies on the `cwd_markers` fallback (finding `package.json`). The project-type lookup will miss Porter unless a `projects` row with `fs_path` and `type='app'` exists. This is by design — the cwd_markers path handles it. Plans must not assume project-type lookup is the primary path for the Porter project specifically.

**No scaffolding needed** — the detection algorithm in the spec already handles this. Plans should document that cwd_markers is the expected detection path for the Porter project.

## Risks and Unknowns

### Risk 1: `boolean` type in Drizzle schema
The existing `schema.ts` does not import or use `boolean` from `drizzle-orm/pg-core`. The `silos.enabled` column is specified as `BOOLEAN`. Either add `boolean` to the Drizzle import and use it, or use `integer` (consistent with existing patterns). Functionally equivalent — the migration SQL uses `BOOLEAN` regardless of Drizzle type.

### Risk 2: `porter-user-prompt.js` hook output blocking
Claude CLI's `UserPromptSubmit` hook block mechanism (`{ "decision": "block" }`) is documented in Claude Code hooks documentation but not tested in this codebase. If the block decision format is wrong, the `/silo` command will pass through to the model as a prompt. Verify the exact JSON format against Claude Code hook docs before shipping.

### Risk 3: SessionStart hook doesn't read stdin today
`porter-session-start.js` currently calls `main()` directly without reading stdin first. The `SessionStart` event payload (containing `session_id`, `cwd`, etc.) is available on stdin but is never consumed. Modifying the hook to read stdin adds async complexity and a risk of timing issues. If stdin reading fails, the hook must still output something (it falls back to minimal header today). The refactor must preserve the fallback.

### Risk 4: `cwd` parameter and project detection
The `detectProject()` function in `porter-session-start.js` uses a regex on `process.cwd()` to extract the project name, not the full fs_path. If a project's `fs_path` in the DB doesn't match the cwd regex pattern, the project-type detection path will miss it. For the success criterion tests, the cwd_markers fallback is the actual detection path — project-type lookup is a secondary enrichment.

### Risk 5: 16 active session_registry rows
There are 16 active sessions in `session_registry`. The `session_silo_overrides` table keys on Claude CLI session_id (from hook), which is different from dispatch session IDs. These are separate namespaces — no collision risk, but it means the override lookup must use the CLI session_id consistently across the hook, the context endpoint, and the silo-command endpoint.

### Risk 6: Memory pruner dedup path
`memory-pruner.ts` dedup logic (lines ~100-165) checks similarity >= 0.85 across active directives in the same scope. The 5 `moe-direct` silo directives are all in `scope='silo'`, `scope_id='software'`. If two of them have high text similarity, the pruner could mark one as `superseded`. The trigger (from Investigation 9) blocks the resulting `UPDATE` or `DELETE`, which would cause the pruner to throw. The trigger must be BEFORE UPDATE with a guard on `OLD.source_type`, not a broad block on all directives.
