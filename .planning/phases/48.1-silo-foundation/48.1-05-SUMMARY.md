## Decisions Made

- **Wave 0 ordering:** Smoke ships before any implementation plan so each downstream task has a single-shot `<verify>` command. Pre-implementation runs surface the correct red signals (table missing, trigger missing, endpoint missing); post-implementation runs go green without script edits.
- **Graceful degradation over hard-fail:** The script does not require the Fastify backend to be running. Schema checks (SC-1, SC-6) execute via psql directly; API checks (SC-2..SC-5) are gated behind `curl -sf $API/health` and exit 0 with a warn line if backend is offline. This makes the script usable from day 1, including in plans that haven't yet restarted the service.
- **Row-existence guard on SC-6 (Plan-mandated):** Without the guard, an UPDATE/DELETE matching zero rows returns success — falsely indicating the trigger blocked the change. Guard ensures the target seed row IS present before attempting the trigger test.
- **SC-4b backward-compat (Plan-mandated):** Added between SC-4 and SC-5 to verify DRM-03 null-return path — pre-Plan-04 callers omitting `cwd` parameter must see zero silo sections.
