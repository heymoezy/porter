## Decisions Made

- **Lazy-load + warmup over startup-required:** `ensureCacheLoaded()` inside `detectSilos` covers cold start (e.g. if startup warmup fails). `loadSiloCache(pool)` at boot is purely a perf optimization — wrapped in `.catch()` to be non-fatal.
- **24h override TTL enforced in SQL (`set_at > NOW() - INTERVAL '24 hours'`)** instead of JS-side filter — keeps the predicate together with the row lookup, no need for a separate cleanup pass on read.
- **`silo_id IS NULL` semantics:** an override row with `silo_id IS NULL` means "explicit none, do not detect" and returns [] immediately. Absence of an override row means "fall through to detection". The distinction matters for `/silo none` in Plan 03.
- **Multi-silo aggregation via `Map<id, DetectedSilo>`:** dedups silos that match via both project-type AND cwd_markers, so a future config that overlaps detect rules can't double-inject.
- **Silo section gets a level-2 heading (`## Silo: ...`) while System/Project Directives use level-3 (`### System Directives`):** matches the CONTEXT.md spec. This makes the silo section visually distinct from generic directive sections in the rendered markdown — silos are a top-level concern, not just another directive bucket.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] SC-2 smoke directive-body grep was failing on JSON-escaped quotes**

- **Found during:** Task 3 (smoke harness run)
- **Issue:** The smoke harness extracted `CTX_PORTER` from raw `curl` JSON output and ran `grep -qF "$SNIPPET"` over it. The directive `silo-sw-compact-means-padding` starts with literal `"Compact"` (containing `"` characters). When Fastify serialized the response, `"` became `\"` in the JSON encoding, so `grep -qF` of the raw DB content (with unescaped quotes) failed even though the directive WAS present and correct in the rendered markdown that CLI consumers actually read.
- **Fix:** Replaced the raw-grep with `node -e "...JSON.parse(...).data.context"` extraction before grepping. This tests the rendered markdown the CLI consumes, not its on-wire JSON encoding — semantically more correct.
- **Files modified:** `tests/smoke-48.1.sh`
- **Verification:** Re-ran `bash tests/smoke-48.1.sh` — SC-2 now reports `[ ok ] SC-2: all 5 seed directive bodies present in context`. SC-4 / SC-4b still pass with their original raw-grep approach (they check for absence of `## Silo:`, which never contains characters that get JSON-escaped).
