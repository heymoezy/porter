## Dormant SSE Bug — Fixed As Side Benefit

The previous useAdminSSE.ts used `es.onmessage = (event) => { switch(payload.type ?? payload.event) {...} }`. Backend sse-hub.ts:22 writes:

```
event: <topic>
data: <json>
```

This is the "named event" SSE format. EventSource.onmessage ONLY fires for UNNAMED events (`data: ...` with no `event:` line). So **every existing switch case in the old hook never fired against the live backend.** All bridge:dispatch/circuit-trip/intelligence/msg-bus/etc. invalidations were dead code — the cache was only refreshing via manual `Refresh` clicks, `refetchOnWindowFocus`, or page navigation.

The refactor uses `es.addEventListener('topic', handler)` per topic, which correctly receives named SSE events. After this plan ships and Porter restarts (Plan 05), **all existing SSE-driven UX (bridge dispatch log live-prepend, capacity bars, gateway cards, msg-bus feed) starts working for the first time**, in addition to the 3 new dreams topics.

Plan 03's spec required a Select primitive, so the page is correct; the test needed the Radix-aware pattern: `locator('#silo-filter').click() → page.getByRole('option').click()`.
