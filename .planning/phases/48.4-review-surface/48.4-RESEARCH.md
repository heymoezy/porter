## Summary

Phase 48.4 closes the silo-scoped dream loop. Substrate is fully in place: 48.3 shipped `memory_proposals` + `dream_runs` with the exact index 48.4 needs (`memory_proposals(silo_id, status, created_at DESC)` — DRW-12 read contract), the placeholder `dream_run` handler now invokes `runDreamWorker` end-to-end, and the manual-trigger endpoint at `POST /api/v1/intellect/dream-run` lives at 127.0.0.1. The only consumers left are (a) admin endpoints under `/api/admin/dreams/*` that mirror the `/api/admin/evolution/*` proposal-review precedent, (b) a new React Router page at `/dreams` plugged into the existing sidebar Ops group next to Intellect, (c) accept-handler transactional logic that fans by `proposal_kind` and respects the `directive_immutable_moe_direct` trigger, (d) an `every_24h` workflow row + handler that flips stale `pending` rows to `expired`, and (e) SSE broadcasts on three event topics (`proposals:created`, `proposals:resolved`, `dreams:run-completed`) wired into the existing `sse-hub.ts` + `useAdminSSE` invalidate-on-event pattern.

No schema changes, no new tables, no new third-party dependencies. The accept handler is a single PostgreSQL transaction that (1) re-reads the proposal under `FOR UPDATE`, (2) re-reads the target directive(s) under `FOR UPDATE`, (3) executes the kind-specific INSERT/UPDATE/DELETE against `directives` (the trigger blocks seed-mutation paths by construction), (4) stamps `memory_proposals.status='accepted'` + `reviewed_at` + `reviewed_by`, (5) writes an `intellect_events` row with kind `proposal_accepted`, and (6) commits — at which point the `broadcast('proposals:resolved', ...)` fires so all open admin tabs invalidate their React-Query caches. Reject is the same shape minus the directives mutation.

**Primary recommendation:** Five plans — (01) Wave 0 smoke + Playwright scaffold, (02) backend admin routes + accept handler matrix + SSE broadcasts + auto-expiry workflow row, (03) frontend Dreams list page + silo filter + dream-runs sidebar, (04) frontend proposal detail panel + accept/reject mutation + evidence drawer, (05) ship + version v6.17.0 + Playwright green + human-verify the full loop (manual dream-run → proposal appears in UI → accept → directive lands → next CLI session injects). Treat the route as `/dreams` (singular noun "Intellect" is taken by `/intelligence`; the dreams surface is a peer of it). The frontend reuses shadcn `Table`, `Card`, `Badge`, `Dialog`, `Select`, `Button`, `Drawer` (already present), the existing `useAdminSSE` hook (extended with three new switch cases), and the existing `api()` helper. No new components except a small `<ProposalKindBadge/>` and a `<DiffBlock/>` for merge/supersede previews.

## User Constraints (from CONTEXT.md)

**No CONTEXT.md exists for Phase 48.4.** This phase was spawned directly from the ROADMAP entry. ### Locked Decisions (inherited from spec + predecessor phases)

- **No auto-accept.** Every proposal goes through human review. Status transitions are one-way: `pending` → (`accepted` | `rejected` | `expired`). Never silently flip back.
- **Seed immutability is a hard guard.** Directives with `source_type='moe-direct'` are sealed by the `directive_immutable_moe_direct` Postgres trigger (installed in 48.1). Accept-handler MUST NOT attempt `delete`/`supersede`/`merge` against them — pre-flight check rejects the operation with `error_message='SEALED_SEED'` BEFORE the trigger fires (defense in depth; the trigger is the final safety net).
- **Silo-scoped review.** UI filters by `silo_id` (defaults to `software` since it's the only enabled silo). No cross-silo bulk-accept. No multi-silo accept transactions.
- **Refinement Doctrine #2 (silence on stale = failure mode).** UI must SHOW expired proposals separately (not just hide them) so Moe can see what the Worker proposed that he ignored. Expired ≠ rejected — they're audit rows.
- **Transactional integrity.** Each accept/reject is atomic. No partial accept (e.g., merge that updates one target but fails on another) ever lands. Either the whole accept succeeds (proposal status flips + directive lands + audit event written) or nothing changes.
- **Auto-expiry default 30 days.** Worker stamps `expires_at = created_at + 30 * 86400` per 48.3 schema lock. Sweep runs daily (`every_24h` tag, already a valid scheduler bucket per workflow-engine.ts) and flips `pending` rows past expiry to `expired`. NEVER deletes the row — expired rows remain queryable.
- **Real-time updates via SSE.** No polling. Per Porter Admin CLAUDE.md: "No polling — use SSE for real-time updates, fetch once otherwise". The list page subscribes to `useAdminSSE` and invalidates React-Query caches on proposal-related events.
- **Admin-only.** All `/api/admin/dreams/*` routes guarded by `requirePlatformAdmin` preHandler (same posture as `/api/admin/evolution/proposals/:id/approve`).
- **Audit posture.** Every accept and every reject writes an `intellect_events` row. Kind = `proposal_accepted` | `proposal_rejected` | `proposal_expired`. Details include proposal_id, dream_run_id, silo_id, reviewer username, target directive ids touched. NO turn content in the audit row (privacy — Refinement Doctrine PII discipline carries through).
- **Cross-silo bleed forbidden.** A proposal with `silo_id='software'` may only target `directives` rows where `scope='silo' AND scope_id='software'`. Accept handler validates this invariant before mutation. Mismatched target rejected with `error_message='SILO_MISMATCH'`.

## Admin API Endpoints

All routes under `/api/admin/dreams/*`, registered in `backend/src/routes/admin/dreams.ts`, prefix configured in `backend/src/routes/admin/index.ts`. Guard: `fastify.addHook('preHandler', fastify.requirePlatformAdmin)` (existing pattern across every other admin route).

### `GET /api/admin/dreams/proposals`

**Query params:**
- `silo_id?: string` — filter (default: no filter, but UI defaults to `software`)
- `status?: string` — filter (`pending` | `accepted` | `rejected` | `expired`)
- `dream_run_id?: string` — filter to a specific run
- `limit?: number` — max 200, default 50
- `offset?: number` — default 0

**Response (200):**
```typescript
ok({
  proposals: Array<{
    id: string;
    dream_run_id: string;
    silo_id: string;
    proposal_kind: 'merge' | 'supersede' | 'delete' | 'new_directive';
    target_directive_ids: string[];
    proposed_content: string;
    proposed_metadata: Record<string, unknown>;
    source_evidence: Record<string, unknown>;
    sort_order: number;
    status: string;
    created_at: number;
    expires_at: number | null;
    reviewed_at: number | null;
    reviewed_by: string | null;
  }>,
  count: number,        // current page size
  total: number,        // total matching rows (SELECT COUNT(*) ... WHERE filters)
  counts_by_status: {   // for the badge UI
    pending: number;
    accepted: number;
    rejected: number;
    expired: number;
  }
})
```

**Query (uses DRW-12 index `memory_proposals_silo_status_created_idx`):**
```sql
SELECT * FROM memory_proposals
WHERE silo_id = COALESCE($1, silo_id)
  AND status = COALESCE($2, status)
  AND (CASE WHEN $3::text IS NOT NULL THEN dream_run_id = $3 ELSE TRUE END)
ORDER BY created_at DESC, sort_order ASC
LIMIT $4 OFFSET $5;
```

### `POST /api/admin/dreams/proposals/:id/accept`

**Body:** `{}` (no body; reviewer username pulled from session)

**Response (200):**
```typescript
ok({
  proposal_id: string,
  status: 'accepted',
  directive_ids_touched: string[],   // ids of directives created/updated/archived
  intellect_event_id: string,
})
```

**Errors:**
- 404 `NOT_FOUND` — proposal id doesn't exist
- 409 `INVALID_STATE` — proposal is not in `pending` status
- 410 `TARGET_GONE` — referenced target directive(s) no longer exist
- 422 `SEALED_SEED` — target is `source_type='moe-direct'` (pre-flight before trigger)
- 422 `SILO_MISMATCH` — target directive's `scope_id` ≠ proposal's `silo_id`
- 500 `ACCEPT_FAILED` — unexpected DB error (transaction rolled back)

**Side effects:**
1. PostgreSQL transaction commits the directive mutation + proposal status flip + audit event
2. `broadcast('proposals:resolved', {proposal_id, status: 'accepted', silo_id, dream_run_id})` on commit

### `POST /api/admin/dreams/proposals/:id/reject`

**Body:** `{}` or optionally `{ reason?: string }` for future use (current handler stores in `intellect_events.details_json` if provided)

**Response (200):**
```typescript
ok({
  proposal_id: string,
  status: 'rejected',
  intellect_event_id: string,
})
```

**Errors:** 404 `NOT_FOUND`, 409 `INVALID_STATE`.

**Side effects:**
1. UPDATE `memory_proposals` (no transaction needed — single statement)
2. INSERT `intellect_events` row
3. `broadcast('proposals:resolved', {proposal_id, status: 'rejected', silo_id, dream_run_id})`

### `GET /api/admin/dreams/runs`

**Query params:**
- `silo_id?: string`
- `status?: string` — (`running` | `completed` | `failed`)
- `limit?: number` — max 100, default 20

**Response (200):**
```typescript
ok({
  runs: Array<DreamRunRow & {
    proposals_count: number;
    pending_count: number;
    accepted_count: number;
    rejected_count: number;
    expired_count: number;
  }>,
  count: number,
})
```

**Query (uses `dream_runs_silo_started_idx`):**
```sql
SELECT dr.*,
  (SELECT COUNT(*) FROM memory_proposals WHERE dream_run_id = dr.id) AS proposals_count,
  (SELECT COUNT(*) FROM memory_proposals WHERE dream_run_id = dr.id AND status='pending')  AS pending_count,
  (SELECT COUNT(*) FROM memory_proposals WHERE dream_run_id = dr.id AND status='accepted') AS accepted_count,
  (SELECT COUNT(*) FROM memory_proposals WHERE dream_run_id = dr.id AND status='rejected') AS rejected_count,
  (SELECT COUNT(*) FROM memory_proposals WHERE dream_run_id = dr.id AND status='expired')  AS expired_count
FROM dream_runs dr
WHERE silo_id = COALESCE($1, silo_id)
  AND status = COALESCE($2, status)
ORDER BY started_at DESC
LIMIT $3;
```

(Note: the correlated COUNT subqueries are cheap because `dream_run_id` is in `memory_proposals_run_sort_idx` so the planner uses an index-only count.)

## Accept Handler Matrix

The accept handler is a single PostgreSQL transaction. The shape is identical across all four `proposal_kind` values — only the inner directive mutation differs. The transaction acquires `FOR UPDATE` row locks on the proposal AND every target directive to prevent concurrent accepts.

### Transaction skeleton (canonical)

```typescript
// Source: pattern verified against backend/src/db/migrate-silos-v1.ts (lines 80-135)
// and backend/src/services/intellect/dream-worker.ts (insertProposalsTransactionally pattern)

const client = await pool.connect();
try {
  await client.query('BEGIN');

  // 1. Re-read proposal under row lock
  const proposalRes = await client.query(
    `SELECT * FROM memory_proposals WHERE id = $1 FOR UPDATE`,
    [proposalId]
  );
  if (proposalRes.rowCount === 0) {
    await client.query('ROLLBACK');
    return err('NOT_FOUND', 'Proposal not found');
  }
  const proposal = proposalRes.rows[0] as MemoryProposalRow;
  if (proposal.status !== 'pending') {
    await client.query('ROLLBACK');
    return err('INVALID_STATE', `Cannot accept proposal with status '${proposal.status}'`);
  }

  // 2. Re-read targets under row lock (zero-or-more, depending on kind)
  let targets: DirectiveRow[] = [];
  if (proposal.target_directive_ids.length > 0) {
    const tres = await client.query(
      `SELECT id, scope, scope_id, content, priority, source_type, status
       FROM directives WHERE id = ANY($1) FOR UPDATE`,
      [proposal.target_directive_ids]
    );
    if (tres.rowCount !== proposal.target_directive_ids.length) {
      await client.query('ROLLBACK');
      return err('TARGET_GONE', 'One or more target directives no longer exist');
    }
    targets = tres.rows;

    // 2a. Pre-flight: silo mismatch
    for (const t of targets) {
      if (t.scope !== 'silo' || t.scope_id !== proposal.silo_id) {
        await client.query('ROLLBACK');
        return err('SILO_MISMATCH', `Target ${t.id} is scope='${t.scope}' scope_id='${t.scope_id}'`);
      }
    }

    // 2b. Pre-flight: sealed seed (catches before the trigger fires; clearer error)
    if (proposal.proposal_kind !== 'new_directive') {
      for (const t of targets) {
        if (t.source_type === 'moe-direct') {
          await client.query('ROLLBACK');
          return err('SEALED_SEED', `Target ${t.id} is a sealed seed directive`);
        }
      }
    }
  }

  // 3. Kind-specific mutation (see matrix below)
  const touched: string[] = await applyProposalMutation(client, proposal, targets);

  // 4. Flip proposal status
  await client.query(
    `UPDATE memory_proposals
     SET status='accepted',
         reviewed_at=EXTRACT(EPOCH FROM NOW()),
         reviewed_by=$1
     WHERE id=$2`,
    [reviewerUsername, proposalId]
  );

  // 5. Audit event
  const eventId = 'ie_' + randomUUID();
  await client.query(
    `INSERT INTO intellect_events (id, event_type, source_type, details_json)
     VALUES ($1, $2, $3, $4::jsonb)`,
    [eventId, 'proposal_accepted', 'review_surface',
     JSON.stringify({
       proposal_id: proposalId,
       dream_run_id: proposal.dream_run_id,
       silo_id: proposal.silo_id,
       proposal_kind: proposal.proposal_kind,
       target_directive_ids_touched: touched,
       reviewer: reviewerUsername,
     })]
  );

  await client.query('COMMIT');

  // 6. SSE broadcast (post-commit — never inside the transaction; broadcast failures
  //    must not roll back a committed accept)
  broadcast('proposals:resolved', {
    proposal_id: proposalId,
    status: 'accepted',
    silo_id: proposal.silo_id,
    dream_run_id: proposal.dream_run_id,
  });

  return ok({ proposal_id: proposalId, status: 'accepted', directive_ids_touched: touched, intellect_event_id: eventId });
} catch (e) {
  await client.query('ROLLBACK');
  throw e;
} finally {
  client.release();
}
```

### Mutation by `proposal_kind`

| Kind | What `applyProposalMutation` does | Returns (touched ids) |
|------|-----------------------------------|----------------------|
| `new_directive` | `INSERT INTO directives (id, scope, scope_id, content, priority, source_type, status, created_by) VALUES ('d_' \|\| randomUUID(), 'silo', proposal.silo_id, proposal.proposed_content, COALESCE(proposed_metadata->>'priority', 50)::int, 'dream_worker', 'active', reviewerUsername) RETURNING id;` | `[newDirectiveId]` |
| `supersede` | One target. `UPDATE directives SET content = proposal.proposed_content, priority = COALESCE(proposed_metadata->>'priority', priority)::int, updated_at = EXTRACT(EPOCH FROM NOW()) WHERE id = targets[0].id;` (immutable trigger blocks if seed — but pre-flight already caught) | `[targets[0].id]` |
| `merge` | Two-or-more targets. `INSERT` a single new directive with `proposed_content` AND `UPDATE` each target SET `status='archived'`. Net effect: many-to-one consolidation. | `[newDirectiveId, ...targets.map(t=>t.id)]` |
| `delete` | One target. `UPDATE directives SET status='archived', updated_at = EXTRACT(EPOCH FROM NOW()) WHERE id = targets[0].id;` (soft-delete; trigger blocks if seed — pre-flight caught) | `[targets[0].id]` |

**Why soft-delete on `delete` kind:** The `directives` table already has `status='active'`/`'archived'` (verified in schema.ts line 769). Hard-DELETE would break audit joins (the `intellect_events` audit row references the directive id). Archiving keeps the row queryable; the silo-injection query already filters `WHERE status='active'` so archived rows simply stop being injected.

**Why `target_directive_ids` validation:** The Worker stamps the ids at proposal-creation time, but accept may fire days later. By the time Moe clicks Accept, a target may have been deleted, merged, or archived by some other path. Pre-flight check catches that explicitly with `TARGET_GONE` rather than a confusing partial result.

**Concurrency:** `FOR UPDATE` on both the proposal row AND every target directive serializes any concurrent accept attempts. If two admin tabs both click Accept simultaneously, the second one blocks until the first commits, then sees `status='accepted'` and returns `INVALID_STATE`. Correct.

## Reject Handler

Simpler than accept — no directive mutation, no transaction needed.

```typescript
// 1. Re-read with status check
const existing = await queryOne<{ status: string; silo_id: string; dream_run_id: string }>(
  `SELECT status, silo_id, dream_run_id FROM memory_proposals WHERE id = $1`, [proposalId]
);
if (!existing) return err('NOT_FOUND', 'Proposal not found');
if (existing.status !== 'pending') return err('INVALID_STATE', `Cannot reject proposal with status '${existing.status}'`);

// 2. UPDATE
await execute(
  `UPDATE memory_proposals
   SET status='rejected', reviewed_at=EXTRACT(EPOCH FROM NOW()), reviewed_by=$1
   WHERE id=$2`,
  [reviewerUsername, proposalId]
);

// 3. Audit
const eventId = 'ie_' + randomUUID();
await execute(
  `INSERT INTO intellect_events (id, event_type, source_type, details_json)
   VALUES ($1, 'proposal_rejected', 'review_surface', $2::jsonb)`,
  [eventId, JSON.stringify({ proposal_id: proposalId, dream_run_id: existing.dream_run_id, silo_id: existing.silo_id, reviewer: reviewerUsername })]
);

// 4. Broadcast
broadcast('proposals:resolved', { proposal_id: proposalId, status: 'rejected', silo_id: existing.silo_id, dream_run_id: existing.dream_run_id });
return ok({ proposal_id: proposalId, status: 'rejected', intellect_event_id: eventId });
```

Single-statement update means no race window between the SELECT and UPDATE that matters — the second attempt simply sees `status='rejected'` already and returns `INVALID_STATE`.

## Auto-Expiry

### Strategy

1. **Default expires_at:** Already set at proposal creation by the worker per 48.3 lock (`created_at + 30 * 86400`). 48.4 doesn't change that.
2. **Sweep cadence:** Daily — `every_24h` schedule tag (already a valid scheduler bucket in `scheduler.ts`).
3. **Sweep handler:** New `memory_proposals_expire` action type registered in `workflow-engine.ts` `actionHandlers` map.
4. **Sweep behavior:** Flip `pending` rows where `expires_at < EXTRACT(EPOCH FROM NOW())` to `expired`. Write one `intellect_events` row with the swept count. Broadcast one SSE event.

### New workflow row

Added via migration script `backend/src/db/migrate-dreams-v2.ts` OR more idiomatically via `BUILTIN_WORKFLOWS` array in `workflow-engine.ts` (which seeds idempotently at startup). Recommendation: `BUILTIN_WORKFLOWS` only (no migration row needed) since 48.3-02 established the precedent of BUILTIN being the seed of record for dream-related workflows.

```typescript
// In workflow-engine.ts BUILTIN_WORKFLOWS array

  name: 'Expire stale memory proposals',
  trigger_type: 'schedule',
  trigger_value: 'every_24h',
  action_type: 'memory_proposals_expire',
  action_config: {},
  enabled: true,
}
```

### Handler

```typescript
// In workflow-engine.ts actionHandlers
memory_proposals_expire: async () => {
  const result = await pool.query(
    `UPDATE memory_proposals
       SET status='expired'
     WHERE status='pending'
       AND expires_at IS NOT NULL
       AND expires_at < EXTRACT(EPOCH FROM NOW())
     RETURNING id, silo_id`
  );
  const expired = result.rowCount ?? 0;
  if (expired > 0) {
    await logIntellectEvent('proposals_expired', 'memory_proposals_expire', {
      count: expired,
      ids: result.rows.map(r => r.id).slice(0, 20),  // cap evidence at 20 ids
    });
    broadcast('proposals:resolved', { event: 'expired', count: expired });
    console.log(`[memory_proposals_expire] flipped ${expired} stale proposal(s) to expired`);
  }
  return { expired };
},
```

**Index used:** `memory_proposals_expiry_idx ON memory_proposals(status, expires_at) WHERE status='pending'` — partial index already provisioned by 48.3-02. The sweep is index-only.

**Why no migration row alongside the BUILTIN entry:** Per 48.3-02 decisions ("BUILTIN_WORKFLOWS seeds are belt-and-braces with migrate-dreams-v1.ts seeds — both idempotent by name"), BUILTIN alone is sufficient when no schema change is needed. A v2 migration would be empty except for the workflow seed — overkill.

| Failure | Backend response | UX |
|---------|------------------|-----|

| Accept clicked on proposal whose target directive vanished | 410 `TARGET_GONE` | Toast: "Target directive no longer exists — proposal removed". Invalidate list. Reject the row instead (operator decision via a follow-up confirm). |
| Accept clicked on proposal targeting a sealed seed | 422 `SEALED_SEED` | Toast: "Seed directives are protected — cannot accept this proposal". | Accept clicked on proposal with silo mismatch | 422 `SILO_MISMATCH` | Toast: "Proposal targets wrong silo — review needed". Same posture as SEALED_SEED — should never happen, surface as an alert. |
| Reject clicked on already-resolved proposal | 409 `INVALID_STATE` | Toast: "Already rejected/accepted". Invalidate list. |
| Network error / 500 on accept | 500 `ACCEPT_FAILED` | Toast: "Accept failed — please retry". Optimistic-update rollback (React Query handles this via mutation `onError`). Transaction has already rolled back server-side; safe to retry. |

| `Run Now` dispatch failure (Bridge timeout) | The worker writes `dream_runs.status='failed'` + intellect_event with error_message | SSE event `dreams:run-completed` with `status: 'failed'` triggers a toast showing the error_message |

## Audit Posture

Every state transition writes one `intellect_events` row. The table schema (verified in `services/intellect/file-watcher.ts:179`):

```sql
intellect_events (id text PK, event_type text, source_type text, details_json jsonb, created_at)
```

| Event type | Source | Details payload |
|-----------|--------|----------------|
| `proposal_accepted` | `review_surface` | `{proposal_id, dream_run_id, silo_id, proposal_kind, target_directive_ids_touched, reviewer}` |

**Privacy:** No turn content, no proposed_content body, NO `source_evidence` content lands in audit rows. The audit row is the metadata trail; the proposal row itself remains queryable for full context.

```typescript
memory_proposals_expire: async () => {
  const result = await pool.query(
    `UPDATE memory_proposals
       SET status='expired'
     WHERE status='pending'
       AND expires_at IS NOT NULL
       AND expires_at < EXTRACT(EPOCH FROM NOW())
```

### Pattern: Transactional admin mutation with row-level locking + post-commit broadcast

**What:** Wrap any multi-statement state change in `BEGIN`/`COMMIT` with `FOR UPDATE` row locks on every row read. Broadcast SSE events ONLY after commit succeeds — never inside the transaction.

**When to use:** Accept handler. Any future "apply N changes as one" admin operation.

**Why:** Atomicity (no partial state), concurrency-safety (no double-accept races), audit consistency (the intellect_events row only exists if the mutation committed), broadcast safety (a failed SSE write doesn't roll back a committed mutation).

### Pattern: SSE topic + React Query cache invalidation

**What:** Backend `broadcast('domain:event', payload)`. Frontend `useAdminSSE` switch invalidates relevant query keys. Components re-fetch on stale + revalidate.

**When to use:** Any admin surface that needs to reflect cross-tab / cross-session state changes in real-time.

**Why:** No polling, no manual refresh, no stale UI. Aligns with Admin CLAUDE.md "No polling" rule.

### Anti-Patterns to Avoid

- **Broadcast inside the transaction:** If the COMMIT fails, the SSE event has already fired and clients invalidate to read state that doesn't exist. Always broadcast post-commit.
- **Hard DELETE on directives:** Breaks audit join paths. Use `status='archived'`.
- **Per-tab SSE channel:** Existing fanout works. Don't multiply EventSource connections.
- **Optimistic UI on accept:** The accept handler has multiple failure modes (TARGET_GONE, SEALED_SEED, SILO_MISMATCH). Optimistic update + later rollback is confusing. Disable the button while the mutation pends; let React Query's `isPending` state handle the spinner.
- **Polling fallback "just in case":** The user constraint locks SSE. Manual refresh button is the fallback. Don't add a 30s background poll "for safety."
- **Editing proposed_content in the UI:** Deferred. Don't sneak it in.

## Common Pitfalls

### Pitfall 1: SSE broadcast before COMMIT
**What goes wrong:** `broadcast('proposals:resolved', ...)` fires, frontend invalidates queries, refetches, sees the proposal still `pending` because the COMMIT hasn't landed yet → UI shows stale state and "ghost" status that flips back.
**Why it happens:** Code organization temptation — broadcast inline with the UPDATE for visual locality.
**How to avoid:** Always broadcast AFTER `client.query('COMMIT')` in the try block, never inside.
**Warning signs:** SSE event has the new status but a fresh GET returns the old status. Race lasting < 50ms but visible in dev tools.

### Pitfall 2: Forgetting `FOR UPDATE` on target directives
**What goes wrong:** Two admin tabs simultaneously accept proposals targeting the same directive. Both succeed, both INSERT a `merge`'s new directive, both archive the original — winding up with two new directives + an archived old one. Or worse, two `supersede` operations write competing content and the last writer wins silently.
**Why it happens:** Looking at the proposal row lock and forgetting the directive rows are also mutable from concurrent sessions.
**How to avoid:** ALL rows read in the transaction that will be MUTATED must be SELECTed with `FOR UPDATE`. The proposal row AND every target directive.
**Warning signs:** Production has two directive rows with the same scope/scope_id and very similar content created seconds apart. Two `intellect_events` rows referencing the same proposal_id with different target_directive_ids_touched arrays.

### Pitfall 3: `target_directive_ids` array type mismatch in `pg`
**What goes wrong:** The column is `TEXT[]`. node-postgres returns it as `string[]` on read but expects an array literal on write (`'{"a","b"}'`) — pass a JS array `['a','b']` and the driver handles it. Mixing JSON.stringify into the value position writes literal `"["a","b"]"` as a single text element. Then `id = ANY($1)` returns nothing.
**Why it happens:** Habit from JSON columns. JSONB takes `JSON.stringify(value)`; TEXT[] takes a raw array.
**How to avoid:** Never JSON.stringify a TEXT[] value. Pass the JS array directly: `await client.query('SELECT ... WHERE id = ANY($1)', [arrayOfIds])`.
**Warning signs:** `TARGET_GONE` errors on every accept even though the targets clearly exist in psql.

### Pitfall 4: Trigger fires with confusing error message
**What goes wrong:** Skip the pre-flight `source_type='moe-direct'` check, attempt to UPDATE/DELETE a sealed seed directive, the Postgres trigger fires with the technical message `directive_immutable_moe_direct: moe-direct directives are sealed (id=X, op=Y). Set LOCAL porter.allow_moe_direct_mutation=true to bypass.` — node-postgres throws an error that bubbles up as 500 ACCEPT_FAILED. User sees "Accept failed" toast with no actionable info.
**Why it happens:** Trusting the trigger as the only safety net.
**How to avoid:** Pre-flight check returns `422 SEALED_SEED` BEFORE the directive mutation runs. The trigger remains as defense-in-depth but should never fire from the accept-handler path under normal operation.
**Warning signs:** 500 ACCEPT_FAILED errors with stack traces mentioning the trigger function.

### Pitfall 5: Frontend silent failure on SSE disconnect during dispatch
**What goes wrong:** User clicks "Run Now". Backend dispatches. Frontend SSE briefly disconnects (network blip, reconnect, etc.). The `dreams:run-completed` event fires while the connection is down. EventSource reconnects but missed events are NOT replayed. UI never updates.
**Why it happens:** SSE has no message queue / replay. Lost events are lost.
**How to avoid:** Belt-and-braces: a 30s `refetchInterval` on the runs query as a fallback, OR a "if any run is `running` in the latest fetch, poll every 5s until it isn't" pattern. The simplest answer: trust SSE for the happy path but offer a manual Refresh button.
**Warning signs:** "It's running but the UI says it isn't" reports. Run completes in the DB but no SSE arrival noted in browser console.

### Pitfall 6: Toast spam on bulk expiry
**What goes wrong:** Auto-expiry sweep flips 47 stale proposals. Backend fires `broadcast('proposals:resolved', {event:'expired', count: 47})`. Frontend matches the `proposals:resolved` case and... fires a toast for every connected admin tab. If the bulk handler accidentally fires per-row, 47 toasts.
**Why it happens:** Confusing single-row and bulk topic payloads.
**How to avoid:** Single broadcast per sweep (one event), and in the frontend handler, branch on payload shape: `if (payload.event === 'expired') showToast(`${payload.count} proposals expired`) else if (payload.proposal_id) {...}`.
**Warning signs:** Multiple toasts stacking after running the sweep handler manually.

### Pitfall 7: `dispatch_id` join returns 0 rows
**What goes wrong:** `GET /api/admin/dreams/runs/:id` joins `bridge_dispatch_log` on `dispatch_id`. The dream worker writes the dispatch_id on a best-effort basis (per 48.3-04 design — "captures it explicitly via logDispatch"). If the dispatch failed or logging failed, the column is NULL. Frontend renders an empty dispatch block.
**Why it happens:** `dispatch_id` is nullable by design.
**How to avoid:** UI handles `dispatch === null` explicitly (show "No dispatch logged" or omit the section). Don't assume it's always present.
**Warning signs:** Frontend errors on `dispatch.tokens_in` reading null property.
