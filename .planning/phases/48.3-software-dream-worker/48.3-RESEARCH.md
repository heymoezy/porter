# Phase 48.3: Software Dream Worker — Research

**Researched:** 2026-05-13
**Domain:** Memory consolidation — model-driven proposal synthesis from silo-tagged transcript corpus, refine-don't-append doctrine, Bridge dispatch with raw passthrough, weekly workflow-engine scheduling, structured JSON output parsing into `memory_proposals`
**Confidence:** HIGH (all integration points verified against live code, live DB rows, and the locked dreams-pipeline spec; the only LOW items are deferred 48.4 UI shape decisions which are explicitly out of scope)

## Summary

Phase 48.3 is the consciousness layer of the Dream Silos series. Substrate is now in place: 48.1 shipped the `silos` registry + `/silo` command + immutable seed-directive trigger, and 48.2 shipped `session_transcript_turns` populated by hooks (live: **661 rows, 633 silo=software, 42 distinct software sessions in the last 7 days** as of research date). This phase wires the *worker* that turns those turns into structured `memory_proposals`.

The worker reads a bounded window of `silo='software'` turns, fetches the current active software-silo directives, dispatches both to a strong reasoner via `POST /api/v1/chat/stream` with `raw: true` (the v6.15.0 passthrough flag that suppresses Memory V3 injection — exactly the leak-prevention contract we need), parses a strict JSON response into a list of proposals, **enforces the refinement doctrine at INSERT time** (delete/supersede/merge proposals are persisted with `sort_order < new_directive` for the same conceptual area), and stamps every row with a `dream_run_id` so 48.4 can group, accept, reject, or expire them atomically.

Two new tables, one new service module (`dream-worker.ts`), one new prompt template (`software.md` — referenced by `silos.prompt_path` since 48.1 but never written), one new workflow scheduler tag (`every_week`), one new manual-trigger admin endpoint, and one shared Bridge dispatch helper. No new dependencies. No new scheduler primitives — extend the existing tick loop with a weekly bucket that mirrors the established `runScheduledWorkflows(tag)` pattern.

**Primary recommendation:** Build `memory_proposals` + `dream_runs` as twin tables (run is the parent, proposals are children). Use the existing `selectWithFallback` non-streaming dispatch path on the routing engine — it's the cleanest "one big prompt → one big answer" shape and avoids SSE plumbing in a worker context. Force `claude_cli` (Sonnet 4.6 default per silo row, Opus 4.7 selectable for manual deep runs) via `ctx.forceModelName`. Enforce strict JSON output via prompt + parse-or-fail with proposal-level error rows. Cap context window at ≤ 200 KB of transcript text (≈ 50K tokens) using a deterministic top-N sample by length + recency stratification. Run weekly, Sunday 03:00 SGT, via a new `every_week` schedule tag in `runScheduledWorkflows`.

## User Constraints (from CONTEXT.md)

**No CONTEXT.md exists for Phase 48.3.** This phase was spawned directly from the ROADMAP entry. The constraints below are inherited verbatim from the canonical spec (`research/porter-dreams-pipeline.md`) and locked decisions of Phases 48.1 + 48.2.

### Locked Decisions (inherited from spec + predecessor phases)

- **Refinement Doctrine #1: Refine, don't append.** Output kinds are biased: `merge` / `supersede` / `delete` ENCOURAGED, `new_directive` is LAST RESORT. The prompt template DEMANDS dedup/delete output before allowing additions. Active-directive count per silo is a tracked health metric — monotonic growth = alarm.
- **Refinement Doctrine #2: Authority to delete with judgment.** Worker proposes deletions for contradictions, staleness (≥6 weeks no reinforcement + no recent evidence), supersession, redundancy (same rule, 3+ active phrasings). Human-review queue is mandatory in v1 — no auto-delete. But silence on stale data is a FAILURE MODE, not a safety feature.
- **Refinement Doctrine #3: Reinforcement learning from every CLI prompt.** Repetition (3+ restatements, different phrasing, same intent) → reinforcement. Contradiction (recent prompts disagree with active rule) → propose supersession with evidence turn IDs. Imperative phrasing patterns (`always X` / `never Y` / `stop doing Z` / `it should be W`) → directive candidates. **Asymmetric: a single fresh contradiction outranks five older confirmations.**
- **Refinement Doctrine #4: Per-prompt silo classification (capture-time + synthesis-time).** Capture-time silo tag is heuristic (already done by 48.2). Synthesis-time, the Worker re-filters: any turn whose model-judged silo ≠ 'software' is discarded from the corpus before proposal extraction. Mistagged turns get corrected; mixed-silo sessions get split per-turn.
- **Seed immutability.** Hand-curated seed directives (`source_type='moe-direct'`, e.g. the 4 software-silo seeds) are immutable to the Worker. It can FLAG them for Moe's review but cannot propose deletion. The DB trigger `directive_immutable_moe_direct` (installed by 48.1) is the hard guard.
- **Cadence: weekly + manual.** Software silo runs once per week. Manual trigger always available via admin endpoint.
- **Model: Sonnet 4.6 nightly default, Opus 4.7 on manual deep runs.** Configurable per-silo via `silos.default_model`, overridable per-run via manual-trigger payload.
- **No auto-accept.** Every proposal goes through 48.4's human-review queue.
- **Privacy: `/silo none` blocks both injection AND capture.** Already enforced by 48.2's capture endpoint. The Worker reads only `silo_id='software'` rows, so `/silo none` rows (silo_id NULL) are invisible by construction.

## memory_proposals Schema

```sql
CREATE TABLE memory_proposals (
  id                  TEXT PRIMARY KEY,                       -- 'mp_' || randomUUID()
  dream_run_id        TEXT NOT NULL,                          -- FK to dream_runs.id (no DB-level FK; orphan-tolerant)
  silo_id             TEXT NOT NULL,                          -- 'software' for this phase; future-proof for legal/admin/etc
  proposal_kind       TEXT NOT NULL,                          -- 'merge' | 'supersede' | 'delete' | 'new_directive'
  target_directive_ids TEXT[] NOT NULL DEFAULT '{}',           -- FK semantics to directives.id: 0 for new_directive, 1 for delete/supersede, 2+ for merge
  proposed_content    TEXT NOT NULL,                          -- the new/merged/superseding directive content; for delete, a justification narrative
  proposed_metadata   JSONB NOT NULL DEFAULT '{}'::jsonb,     -- {"priority": 70, "source_type": "dream_worker", "conceptual_area": "design-system"}
  source_evidence     JSONB NOT NULL DEFAULT '{}'::jsonb,     -- {"sample_turn_ids": [123, 456], "phrasing_examples": ["always X", "never Y"], "reasoning": "..."}
  sort_order          INTEGER NOT NULL DEFAULT 0,             -- enforces refine-before-append at INSERT time (DRW-06); lower = applied first
  status              TEXT NOT NULL DEFAULT 'pending',         -- 'pending' | 'accepted' | 'rejected' | 'expired'
  created_at          DOUBLE PRECISION NOT NULL DEFAULT EXTRACT(EPOCH FROM NOW()),
  expires_at          DOUBLE PRECISION,                       -- typically created_at + 30 days; NULL = no expiry
  reviewed_at         DOUBLE PRECISION,
  reviewed_by         TEXT,                                   -- username (typically 'moe') on 48.4 accept/reject
  CHECK (proposal_kind IN ('merge','supersede','delete','new_directive')),
  CHECK (status IN ('pending','accepted','rejected','expired'))
);

CREATE INDEX memory_proposals_silo_status_created_idx
  ON memory_proposals(silo_id, status, created_at DESC);     -- 48.4 list-pending-by-silo
CREATE INDEX memory_proposals_run_sort_idx
  ON memory_proposals(dream_run_id, sort_order ASC);          -- group-by-run in 48.4 detail view
CREATE INDEX memory_proposals_expiry_idx
  ON memory_proposals(status, expires_at)
  WHERE status = 'pending';                                  -- partial index for the daily expiry sweep
```

**Column rationale:**

- `id` text PK with `mp_` prefix is consistent with directives (`d_*` prefix common) and easier to spot in logs than bare UUIDs.
- `dream_run_id` is **NOT** a hard FK at the DB level. Same orphan-tolerance precedent as `session_silo_overrides.silo_id` and `session_transcript_turns.silo_id` (48.2 anti-pattern note). If a dream_run is somehow lost, proposals are still queryable.
- `silo_id` is replicated on every proposal (denormalization) so 48.4 list queries can filter without join.
- `proposal_kind` is a single column with a CHECK constraint. Four values, no surprises. Drizzle enum-like patterns (TEXT + CHECK) are how Porter does it everywhere (`directives.status`, `episodes.status`, etc.).
- `target_directive_ids TEXT[]` handles all four cases uniformly: empty for `new_directive`, one element for `delete`/`supersede`, two-or-more for `merge`. PostgreSQL native array beats a junction table for this cardinality.
- `proposed_content TEXT` is the body of the proposed directive (for `new_directive`/`supersede`/`merge`) OR the justification narrative (for `delete`). 48.4 displays this verbatim. Cap at 8KB at the parser layer — defensive.
- `proposed_metadata JSONB` carries priority, source_type, conceptual_area (a coarse tag the model produces — used by DRW-06 to group refinement-vs-addition decisions), and any future fields. JSONB so we can extend without migration.
- `source_evidence JSONB` is the audit trail: which turns sourced this proposal, which phrasing examples, what reasoning. 48.4 displays this expanded. **Cap evidence at 16KB at parse time** — defensive.
- `sort_order INTEGER` is the refinement-discipline enforcer. The Worker computes it after parsing (NOT trusting the model to provide it). For each conceptual_area, refinements get lower sort_order than additions. 48.4 detail view orders by it.
- `status` lifecycle: `pending` → (`accepted` | `rejected` | `expired`). No back-transitions. 48.4 reject is terminal — the proposal becomes audit-only. New runs may re-propose the same idea; status='rejected' is never silently flipped.
- `expires_at DOUBLE PRECISION` (epoch seconds, matching the rest of Porter's timestamp dialect). Set to `created_at + 30 * 86400` at insert. 48.4 daily sweep flips `pending`→`expired` past that. NULL = never expires (admin can manually persist high-stakes proposals).
- `reviewed_at` + `reviewed_by` mirror `skill_evolution_proposals` semantics (Phase 35 precedent).

  completed_at             DOUBLE PRECISION,
  CHECK (status IN ('running','completed','failed')),
  CHECK (triggered_by IN ('schedule','manual'))

**Column rationale:**

- `triggered_by` discriminator distinguishes weekly cron from manual admin triggers in audit logs and 48.4's run-history view.
- `action_config JSONB` is the reproducibility record. Stores: `{cap_tokens, model_override, sample_size_override, dry_run, sampling_strategy_decisions: {...}, fixtures_used: bool}`. Critical for "why did the worker produce X last Sunday?" forensics.
- `prompt_token_estimate` and `response_token_estimate` are **estimates**, not authoritative counts. Bridge logs authoritative counts in `bridge_dispatch_log`. The estimates here are byte-length / 3 heuristics — good enough for a "did we blow the budget?" health check.
- `proposals_extracted` is denormalized for fast dashboard queries (avoids `COUNT(*)` on memory_proposals).
- `error_message` is populated on `status='failed'`. Plain text, 1KB cap. Examples: `"Bridge dispatch failed: 429 rate limit"`, `"JSON parse failed at position 1247: Unexpected token }"`, `"Doctrine violation: new_directive proposed without prior refinement (active dir count: 6, refinement proposals: 0)"`.
- `duration_ms` is wall-clock from `started_at` to `completed_at`. Useful for tracking model latency over time.

## Dream Prompt Template (software.md)

```markdown
# Software Silo Dream — Refinement Synthesis

You are reading the last 7 days of a developer's CLI prompts and assistant replies, captured in the `software` silo of a memory-consolidation system called Porter. Your job is to refine the system's operating rules — NOT to add new ones.

## Mission

Read the transcripts and the current active software-silo directives. Produce a list of **memory proposals** that make the rule set SHARPER and SMALLER, not bigger.

## The Refinement Doctrine

1. **Refine, don't append.** Your proposal list MUST be biased toward `merge`, `supersede`, `delete`. New directives are a last resort. If the active rule set fully covers the signal you see, your job is to propose deletions of stale rules — NOT to invent new ones.
2. **You may delete with judgment.** Propose `delete` for: directives contradicted by ≥3 recent prompts; directives with no reinforcement in the corpus AND no recent matching evidence; redundant phrasings of the same rule (collapse to one); directives superseded by a newer one (mark the OLD one for delete).
3. **Reinforcement is asymmetric.** A single fresh contradiction outranks five older confirmations. The most recent stated preference wins. Recency > frequency for conflicting signals.
4. **Re-filter for silo.** Some captured turns were tagged `software` by a fast heuristic but are not actually software work (e.g., legal review, vacation chat). DISCARD those from your reasoning. Only mine turns that exhibit software-development signal: code, file paths, type errors, ship/deploy verbs, design-system / component / UI discussion, infra, refactor, build, test.

## Hard Rules

- Sealed seeds: directives with `source_type: "moe-direct"` are immutable. You may NEVER propose `delete` or `supersede` on them. You may propose `new_directive` that COMPLEMENTS them. If you find a seed is contradicted by overwhelming evidence, surface that as a `flag_seed_for_human_review` field in your reasoning but do NOT emit a delete/supersede proposal for it.
- Software-only scope: every proposal MUST be about software development judgment (design, architecture, code style, build/ship/deploy patterns, tooling preferences). NOT about UI copy, legal text, fund management, or any non-software domain.
- One conceptual area per proposal. Don't bundle "use design system" with "compact means padding" — those are two proposals.
- Evidence required: every proposal MUST cite ≥2 turn IDs from the supplied transcripts. Proposals without evidence are invalid.

## Output

Return EXACTLY ONE JSON object. No markdown fence. No prose before or after. The JSON must validate against this schema:

```json

## Expected JSON Output Contract (parser-facing)

The Worker parses the model's response with `JSON.parse`. Validation steps:

1. **Top-level required fields:** `summary` (string), `proposals` (array, may be empty), `active_directive_count_before` (number), `active_directive_count_after_proposed` (number).
2. **Optional:** `flagged_seeds` (array of `{seed_directive_id, contradicting_turn_ids, note}`).
3. **For each proposal:** `kind` ∈ {merge,supersede,delete,new_directive}; `conceptual_area` (non-empty string ≤ 60 chars); `target_directive_ids` (string array — empty IFF kind='new_directive', length 1 for delete/supersede, length ≥2 for merge); `proposed_content` (non-empty string ≤ 8KB); `priority` (integer 1-100, defaults to 70 if missing); `source_evidence.sample_turn_ids` (array of integers, length ≥ 2); `source_evidence.phrasing_examples` (array of strings, length ≥ 1); `source_evidence.reasoning` (non-empty string ≤ 4KB).
4. **Doctrine validation:** if `active_directive_count_before > 4` (seed baseline) AND `proposals` contains any `kind='new_directive'` AND `proposals` contains zero of `kind ∈ {merge,supersede,delete}` → run is FAILED with error_message `"Doctrine violation: new_directive proposed without prior refinement"`.
5. **Sort-order assignment (post-parse, pre-insert):** group proposals by `conceptual_area`. Within each group, assign sort_order: 100 + n for `delete`, 200 + n for `supersede`, 300 + n for `merge`, 900 + n for `new_directive`. Across groups, conceptual_areas sort lexicographically. The Worker, NOT the model, owns sort_order. The model's job is to produce proposals; the Worker's job is to enforce refinement-before-append ordering.

**Why the Worker (not the model) sets sort_order:** Models drift. The DB-level enforcement of refine-before-append is non-negotiable and must not depend on the model honoring an instruction. Even if the model's prompt obedience is perfect, the Worker is the deterministic gatekeeper.

## Bridge Dispatch Shape

The Worker dispatches **non-streaming** through the existing routing engine. Code shape:

```typescript
// Source: backend/src/services/intellect/dream-worker.ts (NEW)
import { routingEngine } from '../bridge/routing-engine.js';
import type { BridgeDispatchRequest, RoutingContext } from '../bridge/types.js';

async function dispatchDream(
  promptBody: string,
  modelName: string,
): Promise<{ response: string; latencyMs: number; modelUsed: string }> {
  // Note: we do NOT build Memory V3 context, do NOT inject identity prefix,
  // do NOT call selectSkills, do NOT call decideDoctrine. The raw passthrough
  // contract is replicated *by what we don't do here*, not by an HTTP flag.
  const ctx: RoutingContext = {
    message: promptBody,                          // used for logging/scoring only — model sees only req.messages
    forceGatewayType: 'claude_cli',               // Sonnet 4.6 / Opus 4.7 live here
    forceModelName: modelName,                    // resolves on the gateway row
    // intentionally omit: agentId, projectId, chatId, skillsUsed, directiveStats, dispatchStrategy
  };
  const req: BridgeDispatchRequest = {
    messages: [{ role: 'user', content: promptBody }],
    model: modelName,
    temperature: 0.2,                             // determinism > creativity for refinement
    maxTokens: 16000,                             // enough for ~50 proposals worth of JSON
  };
  const { decision, result } = await routingEngine.selectWithFallback(ctx, req);
  return {
    response: result.response,
    latencyMs: result.latencyMs,
    modelUsed: decision.modelName,
  };
}
```

**Timeout:** `selectWithFallback` is wrapped in `withRetry` + per-gateway queue + circuit breaker. The `claude_cli` adapter has its own internal timeout (defaults to 300s for non-streaming). We add a hard outer timeout via `AbortSignal.timeout(180_000)` wrapping the call — fail the run with `error_message="Bridge dispatch timeout (180s)"` rather than wait forever.

**Why NOT the streaming endpoint:** The chat stream endpoint exists for SSE-rendering UIs. A worker that needs the full response before parsing JSON has no use for streaming. Non-stream returns `{response, latencyMs, ...}` in one promise. Cleaner.

**The "raw: true" semantics from chat.ts:305:** We're not calling the HTTP endpoint, we're calling the routing engine directly. The `raw` flag in `chat.ts` is a guard that says "don't run buildMemoryContext + skill selection + doctrine + identity prefix before dispatch." In the Worker, we simply DON'T call those services. Same effect. The "raw" semantics are achieved by *omission*. Document this clearly in the worker module's header comment so future devs don't add Memory V3 injection thinking they're being helpful.

**Backend choice:** `claude_cli` only. Codex CLI is a tool-using executor (not the right shape for "read a wall of text and produce JSON"). Ollama is too small for nuanced refinement reasoning. OpenClaw/GPT-5.4 is acceptable as a secondary configurable option but not the default — the spec locks Sonnet 4.6 / Opus 4.7. Force gateway type via `RoutingContext.forceGatewayType='claude_cli'`.

## Refine Doctrine Enforcement (DRW-06)

Three layers of enforcement, in order:

### Layer 1: Prompt instruction (model-side)
The prompt template TELLS the model to bias toward refinement and includes a self-check (`If active count > 4, my proposals include at least one merge/supersede/delete`). The model can fail this — it's instruction, not enforcement.

### Layer 2: Validation (Worker rejects bad runs)
After parsing, before insert, run `validateRefinementDoctrine(parsed, activeCountBefore)`:

```typescript
function validateRefinementDoctrine(parsed: ParsedDreamResponse, activeCountBefore: number) {
  // Doctrine rule: if active count > seed baseline (4), and we're adding new_directive proposals,
  // we must ALSO have at least one merge/supersede/delete proposal.
  const SEED_BASELINE = 4;
  if (activeCountBefore <= SEED_BASELINE) return; // OK to add when only seeds exist
  const hasNew = parsed.proposals.some(p => p.kind === 'new_directive');
  const hasRefinement = parsed.proposals.some(p => p.kind === 'merge' || p.kind === 'supersede' || p.kind === 'delete');
  if (hasNew && !hasRefinement) {
    throw new Error(`Doctrine violation: new_directive proposed without prior refinement (active dir count: ${activeCountBefore}, refinement proposals: 0). Worker rejecting run to enforce refine-before-append.`);
  }
}
```

This intentionally rejects the WHOLE run. We don't strip new_directive proposals and accept partial output — that would silently weaken the doctrine. Failed runs are visible to Moe (audit log + dream_runs.status='failed') so the pattern becomes obvious if the model is consistently noise-y.

### Layer 3: sort_order at insert (deterministic)
Even when validation passes, the Worker assigns sort_order to enforce *display ordering* in 48.4. Within each conceptual_area:
- `delete` proposals: sort_order = 100 + (n-th-in-area)
- `supersede`: 200 + (n-th)
- `merge`: 300 + (n-th)
- `new_directive`: 900 + (n-th)

48.4's run-detail view sorts by sort_order ASC, so refinements appear FIRST. This shapes the human reviewer's frame: "look at what's being removed before what's being added."

**Conceptual area extraction:** the model produces `conceptual_area` as a tag (e.g. `design-system`). The Worker groups by this raw string. If two proposals share an area, they sort against each other; otherwise they sort across areas lexicographically. No fuzzy matching — keep the Worker dumb and the model precise.

## Sampling Strategy (DRW-07)

The 7-day software-silo corpus is currently ~950 KB across ~2200 turns (extrapolated from the 633 turns in last 2 days @ ~1.5 KB/turn). Sonnet 4.6 can handle ~200K tokens (~800 KB) input, but we want to reserve at least 100K tokens for response + reasoning, so input is capped at **50K tokens ≈ 200 KB**.

### Algorithm (deterministic, reproducible)

```
1. Read all turns: SELECT * FROM session_transcript_turns
     WHERE silo_id='software' AND captured_at >= NOW() - INTERVAL '7 days'
     ORDER BY session_id, turn_index
   -- Use index transcript_turns_silo_captured_idx (silo_id, captured_at DESC)

2. Tag each turn with stratum:
     a) "today"   = captured_at >= NOW() - INTERVAL '1 day'
     b) "1-2d"    = NOW() - INTERVAL '2 days' <= captured_at < NOW() - INTERVAL '1 day'
     c) "3-7d"    = NOW() - INTERVAL '7 days' <= captured_at < NOW() - INTERVAL '2 days'
     d) "imperative" = matches /^(always|never|stop|do not|don't|it should|never use|always use|use only|don't use)\b/i
        (this is a TAG, not a stratum — a turn can be in "today" AND "imperative")

3. Compute target budget allocation:
     - 40% of 200KB = 80KB for "today"
     - 30% of 200KB = 60KB for "1-2d"
     - 20% of 200KB = 40KB for "3-7d"
     - 10% of 200KB = 20KB for "imperative" (across all strata, force-include)

4. Pass 1: Force-include all "imperative" turns up to 20KB cap.
   Pass 2: Within each stratum, include all turns ≤ 8KB by descending length until budget exhausted.
   Pass 3: If budget remains in a stratum, fill with shorter turns (longest-first ordering).
   Pass 4: If total < 200KB, prepend session-end summary lines from sessions that contributed turns
           (one-line marker "─── session XXX ended ───" — no extra content).

5. Sort the final selection by (session_id, turn_index) so the model sees conversational flow per session.

6. Log to dream_runs.action_config:
     {
       "sampling": {
         "total_corpus_kb": 950,
         "total_turns": 2200,
         "selected_turns": 480,
         "selected_kb": 198,
         "strata": {"today": {...}, "1-2d": {...}, "3-7d": {...}},
         "imperatives_forced": 87
       }
     }
```

**Why deterministic:** if Moe asks "why did this run produce X?", we can re-run with the same `action_config` and get the same input. Random sampling would make reproducibility impossible.

**Why imperative-forced:** the doctrine demands reinforcement-learning from imperative phrasing (`always`, `never`, `stop`). Dropping a "never use SQLite" turn because it was 3 days old and stratum was full would be a doctrine failure.

**Sample-size override (manual deep runs):** `sampleSizeOverride: <bytes>` raises the cap to up to 800 KB (Sonnet's max input) or 3 MB (Opus). Default cap stays at 200 KB to keep weekly runs cheap.

**Per-turn cap:** any single turn > 8 KB is truncated to 8 KB with a `... [truncated]` suffix. Same posture as 48.2's 32 KB capture cap, but stricter at synthesis time because we want signal density.

### Cadence behavior

The scheduler uses **tick-count modulo**, NOT wall-clock day-of-week. So "weekly" means "every 7 days from process start" (or from any tick rollover). This is fine for the weekly cadence — we don't need Sunday 03:00 SGT alignment for the worker (it just needs to fire roughly once per week). If Moe wants Sunday 03:00 SGT specifically, that requires a separate scheduler primitive (cron-style); recommend deferring that to a future "scheduled workflows v2" phase. **Decision: tick-modulo is fine for v1.**

**Skip-first-tick rationale:** the `tickCount > 0` guard prevents firing on the initial startup tick (where tickCount=0 might or might not pass the modulo depending on interpretation). The pattern is copied verbatim from the existing every_30m/every_6h/every_24h blocks.

**First fire after restart:** if Porter restarts (`systemctl --user restart porter-fastify`), tickCount resets to 0 and the first weekly fire is 7 days out. This means restarts effectively "reset" the weekly cadence. For an actively-developed system this is acceptable; for production-stable use we'd want a `workflows.last_run_at` check to gate runs ("if last_run was < 6.5 days ago, skip"). **Recommend that guard for v1.** Pattern:

```typescript
// In runDreamWorker entry, before doing any work:
const last = (await pool.query(`SELECT MAX(started_at) AS last FROM dream_runs WHERE silo_id=$1 AND status='completed'`, [siloId])).rows[0];
if (args.triggeredBy === 'schedule' && last?.last && Date.now()/1000 - Number(last.last) < 6.5 * 86400) {
  await logIntellectEvent('dream_run_skipped', 'dream_worker', { siloId, lastRun: last.last, reason: 'recent_run' });
  return { dreamRunId: '', proposalsExtracted: 0, status: 'skipped' };
}
```

## Failure Modes & Recovery (DRW-10)

| Failure Point | Detection | Recovery | DB State |
|---|---|---|---|
| Bridge dispatch timeout (>180s) | `AbortSignal.timeout(180_000)` fires | Catch in worker → log + set dream_runs.status='failed' | dream_runs.status='failed', error_message='Bridge dispatch timeout (180s)', no proposals inserted |
| Bridge dispatch error (5xx, 429, circuit breaker open) | `routingEngine.selectWithFallback` throws | Same — caught, logged, status='failed' | dream_runs.status='failed', error_message=err.message |
| Model returns non-JSON | `JSON.parse` throws | First try fence-extraction (regex `/```(?:json)?\s*(\{[\s\S]*\})\s*```/`); if that also fails, status='failed' | dream_runs.status='failed', error_message='JSON parse failed: ...' |
| Model returns malformed schema (missing fields, bad types) | Zod validation throws | status='failed' with specific error | dream_runs.status='failed', error_message='Schema validation failed: ...' |
| Doctrine violation (new_directive without refinement) | `validateRefinementDoctrine` throws | status='failed' with specific error; PROPOSALS NOT INSERTED | dream_runs.status='failed', error_message='Doctrine violation: ...' |
| Individual proposal validation fails (bad target_directive_id, evidence missing, etc.) | Pre-insert validator throws | Whole run fails — no partial insert | dream_runs.status='failed', error_message='Proposal N validation: ...' |
| DB insert error (constraint violation, connection drop) | `pool.query` throws inside transaction | Transaction ROLLBACK, status='failed' | dream_runs.status='failed', no memory_proposals rows |
| Worker crashes (process killed, OOM) | Process restart; dream_runs row stuck in 'running' | Daily sweep (NEW workflow `dream_runs_stuck_sweep`) scans for status='running' rows older than 30 min and flips to 'failed' with error_message='Worker process lost (stuck in running state)' | dream_runs.status='failed' on next sweep |
| `flagged_seeds` array present | Audit only — no failure | Log a separate intellect_events kind='dream_seed_flagged' for each entry | dream_runs.status='completed' (flagged seeds don't break the run) |
| Sampling produces 0 turns (no software silo data) | `sampledTurns.length === 0` check | Skip dispatch entirely, status='completed' with proposals_extracted=0 | dream_runs.status='completed', proposals_extracted=0, error_message=NULL |

## Privacy & Audit (DRW-11)

### Privacy posture
- The worker reads `WHERE silo_id='software'` only. `/silo none` rows (silo_id NULL) and other silos are invisible by construction.
- The prompt body sent to the model contains PII-scrubbed content (48.2 already scrubbed at capture).
- No turn content is logged to `intellect_events`. Only metadata (counts, IDs, status).
- The model's response is stored in `dream_runs.action_config.raw_response`? **NO** — do not store full responses. Store just `response_token_estimate` and the parsed proposals. The raw response is bytes that contain (scrubbed) transcript echoes; we don't want them sitting in another table.

### Audit posture

Three event kinds in `intellect_events`:

1. `dream_run_started` — fields: `{ dreamRunId, siloId, triggeredBy, triggeredByUser? }`
2. `dream_run_completed` — fields: `{ dreamRunId, proposalsExtracted, durationMs, modelUsed }`
3. `dream_run_failed` — fields: `{ dreamRunId, errorMessage, modelUsed? }`

Plus optionally:
4. `dream_run_skipped` — fields: `{ siloId, lastRunAt, reason }` (when guard fires)
5. `dream_seed_flagged` — fields: `{ dreamRunId, seedDirectiveId, note }` (when model surfaces a seed contradiction)

All audit rows use existing `logIntellectEvent(kind, source, payload)` helper.

### Cost audit
The Bridge already logs every dispatch to `bridge_dispatch_log` with token counts and cost estimates. The dream_run's response_token_estimate is a sanity figure; `bridge_dispatch_log` is the source of truth for cost. 48.4 can join `dream_runs ↔ bridge_dispatch_log` via a stamped dispatch_id if 48.3 captures it. **Recommendation:** add a `dispatch_id TEXT` column to dream_runs and stamp it from `decision.dispatchLogId` returned by the routing engine. Cheap. Future-proofs cost-per-run analysis without joining on timestamps.

### Pattern 1: All-or-nothing transactional insert (NOT per-proposal commits)
**What:** Every proposal in a run inserts inside a single BEGIN/COMMIT. Any single failure rolls back the whole run.
**When:** Always. 48.4 must never see "half a run."
**Why:** A partial run is worse than a failed run — it lies about doctrine enforcement and creates an inconsistent state for the human reviewer.

### Pattern 2: Worker reads template from disk on every run
**What:** `fs.promises.readFile(siloRow.prompt_path, 'utf8')` is called inside the worker every run.
**When:** On every dispatch.
**Why:** Lets Moe edit the prompt without redeploying. The template is data, not code. Cost is negligible (~5ms file read).

### Pattern 3: Deterministic sampling with logged decisions
**What:** Sampling decisions are logged to `dream_runs.action_config.sampling`.
**Why:** Reproducibility. If a run produces a surprising proposal, we can replay with the same sample.

### Pattern 4: Fire-and-forget admin endpoint (202 + poll URL)
**What:** Manual trigger returns immediately; worker runs in `setImmediate`.
**Why:** 60-180s wall clock per run. Holding HTTP open is a UX disaster. Polling pattern matches Phase 39 task dispatch.

### Pattern 5: Three-layer doctrine enforcement
**What:** Prompt instruction → Worker validation → DB sort_order.
**Why:** Defense in depth. Even if one layer fails, the doctrine still holds.

### Anti-Patterns to Avoid

- **Don't call HTTP from the worker.** The worker is in the same process as the Bridge. Call `routingEngine.selectWithFallback` directly. HTTP would self-loop and add ~5ms of network overhead for no reason.
- **Don't store the raw model response.** Wasteful, redundant (proposals already extracted), and risks leaking transcript echoes.
- **Don't trust the model to set sort_order.** Always assign it Worker-side.
- **Don't auto-merge proposals across runs.** If Run A's `delete d_x` is still pending when Run B proposes the same delete, leave them both as pending. 48.4 can deduplicate at review time. Worker doesn't try to be clever.
- **Don't extend the existing `correction-detector.ts` for dream output.** Different lifecycle, different doctrine, different cadence. Keep them separate.
- **Don't make the manual-trigger endpoint synchronous "for testing."** Even in smoke tests, use the 202+poll pattern. Mocked Bridge response can be fast (< 1s), so polling completes in one iteration anyway.
- **Don't add a hard FK from memory_proposals.target_directive_ids to directives.id.** TEXT[] elements can't FK in Postgres natively (would require trigger-based check). Orphan-tolerant by-value is the precedent (silo_id, dream_run_id).
- **Don't run two dream workers for the same silo concurrently.** Add a `WHERE NOT EXISTS (SELECT 1 FROM dream_runs WHERE silo_id=$1 AND status='running')` guard at the top of `runDreamWorker`. If concurrent, the second run aborts immediately with `dream_run_skipped` event.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Bridge model dispatch | Custom claude CLI subprocess wrapper | `routingEngine.selectWithFallback` | Existing path has circuit breaker, retry, queue, cost logging, 429 tracking. All free. |
| JSON parsing with fallback | Hand-rolled regex JSON extractor | Try `JSON.parse(response.trim())` first; on fail, try one fence-strip regex; on second fail, mark run failed | Two-tier parse is enough; deeper recovery is fragile and hides model misbehavior |
| Schema validation | Manual property checks | Zod schema for `ParsedDreamResponse` | Zod is already in use across the codebase (chat.ts uses it). One schema, exhaustive checks, clear errors. |
| Scheduled job runner | New cron primitive | Workflow engine + `every_week` tag in scheduler | Existing pattern: 13 builtin workflows already work this way. |
| Manual trigger queue | New job table | Direct call to `runDreamWorker` from `setImmediate` | The bridge already queues per-gateway concurrency. No need for a meta-queue. |
| Sampling | Random sample with no audit | Deterministic stratified sample with `action_config.sampling` log | Reproducibility is non-negotiable for memory consolidation; randomness defeats forensics. |
| Doctrine enforcement | Trust the prompt | Three-layer enforcement (prompt + worker validation + DB sort_order) | Models drift; deterministic enforcement is the only thing that scales. |
| Stuck-run cleanup | Manual SQL on the rare occasion | `dream_runs_stuck_sweep` workflow at every_30m | Worker crashes are rare but real; auto-cleanup is cheap insurance. |
| PII filtering | Re-scrub at synthesis | Trust 48.2's capture-time scrub | One scrub-site. 48.2 already proved it works. Re-scrubbing risks double-redaction artifacts. |

**Key insight:** This phase is dispatch + parse + persist. Every primitive already exists — sampling, prompt rendering, JSON validation, scheduling, audit logging, transactional writes. The new code is the *orchestration* and the *prompt content*, not the infrastructure.

## Common Pitfalls

### Pitfall 1: Model produces JSON wrapped in markdown fences despite the prompt
**What goes wrong:** Models love ` ```json ... ``` ` fences even when told "no fence."
**Why it happens:** Training pressure for "readable" output overrides explicit instructions.
**How to avoid:**
1. First parse attempt: `JSON.parse(response.trim())`.
2. On fail: try `response.match(/```(?:json)?\s*(\{[\s\S]*\})\s*```/)`, parse capture group.
3. On second fail: dream_runs.status='failed' with error_message='JSON parse failed: ...'.

**Warning signs:** dream_runs.status='failed' with parse errors despite "looks like the model tried." Tighten the prompt's self-check.

### Pitfall 2: Model invents target_directive_ids that don't exist
**What goes wrong:** Hallucinated IDs like `d_abc123` that aren't real directives.
**Why it happens:** Models confabulate when reasoning about IDs that weren't shown.
**How to avoid:**
1. The prompt shows every active directive with its real ID. Hallucination should be rare.
2. Worker validates: every target_directive_id MUST exist in directives table (active or archived) for this silo. Single proposal failure → whole run fails.
3. Pre-flight check before insert: `SELECT id FROM directives WHERE id = ANY($1::text[])` and assert count matches input.

**Warning signs:** Runs failing with "Proposal N target_directive_id not found." If chronic, the prompt's directive block formatting is bad.

### Pitfall 3: Doctrine validation feels too strict — kills legit runs
**What goes wrong:** Active count = 5 (4 seeds + 1 dream_worker-added). Worker proposes 1 new_directive but no refinement. Run fails. Moe complains: "but the new directive was good!"
**Why it happens:** The doctrine is asymmetric by design. Once we've added one, we must refine before adding more.
**How to avoid:** This is INTENDED behavior, not a bug. The failure forces the model to re-evaluate whether the new directive is really new vs subsumable into an existing rule. Document this explicitly.

**Warning signs:** Multiple consecutive runs failing with "Doctrine violation." Either the model is generating noise (legit failure) or the model is genuinely finding new ground (rare — relax via manual trigger with a flag if needed).

### Pitfall 4: Token budget over-runs with manual deep runs
**What goes wrong:** Manual Opus run with sample_size_override=2_000_000 hits Opus 4.7's input limit AND the response token budget.
**Why it happens:** No hard cap on either side.
**How to avoid:**
1. Worker validates: `sample_size_override <= 800_000` for Sonnet, `<= 2_500_000` for Opus.
2. `maxTokens` in dispatch is bumped from 16000 to 32000 for Opus deep runs.
3. If we suspect we're over, dispatch will return a model-side error (context overflow) — handled by the generic failure path.

**Warning signs:** Deep runs failing with "context length exceeded" from claude_cli adapter.

### Pitfall 5: prompt_path is relative — worker resolves it from wrong cwd
**What goes wrong:** `silos.prompt_path = 'backend/src/services/intellect/dream-prompts/software.md'`. If the worker process cwd is `/`, the read fails.
**Why it happens:** Porter's systemd unit sets `WorkingDirectory=/home/lobster/projects/porter/backend` but local dev cwd varies.
**How to avoid:** Resolve as `path.resolve(process.env.PORTER_REPO_ROOT ?? path.resolve(__dirname, '../../../..'), siloRow.prompt_path)`. Test in smoke harness.

**Warning signs:** ENOENT errors on first run after deploy. Smoke catches it.

### Pitfall 6: Two manual triggers fired in parallel race on the same silo
**What goes wrong:** Moe clicks "Run software dream" twice. Two workers start. Two dream_runs rows exist. Both try to insert proposals from overlapping (but slightly different) prompt-output pairs. 48.4 sees duplicate-ish proposals.
**Why it happens:** Endpoint returns 202 instantly; no concurrency guard.
**How to avoid:** Worker top-of-function check: `SELECT id FROM dream_runs WHERE silo_id=$1 AND status='running'` — if found, abort with `dream_run_skipped` event and return immediately. Second click sees a 409 (or a 202 with the existing run_id — friendlier).

**Warning signs:** Two dream_runs.status='running' rows for same silo at the same time. Health check catches it.

### Pitfall 7: Worker crashes mid-transaction
**What goes wrong:** Worker process dies between `BEGIN` and `COMMIT` (OOM, SIGKILL).
**Why it happens:** Long-running async worker is exposed to process-level failures.
**How to avoid:**
1. PostgreSQL auto-rolls-back aborted transactions on connection drop. No half-state in memory_proposals.
2. dream_runs.status stays 'running' forever — that's what the `dream_runs_stuck_sweep` handles (flips to failed after 30min).
3. Pattern: write dream_runs FIRST with status='running' BEFORE the transaction, then update at the end. Crash leaves a sweepable orphan, not a phantom.

**Warning signs:** Stuck running rows. Sweep workflow surfaces them.

### Pitfall 8: Sealed seed accidentally targeted by delete
**What goes wrong:** Model proposes `delete` on `silo-sw-design-system` (a sealed seed). Worker accepts. 48.4 displays it. Moe clicks Accept by mistake. Trigger raises an exception.
**Why it happens:** The model didn't read the SEAL marker carefully.
**How to avoid:**
1. Prompt explicitly forbids delete/supersede on `source_type='moe-direct'`.
2. **Worker validates pre-insert:** for any delete/supersede proposal, fetch `directives.source_type` for the target_directive_id. If `source_type='moe-direct'`, REJECT the proposal (whole run fails with "Doctrine violation: targeted sealed seed").
3. Trigger in DB is the last line of defense — but the run should never get that far.

**Warning signs:** Runs failing with "Doctrine violation: targeted sealed seed." Tighten the prompt.

### Pitfall 9: Empty corpus (no software-silo turns in 7 days)
**What goes wrong:** Moe didn't do any code work for a week. Sample returns 0 turns. Worker dispatches with empty TRANSCRIPT_BLOCK. Model produces irrelevant proposals (or 0).
**How to avoid:** Worker checks `sampledTurns.length` BEFORE dispatch. If 0, write dream_runs.status='completed' with proposals_extracted=0 and skip the model call entirely. Log a single intellect_events row.

**Warning signs:** Wasted Bridge dispatches on empty corpora.

### Pitfall 10: Prompt template drift — Moe edits software.md on disk but worker has cached old version
**What goes wrong:** Moe rewrites the template. Worker still uses old version because Node ESM cached it.
**Why it happens:** `fs.readFileSync` is NOT cached, but if the worker were to `import` it, it would be.
**How to avoid:** Always use `fs.promises.readFile` per-run. Never `import` the template.

import { z } from 'zod';

## Open Questions

1. **Should the prompt template support per-run augmentation (e.g. "focus on UI patterns this week")?**
   - What we know: 48.4 doesn't need it; weekly cadence is uniform.
   - What's unclear: would Moe want a "themed" deep run via manual trigger?
   - Recommendation: **Defer.** Manual trigger can pass a `focus_area` field in `action_config` that the template would substitute as `{{FOCUS_AREA}}` (empty by default). Schema-future-proof but not implemented in v1. Add `focus_area?: string` to the manual-trigger body and document as "no-op until prompt template uses it."

2. **What if Sonnet 4.6 is unavailable (model rotation, deprecation)?**
   - What we know: silo row stores `default_model='claude-sonnet-4-6'`; gateway has fallback.
   - What's unclear: does Bridge auto-rotate to a successor? `selectWithFallback` returns the gateway's resolved model.
   - Recommendation: **Trust Bridge.** If the model name is unavailable, the gateway adapter resolves to the best available. Log `dream_runs.model_used` from the actual decision (not the request). Bridge cost-tracking sees the right model.

3. **Should worker accumulate cross-run statistics (active-directive count trend, proposal-acceptance rate)?**
   - What we know: 48.4 needs acceptance-rate UI; trend curves are nice-to-have.
   - What's unclear: where do they live?
   - Recommendation: **Computed on read, not stored.** 48.4's dashboard can `SELECT count(*), status FROM memory_proposals GROUP BY status` and friends. No new table needed. If perf bites, materialized view at 48.4-time.

4. **Should dream worker emit any event consumers (e.g. agent notifications) when proposals are produced?**
   - What we know: 48.4 will surface in admin UI; no agent-side consumer is planned.
   - What's unclear: do we want a Memory V2 signal row when proposals_extracted > 0?
   - Recommendation: **No new signal.** The intellect_events row IS the audit/notification surface. 48.4 polls memory_proposals directly. Lean.

5. **What about the 3 minor variant of the spec where `cwd` is per-turn (added in 48.2)?**
   - What we know: 48.2 added `cwd` to `session_transcript_turns` (nullable).
   - What's unclear: does the Worker use it for finer per-project sub-filtering?
   - Recommendation: **Pass through to the prompt as metadata.** Include `cwd` in the per-turn block: `─── turn 1234 ─── cwd: /home/.../Porter ───`. Lets the model see project context. Don't filter on it in the Worker — that's the model's judgment call (synthesis-time silo re-filter).

6. **Should we batch multiple silos in one weekly run?**
   - What we know: only `software` exists today; future silos will land separately.
   - What's unclear: if 3 silos exist, do they all run on the same Sunday?
   - Recommendation: **One workflow row per silo.** Each silo gets its own `Software dream`, `Legal dream`, etc. workflow row. They share the `every_week` tag and fire in the same tick but each is independent. The concurrency guard (only one running per silo) keeps them safe.

7. **Should we add a "preview" or "dry-run" mode that skips DB writes but returns proposals for inspection?**
   - What we know: smoke tests want this; debug-on-prod wants this.
   - What's unclear: where do dry-run proposals go?
   - Recommendation: **Already in the design** — `dry_run: true` in manual-trigger body. Dry run still creates a `dream_runs` row (status='completed', proposals_extracted=0, action_config.dryRun=true) but skips the INSERT into memory_proposals. The parsed proposals are returned in the GET `/dream-runs/:id` response under a `_preview_proposals` field for dry-runs only. Inspect-without-commit.

8. **Should we add a Drizzle relation/FK that links memory_proposals → dream_runs?**
   - What we know: orphan-tolerant by-value is the established precedent.
   - What's unclear: does the lack of FK make 48.4 queries harder?
   - Recommendation: **No FK.** 48.4 joins by value (`memory_proposals.dream_run_id = dream_runs.id`). Same precedent as session_silo_overrides → silos. Robustness over strictness.

## Risks and Unknowns

### Risk 1: First weekly tick fires before code is ready
The `every_week = 302400 ticks` modulo math means the FIRST fire after deploy is 7 days out. Since the worker doesn't fire until Day 7, there's no risk of an unintended run on Day 0. Skip-recent guard also protects against the case where a manual trigger ran in the last 6.5 days.

### Risk 2: Sonnet 4.6 latency exceeds 180s timeout
Empirical Claude CLI latency on 50K-input prompts: typically 30-90s, occasionally 120s+. The 180s hard timeout has some headroom but a "slow" Sunday could fail a run. Mitigation: error_message='Bridge dispatch timeout' is recoverable via manual retry. If chronic, raise timeout to 300s (cheap config change).

### Risk 3: PostgreSQL TEXT[] handling in Drizzle
Drizzle's `text('foo').array()` syntax is correct but the precise default value `sql\`'{}'::text[]\`` is the right shape (NOT a JS empty array). Pattern check: Phase 35's `triggering_feedback_ids TEXT[] NOT NULL DEFAULT '{}'::text[]` works (verified live). Same pattern here. Low risk.

### Risk 4: gen_random_uuid() requires pgcrypto extension
Used in the workflow seed INSERT. Check: pgcrypto is enabled in Porter's DB (verified in migrate-consolidated). Low risk.

### Risk 5: routingEngine.selectWithFallback retry semantics may cause cost spikes
Workers retry on 429 / 5xx with backoff. A flaky Sonnet endpoint could 3x the cost of a single dream run. Mitigation: max-retry is 3 (default in `withRetry`). Hard outer timeout of 180s caps total wall clock. Cost is bounded.

### Risk 6: Empty corpus is a regression signal, not an error
If a software-silo week has 0 turns, the worker completes with proposals_extracted=0 and no error. From an alerting perspective, this *might* be a problem (capture broken? silo detector broken?). Mitigation: 48.4's dashboard surfaces "weeks with 0 sample" as an info badge, not an error. Don't fail the run for legitimate quiet weeks.

### Risk 7: Concurrency guard race
Two manual triggers fired within milliseconds could both pass the `SELECT WHERE status='running'` check. Mitigation: add a Postgres unique partial index `CREATE UNIQUE INDEX dream_runs_one_running_per_silo ON dream_runs(silo_id) WHERE status='running'`. Second concurrent INSERT fails with `23505`, caught and translated to a 409 in the endpoint.

### Risk 8: Sealed-seed validation requires loading directives twice
Once for the prompt block, once for pre-insert validation. Cache in memory for the run duration. Trivial.

### Risk 9: Worker logs leak transcript content to journald
Even with intellect_events excluding content, `console.error` in failure paths might include prompt text in stack traces. Mitigation: catch+rethrow with sanitized error messages. NEVER log promptBody or response text. Code review focus.

### Risk 10: 48.4 doesn't exist yet — read contract may shift
We're building memory_proposals before 48.4. If 48.4 wants a column we didn't include, migration. Mitigation: design conservatively, JSONB columns for extensibility (proposed_metadata, source_evidence absorb most extensions). Estimated rework: low.
