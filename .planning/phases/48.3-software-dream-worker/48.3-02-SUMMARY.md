## Issues Encountered

**Issue 1: First `systemctl --user restart porter-fastify` did not actually restart the service.**
- Symptom: `curl /health` failed; service status showed "active (running) since 21h ago" (pre-existing instance)
- Root cause: tsx process was still serving from before — restart command didn't propagate SIGTERM through the npx wrapper to the tsx subprocess
- Fix: Explicit `systemctl --user stop porter-fastify` + `pkill -9 -f porter/backend` + `systemctl --user start porter-fastify` separately
- Outcome: New PID 991836, migration ran on fresh boot, all 5 `[migrate-dreams-v1]` log lines emitted
- Lesson: For Porter restarts, stop + force-kill + start is more reliable than `restart` in one shot
