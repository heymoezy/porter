## Nyquist Coverage

Every Wave≥1 task has an automated `<verify>` block. This is established up-front by Plan 48.3-01 shipping the smoke harness in Wave 1 alongside Plan 48.3-02:

- **All 4 implementation plans (02, 03, 04, 05) have automated `<verify>` blocks on every task.**
- **All 13 DRW requirements are exercised by `tests/smoke-48.3.sh`** (Plan 48.3-01).
- **Wave 0 ships in Wave 1** alongside Plan 02 because the smoke script's `tests/smoke-48.3.sh` path + `tests/fixtures/*.json` paths are disjoint from `backend/src/...`.
- **Graceful skip is built-in at two layers**:
  1. Backend unreachable → schema checks (DRW-01/02/03/08) still run via psql; the rest skip cleanly.
  2. `backend/src/services/intellect/dream-worker.ts` missing → DRW-04..DRW-12 SKIP via `SKIP_WORKER=1` (Wave 1/2 runs complete green on schema + prompt-template criteria; Wave 3 flips DRW-04..DRW-12 to live).
- **Poll-not-sleep**: DRW-04, DRW-06, DRW-10 use a polling loop (up to 20s @ 0.5s interval) checking `dream_runs.status` transition, eliminating flakiness on async setImmediate-fired worker runs.
- **Live unmocked verification gated to Plan 05 human-verify**: smoke uses 3 hand-crafted fixtures (deterministic, version-controlled, no Bridge dependency); the live raw-passthrough proof happens once in Plan 05's checkpoint.

The Nyquist criterion (sample at twice the highest frequency of state change) is met: every state-changing task (any task that creates files, migrates schema, or alters worker behavior) has an automated verify, and every DRW requirement has at least one assertion in the Wave-0 fixture-driven smoke harness.
