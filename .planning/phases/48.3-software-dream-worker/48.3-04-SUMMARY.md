## runDreamWorker Entry Point Signature

```typescript
export interface RunDreamArgs {
  siloId: string;
  triggeredBy: 'schedule' | 'manual';
  triggeredByUser?: string;
  modelOverride?: string;
  sampleSizeOverride?: number;
  dryRun?: boolean;
  dreamRunIdOverride?: string;
}

export interface RunDreamResult {
  dreamRunId: string;
  proposalsExtracted: number;
  status: 'completed' | 'failed' | 'skipped';
}

export async function runDreamWorker(args: RunDreamArgs): Promise<RunDreamResult>
```

Throws on dispatch/parse/doctrine/preFlight/DB failure (caller catches; the worker has already written `dream_runs.status='failed'` and logged `dream_run_failed` audit event before re-throwing).

## The 5 Guards

| Guard | When fires | Effect |
| ----- | ---------- | ------ |
| **Concurrency** | Top of runDreamWorker; another `dream_run` row has `status='running'` for same silo | Log `dream_run_skipped` (`reason: concurrent_run_in_progress`) and return `{status:'skipped'}`. Does NOT insert a new dream_runs row. |
| **Skip-recent** | Schedule-triggered only; last completed run for silo started within 6.5 days | Log `dream_run_skipped` (`reason: recent_run_within_6_5_days`) and return `{status:'skipped'}`. Manual triggers bypass this guard. |
| **Empty corpus** | `sampleSoftwareTurns` returns `turns:[]` (zero silo-tagged turns in last 7 days) | Finalize `dream_runs.status='completed'`, `proposals_extracted=0`, `model_used='n/a (empty corpus)'`. Skip dispatch entirely. Legitimate quiet week = success, not failure. |
| **Sealed-seed (pre-flight)** | Any `delete`/`supersede` proposal targets a directive with `source_type='moe-direct'` | Throw `Doctrine violation: targeted sealed seed <id> (kind=<delete|supersede>)`. Whole run fails with no orphan proposals. Catches BEFORE the DB trigger so error_message is diagnostic. |
| **Target-id-exists (pre-flight)** | Any `target_directive_ids` references an id not present in `directives` table | Throw `Proposal validation: target_directive_id <id> not found`. Whole run fails. Catches model-hallucinated directive references. |

## The 3 Doctrine Enforcement Layers

| Layer | Lives in | Mechanism |
| ----- | -------- | --------- |
| **Layer 1: Prompt** | `backend/src/services/intellect/dream-prompts/software.md` (Plan 03) | The Refinement Doctrine section instructs the model to refine first and not append without prior refinement when active count exceeds the seed baseline. Model-side advisory. |
| **Layer 2: Validator** | `validateRefinementDoctrine(parsed, directiveRows.length)` in `dream-parser.ts` (Plan 03), CALLED from `runDreamWorker` step 9 (this plan) | DB-as-ground-truth: takes the DB count as a parameter, ignores `parsed.active_directive_count_before` (model self-report could be spoofed). If active > SEED_BASELINE(4) and proposals contain `new_directive` without any `merge`/`supersede`/`delete`, throws `Doctrine violation: new_directive proposed without prior refinement`. |
| **Layer 3: sort_order** | `assignSortOrder(parsed)` in `dream-parser.ts` (Plan 03), CALLED from `runDreamWorker` step 11 (this plan); MEMORY_PROPOSALS.sort_order column written during INSERT | Deterministic per-area, per-kind sort_order: 100=delete, 200=supersede, 300=merge, 900=new_directive (within an area), +1000 between areas alphabetically. Reviewers ALWAYS see refinements before additions at the DB query level. |

## Explicit logDispatch Capture

`routingEngine.selectWithFallback` does NOT call `logDispatch` internally — only `dispatchStream` does (for the streaming SSE path). Without an explicit call, `dream_runs.dispatch_id` would always be null and Plan 05's raw-passthrough verification (system_prompt inspection on the bridge_dispatch_log row) would have nothing to inspect.

`dispatchDream` therefore calls `routingEngine.logDispatch(decision, ctx, result, undefined)` after `selectWithFallback` returns, captures the returned id, and returns it as `dispatchLogId` to be written into `dream_runs.dispatch_id`. Mock-injection path returns `dispatchLogId: undefined` so `dream_runs.dispatch_id` stays null for mocked runs (intentional — mocked runs have no real Bridge interaction to log).

## ESM-Safe __dirname Shim

Porter backend is ESM (verified — adjacent files use `.js` import suffix). `__dirname` is `undefined` in ESM. Worker uses:

```typescript
import { fileURLToPath } from 'node:url';
const __filename = fileURLToPath(import.meta.url);
const __dirname = path.dirname(__filename);
```

Repo-root resolution: `process.env.PORTER_REPO_ROOT ?? path.resolve(__dirname, '../../../..')`. From compiled `backend/dist/services/intellect/dream-worker.js`, four `..` traverse `intellect → services → dist → backend → Porter`, landing at the repo root where `silos.prompt_path = 'backend/src/services/intellect/dream-prompts/software.md'` resolves correctly via `path.resolve(repoRoot, siloRow.prompt_path)`.

## Mock-Injection Contract

When `process.env.DREAM_WORKER_MOCK_RESPONSE_PATH` is set, `dispatchDream` reads the file at that path and returns its contents as if the model produced them — Bridge is never called. Implemented as one if-statement at the top of `dispatchDream`:

```typescript
const mockPath = process.env.DREAM_WORKER_MOCK_RESPONSE_PATH;
if (mockPath && mockPath.length > 0) {
  const response = await fs.promises.readFile(mockPath, 'utf8');
  return { response, latencyMs: 0, modelUsed: 'mock', dispatchLogId: undefined };
}
```

Production code paths never set this env var; smoke harness is the only consumer. The branch returns `modelUsed: 'mock'` and `dispatchLogId: undefined` so downstream code can identify mocked runs by inspecting `dream_runs.model_used` and `dream_runs.dispatch_id`.

**Issue 1: Inline failure-mode test initially passed (false negative) due to ON CONFLICT (id) DO NOTHING + cross-silo directive ID collision**

- **Symptom:** The DRW-06 doctrine-violation test in `/tmp/dream-worker-failure-smoke.mjs` initially reported `proposalsExtracted: 2, status: completed` — the doctrine should have fired since the fixture has 2 new_directives and the silo was supposed to have 6 active directives.
- **Root cause:** The test seeded 2 dream_worker directives with IDs `mp-smoke-target-stale` and `mp-smoke-target-supersede` using `ON CONFLICT (id) DO NOTHING`. Those IDs ALREADY existed from a previous /tmp/dream-worker-mock-smoke.mjs run in a different silo (`software-smoke-48.3-inline`). `ON CONFLICT (id)` ignores `scope_id` — the conflict matched the existing rows in the OTHER silo and skipped insertion. Result: only 4 sealed seeds in the failmode silo. activeCountBefore=4 hit the SEED_BASELINE early-return, doctrine never fired.
- **Fix:** Renamed the failmode-silo dream_worker directives to `mp-failmode-target-stale` / `mp-failmode-target-supersede` to avoid the cross-silo collision.
- **Outcome after fix:** DRW-06 fires correctly — status=failed, error_message contains "Doctrine violation", 0 orphan proposals.
- **Lesson:** `ON CONFLICT (id)` ignores other columns even if they're in the WHERE clause of an associated query — directives are keyed by `id` alone, so cross-silo seeding requires unique IDs per silo.

**Issue 2: PORTER_REPO_ROOT env var required for inline /tmp/*.mjs tests because cwd defaults to /tmp**

- **Symptom:** When running `node /tmp/dream-worker-mock-smoke.mjs`, the worker would fail to load the prompt template (`ENOENT: no such file or directory`) because `process.cwd()` was `/tmp` and `path.resolve('/tmp', 'backend/src/services/intellect/dream-prompts/software.md')` doesn't exist.
- **Wait — does the worker use process.cwd()?** No, the worker uses `process.env.PORTER_REPO_ROOT ?? path.resolve(__dirname, '../../../..')`. From the compiled `backend/dist/services/intellect/dream-worker.js`, `__dirname` is the directory of that compiled file, and four `..` reach Porter repo root correctly.
- **Actual issue:** When run from /tmp via node, the import resolves the compiled file from its absolute path, so `__dirname` resolution worked correctly without `PORTER_REPO_ROOT`. The env var was passed for belt-and-braces but turned out not strictly necessary for inline tests run from /tmp because of the ESM shim.
- **Outcome:** Confirmed the ESM `__dirname` shim is robust — works regardless of `process.cwd()`. `PORTER_REPO_ROOT` is the override for non-default deployments but not required for development.
