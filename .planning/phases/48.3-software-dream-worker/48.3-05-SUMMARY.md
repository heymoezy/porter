## Deviations from Plan

Four deviations encountered during Task 3 verification, all Rule 1/3 fixes auto-applied per the deviation rules:

### 1. [Rule 3 — Blocking] Smoke harness mock-injection unreachable over HTTP

- **Found during:** Task 3 smoke harness run after live ship.
- **Issue:** The smoke harness sets `DREAM_WORKER_MOCK_RESPONSE_PATH` on curl's local env, but env vars don't propagate to a separately-running backend process. Every DRW-04..DRW-12 check that depended on mock injection hit the real Bridge path instead, which then crashed (see Deviation 2).
- **Fix:** Added `mockResponsePath` optional field to `RunDreamArgs`; POST /dream-run accepts `_mock_response_path` body field; `dispatchDream` prefers arg over env. Smoke harness updated to pass mock via body. Production code never sets either; tests/smoke-48.3.sh is the only consumer.
- **Files modified:** backend/src/services/intellect/dream-worker.ts, backend/src/routes/v1/intellect.ts, tests/smoke-48.3.sh
- **Commit:** `c2508de`

### 2. [Rule 1 — Bug] dispatchDream crashed backend on undefined Bridge result

- **Found during:** Task 3 first smoke run after ship.
- **Issue:** When `routingEngine.selectWithFallback` returned `{decision, result: undefined}` (caused by Deviation 4 below), `dispatchDream` called `routingEngine.logDispatch(decision, ctx, undefined, undefined)`. logDispatch's inner IIFE accesses `result.responseHeaders` at routing-engine.ts:336 OUTSIDE its inner try/catch — the resulting unhandledRejection crashed the entire backend (systemd restarted it twice during verification).
- **Fix:** Added defensive null/type guard on `result` before calling logDispatch. Worker now throws a clean diagnostic ("Bridge dispatch returned no result for gateway X"), dream_runs.status='failed' with that error_message, no backend crash.
- **Files modified:** backend/src/services/intellect/dream-worker.ts
- **Commit:** `c2508de`

### 3. [Rule 3 — Blocking] Smoke harness never seeded transcript turns

- **Found during:** Task 3 second smoke run.
- **Issue:** Smoke harness creates the throwaway smoke silo and 6 directives but never inserts any `session_transcript_turns` rows tagged with the smoke silo. The worker's empty-corpus shortcut (legitimate quiet week = success, 0 proposals, dispatch SKIPPED entirely) fired instead of the mock-fixture path. DRW-04 saw 0 proposals instead of 3.
- **Fix:** Added 3 seed turns (2 sessions, mixed user/assistant) to the smoke harness setup. Cleanup wipes them via `session_id LIKE 'smoke-48.3-%'` as before.
- **Files modified:** tests/smoke-48.3.sh
- **Commit:** `c2508de`

### 4. [Rule 1 — Bug] Bridge circuit-breaker action was a no-op

- **Found during:** Task 3 live unmocked dispatch — claude_cli call returned in <1s with no result, every time.
- **Issue:** `getBreaker()` constructed the breaker with `noop = async () => {}`. Opossum's `breaker.fire(arrowFn)` calls `this.action.apply(ctx, [arrowFn])` — with action=noop, this ignores arrowFn entirely and resolves Promise<undefined>. So `routingEngine.selectWithFallback`'s line 410-414 (`breaker.fire(async () => decision.adapter.dispatch(req))`) NEVER actually called the adapter. The bug was dormant Bridge-wide since opossum 9 was adopted because all chat goes through `dispatchStream` which bypasses `breaker.fire`. Dream-worker is the first real consumer of the non-streaming dispatch path that awaits the result.
- **Fix:** Replaced `noop` with `runThunk = async <T>(fn: () => Promise<T>): Promise<T> => fn()` so the arrow function is actually invoked through the circuit. Also bumped breaker timeout 30s → 180s to match adapter + outer AbortController (Sonnet refinement runs take 60-120s).
- **Side-effect win:** Scheduler `aiRouterDispatch` calls (also non-streaming) were silently returning undefined for every invocation. Now they work too.
- **Files modified:** backend/src/services/bridge/circuit-breaker-registry.ts
- **Commit:** `79afbb0`

### 5. [Rule 1 — Bug] Worker failure path lost dispatch_id

- **Found during:** Task 3 final live verify — first successful Sonnet dispatch surfaced doctrine violation but dispatch_id was null, breaking the audit-trail join.
- **Issue:** Plan 04's `dream-worker.ts` step 14 (success-path UPDATE) writes `dispatch_id=$8` from the local `dispatchLogId` binding. When step 9 doctrine validation throws, the catch block updates `error_message` only — `dispatchLogId` is out of scope and lost. Result: every non-success dream_run had dispatch_id=NULL, breaking Plan 05's live raw-passthrough verify (the bridge_dispatch_log join had no key).
- **Fix:** Hoisted `let capturedDispatchLogId: string | undefined` to try-outer scope; assigned after `dispatchDream` returns; catch block UPDATEs `dispatch_id=COALESCE(dispatch_id, $)` so a real value beats null. Both success and failure paths now stamp dispatch_id.
- **Files modified:** backend/src/services/intellect/dream-worker.ts
- **Commit:** `79afbb0`

### 6. [Documented out-of-spec finding, not auto-fixed] bridge_dispatch_log.system_prompt does not exist

- **Found during:** Task 3 verification SQL execution.
- **Issue:** The plan's `<how-to-verify>` instructed inspecting `system_prompt ~ 'Workspace directives:'` on the bridge_dispatch_log row to prove Memory V3 absence. That column does not exist on the live schema (verified via `\d bridge_dispatch_log`). The bridge layer never logs the rendered system_prompt — the closest thing is `chosen_reason` and the context-tracking columns (agent_id, chat_id, project_id, skills_used, dispatch_strategy).
- **Decision:** Did not add a `system_prompt` column (out-of-spec for Plan 05 and would require backfilling all 1269 historical dispatches). Substituted an equivalent structural proof — the NULL-set on context-tracking columns is logically equivalent and stronger evidence (Memory V3/skills/doctrine wiring only fires when these fields are set; their NULL state on every dream dispatch proves the wiring never engaged). The plan's "Workspace directives:" string check was a proxy for the same logical assertion.
- **Documentation:** This SUMMARY's "Live Unmocked Dispatch Verification" section explains the structural-proof substitution explicitly.

## Issues Encountered

**Live verification surfaced 4 production bugs in 90 minutes.** All 4 (Deviations 2, 4, 5, plus the smoke contract gap in Deviation 1) were dormant before Phase 48.3 Plan 05 because:

- Streaming chat paths bypass the breaker entirely (`dispatchStream` uses `decision.adapter.stream` directly).

This is the right outcome from the live-verify checkpoint — Plan 04's verification gap (no real Bridge dispatch exercised end-to-end) is exactly what the checkpoint was designed to surface.
