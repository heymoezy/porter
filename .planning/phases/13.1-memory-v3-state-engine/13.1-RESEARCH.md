## Summary

Memory V2 stores all memory kinds in a single `concepts` table using soft-discriminators (`memory_kind`, `trust_tier`). This works for storage but creates injection-time ambiguity — the pipeline does not know without runtime discrimination whether a row is a stable operating directive, a project-scoped note, or an agent-specific learning. The result is a flat blob injected into every prompt.

Memory V3 replaces this with explicit tables that carry type semantics at the schema level: `directives` (workspace-scoped, high-trust, rarely-changing rules), `project_notes` (project-scoped state attached to a `project_id`), and `agent_notes` (agent-scoped learnings attached to a `persona_id`). The existing `concepts` table is not dropped — it is retained as the archival/FTS search layer. High-value accepted concepts migrate into the appropriate new table.

The injection pipeline reads these tables in strict priority order: identity context (from persona record) → directives → project state (project_notes) → agent state (agent_notes) → archival FTS search into concepts. Token budgets are applied per tier so directives always fit, with later tiers clipped as needed. A consolidation service uses PostgreSQL `pg_trgm` similarity to detect near-duplicate concepts and merges them with `superseded_by_id` linkage. An agent self-edit API lets agents promote, dismiss, or create memory entries mid-run. An admin aggregation endpoint provides fleet-wide memory health visibility.

**Primary recommendation:** Add three new tables + one migration script; build an injection service that wraps all five tiers; port learner.ts writes to agent_notes; consolidation and self-edit endpoints are isolated services with no changes to existing routes.

### Pattern 1: Additive Migration (MEMV3-01)

The consolidated migration at `migrate-consolidated.ts` runs inside a single transaction with an idempotency key `consolidated_pg_v1`. It must not be modified. New V3 tables need a separate additive migration function with its own idempotency key.

**What:** New `migrateMemoryV3()` function in a new file `migrate-memv3.ts`, called from the server startup sequence alongside `migrateConsolidated()`.

**Idempotency key pattern (established):**
```typescript
// Source: backend/src/db/migrate-consolidated.ts (lines 12-24)
const check = await client.query(
  `SELECT 1 FROM schema_migrations WHERE id = 'memory_v3'`
);
if (check.rowCount && check.rowCount > 0) {
  await client.query('COMMIT');
  return;
}
```

**New table DDL (directives):**
```sql
CREATE TABLE IF NOT EXISTS directives (
  id TEXT PRIMARY KEY,
  scope TEXT NOT NULL DEFAULT 'workspace',   -- 'workspace' | 'project'
  scope_id TEXT,                             -- null for workspace; project_id for project
  content TEXT NOT NULL,
  priority INTEGER NOT NULL DEFAULT 50,       -- lower = higher priority injection order
  status TEXT NOT NULL DEFAULT 'active',      -- 'active' | 'archived'
  created_by TEXT,                            -- username or 'system'
  created_at DOUBLE PRECISION DEFAULT EXTRACT(EPOCH FROM NOW()),
  updated_at DOUBLE PRECISION DEFAULT EXTRACT(EPOCH FROM NOW())
);
```

**New table DDL (project_notes):**
```sql
CREATE TABLE IF NOT EXISTS project_notes (
  id TEXT PRIMARY KEY,
  project_id TEXT NOT NULL,
  content TEXT NOT NULL,
  note_type TEXT NOT NULL DEFAULT 'state',   -- 'state' | 'decision' | 'constraint'
  confidence_score INTEGER NOT NULL DEFAULT 70,
  source_type TEXT NOT NULL DEFAULT 'agent', -- 'agent' | 'human' | 'system'
  status TEXT NOT NULL DEFAULT 'active',
  created_by TEXT,
  created_at DOUBLE PRECISION DEFAULT EXTRACT(EPOCH FROM NOW()),
  updated_at DOUBLE PRECISION DEFAULT EXTRACT(EPOCH FROM NOW())
);
CREATE INDEX IF NOT EXISTS idx_project_notes_project ON project_notes(project_id, status);
```

**New table DDL (agent_notes):**
```sql
CREATE TABLE IF NOT EXISTS agent_notes (
  id TEXT PRIMARY KEY,
  agent_id TEXT NOT NULL,                    -- personas.id
  content TEXT NOT NULL,
  note_type TEXT NOT NULL DEFAULT 'learning', -- 'learning' | 'directive' | 'constraint'
  confidence_score INTEGER NOT NULL DEFAULT 50,
  source_type TEXT NOT NULL DEFAULT 'learning', -- 'learning' | 'self_edit' | 'human'
  status TEXT NOT NULL DEFAULT 'active',
  superseded_by_id TEXT,                     -- for consolidation linkage
  created_by TEXT,
  created_at DOUBLE PRECISION DEFAULT EXTRACT(EPOCH FROM NOW()),
  updated_at DOUBLE PRECISION DEFAULT EXTRACT(EPOCH FROM NOW())
);
CREATE INDEX IF NOT EXISTS idx_agent_notes_agent ON agent_notes(agent_id, status);
```

**Data migration (concepts → structured tables):**
Run as part of `migrateMemoryV3()` after table creation. Migrate concepts where `review_state = 'accepted'` and `status = 'active'`:
- `memory_kind = 'directive'` → insert into `directives`
- `memory_kind = 'concept'` with `scope = 'project'` → insert into `project_notes`
- `memory_kind = 'concept'` with `scope = 'agent'` → insert into `agent_notes`
- All others (global concepts, episodes, signals) stay in `concepts` only

Mark migrated rows with a `migrated_to_v3 = true` flag rather than deleting them — zero data loss.

### Pattern 2: Tiered Injection Service (MEMV3-02)

**What:** A `buildMemoryContext(agentId, projectId?, tokenBudget)` function that queries all five tiers in sequence and returns a formatted string for inclusion in system prompts.

**Tier order and budget allocation (example for 2000-token budget):**
| Tier | Source | Typical Tokens | Cut Strategy |
|------|--------|----------------|--------------|
| 1 | Agent identity (from personas.config) | ~200 | Always included |
| 2 | Directives (workspace + project) | ~300 | ORDER BY priority ASC; trim to budget |
| 3 | Project notes (project_notes) | ~400 | ORDER BY confidence_score DESC; top N |
| 4 | Agent notes (agent_notes) | ~400 | ORDER BY confidence_score DESC; top N |
| 5 | Archival FTS (concepts) | ~700 | websearch_to_tsquery; top K by rank |

**Token estimation:** Use `content.length / 4` as character-to-token approximation — fast, no external library.

**Pattern:**
```typescript
// backend/src/services/memory-injection.ts
export async function buildMemoryContext(opts: {
  agentId: string;
  projectId?: string;
  tokenBudget?: number;
  searchQuery?: string;
}): Promise<string>
```

**Key rule:** If Tier 5 FTS query returns 0 results (no search query available), skip it silently — never error.

### Pattern 3: Consolidation Service (MEMV3-03)

**What:** A background service that runs over `agent_notes` and `concepts` for a given agent, finds near-duplicates using `pg_trgm` similarity, merges them, and sets `superseded_by_id`.

**PostgreSQL pg_trgm similarity query:**
```sql
-- Must first enable extension:
CREATE EXTENSION IF NOT EXISTS pg_trgm;

-- Find pairs with high similarity:
SELECT a.id AS a_id, b.id AS b_id,
       similarity(a.content, b.content) AS sim
FROM agent_notes a
JOIN agent_notes b ON a.id < b.id  -- avoid self-join and duplicates
WHERE a.agent_id = $1
  AND b.agent_id = $1
  AND a.status = 'active'
  AND b.status = 'active'
  AND similarity(a.content, b.content) > 0.6  -- threshold: 0.6 is aggressive enough
ORDER BY sim DESC;
```

**Success criterion check:** Phase requires 40% reduction on 50+ concepts. With a 0.6 similarity threshold, this is achievable because learning sessions often extract near-identical patterns from similar sources.

**Merge strategy:** Keep the concept with higher `confidence_score`; set the loser's `status = 'superseded'` and `superseded_by_id = winner.id`. Do NOT delete — audit trail preserved.

**Invocation:** Expose as `POST /api/v1/memory/consolidate` (requires auth, agent_id in body). This is not a cron job in this phase — it runs on-demand. The planner can schedule it via agent_jobs if needed later.

### Pattern 4: Agent Self-Edit API (MEMV3-04)

**What:** `POST /api/v1/memory/self-edit` — accepts a structured action object; validates via Zod; writes to the appropriate table; returns the updated record.

**Request schema:**
```typescript
const selfEditSchema = z.object({
  agent_id: z.string().min(1),
  action: z.enum(['promote', 'dismiss', 'create_directive']),
  // For promote: signal concept_id to migrate to agent_notes
  concept_id: z.string().optional(),
  // For create_directive: direct content
  content: z.string().min(1).max(500).optional(),
  note_type: z.enum(['learning', 'directive', 'constraint']).optional(),
});
```

**Actions:**
- `promote`: move a concept from `concepts` table into `agent_notes` (copy content, set source_type='self_edit', mark original status='archived')
- `dismiss`: set `status = 'archived'` on a concept or agent_note
- `create_directive`: insert new row into `agent_notes` with `note_type='directive'`

**Session reflection:** "reflected in subsequent injection within the same session" — since injection reads from DB directly (no in-process cache), any write is immediately visible on the next `buildMemoryContext()` call. No cache invalidation needed.

### Pattern 5: Admin Memory Overview (MEMV3-05)

**Endpoint:** `GET /api/v1/admin/memory/overview`

**Response shape:**
```typescript
{
  agents: Array<{
    agent_id: string;
    agent_name: string;
    concept_count: number;
    note_count: number;
    avg_confidence: number;       // 0-100
    pending_review_count: number; // concepts where review_state = 'pending'
    health_score: number;         // computed: see formula below
  }>;
  totals: {
    total_concepts: number;
    total_notes: number;
    total_directives: number;
    total_pending_review: number;
  };
}
```

**Health score formula:** `LEAST(100, GREATEST(0, avg_confidence - (pending_review_count * 5)))`
This penalises agents with many unreviewed signals without destroying health entirely.

**Query pattern:** Single aggregation query joining `personas`, `concepts` (scope='agent'), and `agent_notes`:
```sql
SELECT
  p.id AS agent_id,
  p.name AS agent_name,
  COUNT(DISTINCT c.id) AS concept_count,
  COUNT(DISTINCT an.id) AS note_count,
  COALESCE(AVG(COALESCE(an.confidence_score, c.confidence_score)), 50) AS avg_confidence,
  COUNT(DISTINCT CASE WHEN c.review_state = 'pending' THEN c.id END) AS pending_review_count
FROM personas p
LEFT JOIN concepts c ON c.scope = 'agent' AND c.scope_id = p.id AND c.status = 'active'
LEFT JOIN agent_notes an ON an.agent_id = p.id AND an.status = 'active'
WHERE p.is_system = 0
GROUP BY p.id, p.name
ORDER BY p.name
```

Admin endpoint requires `platform_admin` role (same pattern as `admin/` routes with `requireAuth` + role check).

### Anti-Patterns to Avoid

- **Modifying migrate-consolidated.ts:** The existing migration uses idempotency key `consolidated_pg_v1` and runs in a single transaction. Adding new tables there risks breaking existing deployments. Use a separate migration with a new key.
- **Deleting from concepts during migration:** The consolidation requirement explicitly tracks `superseded_by_id` — soft deletes only, never hard deletes.
- **Auto-running consolidation on startup:** Background consolidation on every boot is expensive and unpredictable. Expose as on-demand endpoint; optionally wire to a scheduled agent_job later.
- **In-memory injection caching:** Since agents may self-edit during a run, never cache injection results in-process. Each `buildMemoryContext()` call hits the DB. PostgreSQL connection pooling (pool size 10) handles the load.
- **Injecting all tiers unconditionally:** If no `project_id` is present, skip Tier 3. If no `searchQuery`, skip Tier 5 FTS. Build the pipeline as conditional per-tier, not unconditional.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Fuzzy text similarity | Custom Levenshtein or LLM-based comparison | `pg_trgm` `similarity()` | Runs in DB, no round-trip, no extra service |
| Token counting | tiktoken or external tokenizer | `content.length / 4` approximation | Good enough for budget slicing; no new dependency |
| Concept deduplication | Hash-based exact match | `pg_trgm` with 0.6 threshold | Catches paraphrased duplicates that hash differently |

**Key insight:** `pg_trgm` is a PostgreSQL built-in extension (enabled with `CREATE EXTENSION IF NOT EXISTS pg_trgm`). It provides `similarity(text, text)` returning 0.0–1.0 and a `%` operator for fast GIN-indexed similarity search. No npm package needed.

## Common Pitfalls

### Pitfall 1: Migration idempotency collision
**What goes wrong:** Adding V3 tables inside the existing `consolidated_pg_v1` migration block fails on already-migrated databases (the idempotency guard returns early before reaching new tables).
**Why it happens:** `migrate-consolidated.ts` checks `SELECT 1 FROM schema_migrations WHERE id = 'consolidated_pg_v1'` and returns immediately if found.
**How to avoid:** Create a separate function `migrateMemoryV3()` in `backend/src/db/migrate-memv3.ts` with its own idempotency key `memory_v3`. Call it from `server.ts` after `migrateConsolidated()`.
**Warning signs:** Tables missing in production despite migration appearing to succeed.

### Pitfall 2: pg_trgm extension not enabled
**What goes wrong:** `similarity()` function not found at runtime.
**Why it happens:** `pg_trgm` must be explicitly enabled even on PostgreSQL 16.
**How to avoid:** Add `CREATE EXTENSION IF NOT EXISTS pg_trgm` in `migrateMemoryV3()` before any similarity query. Already established pattern — `vector` extension is created this way in `migrate-consolidated.ts` line 9.
**Warning signs:** `ERROR: function similarity(text, text) does not exist`.

### Pitfall 3: Injection bloat with no token ceiling
**What goes wrong:** Agent receives 5 tiers × unlimited rows = context window overflow.
**Why it happens:** No per-tier token budget slicing during injection pipeline.
**How to avoid:** Calculate running token estimate after each tier; stop adding rows when budget is consumed. Default budget: 2000 tokens (configurable per agent via config JSONB field).
**Warning signs:** API calls returning 413 or model errors about context length.

### Pitfall 4: Self-edit race condition on concurrent runs
**What goes wrong:** Two concurrent agent runs editing the same concept produce inconsistent state.
**Why it happens:** No row-level locking on memory rows during self-edit.
**How to avoid:** Use `UPDATE ... WHERE id = $1 AND status = 'active' RETURNING id` — if 0 rows returned, the concept was already modified. Return 409 to the caller. This is a best-effort pattern acceptable for the current scale.
**Warning signs:** Duplicate or conflicting agent_notes rows appearing.

### Pitfall 5: Admin overview query timeout on large datasets
**What goes wrong:** The aggregation query scanning all agents × all concepts times out.
**Why it happens:** No index on `concepts(scope, scope_id, status)` for the JOIN condition.
**How to avoid:** Add `CREATE INDEX IF NOT EXISTS idx_concepts_scope ON concepts(scope, scope_id, status)` in the migration. The `agent_notes` table already has `idx_agent_notes_agent`.
**Warning signs:** Slow admin endpoint response when concept count > 10k.

// Source: backend/src/db/migrate-consolidated.ts (lines 1-24)
import pg from 'pg';

export async function migrateMemoryV3(pool: pg.Pool): Promise<void> {
  const client = await pool.connect();
  try {
    await client.query('BEGIN');
    // Ensure schema_migrations exists (already created by consolidated migration)
    const check = await client.query(
      `SELECT 1 FROM schema_migrations WHERE id = 'memory_v3'`

      await client.query('COMMIT');
      return;
    }
    await client.query(`CREATE EXTENSION IF NOT EXISTS pg_trgm`);
    // ... create tables, indexes, migrate data ...
    await client.query(
      `INSERT INTO schema_migrations (id) VALUES ('memory_v3')`
    );
    await client.query('COMMIT');
  } catch (err) {
    await client.query('ROLLBACK');
    throw err;
  } finally {
    client.release();
  }
}

## Integration Points

### Route Registration
Current `memory.ts` is registered at prefix `/memory/concepts` in `index.ts` (line 50). To add `POST /api/v1/memory/self-edit` and `GET /api/v1/admin/memory/overview`, two options:

**Option A (recommended):** Change prefix to `/memory` in `index.ts`, rename the existing `GET /` route to `GET /concepts`. Add new routes under the same plugin. Cleaner URL structure.

**Option B:** Register a second plugin for the new memory routes at prefix `/memory`. Avoids touching existing route structure.

Use Option A — it results in `/api/v1/memory/concepts` (unchanged externally), `/api/v1/memory/self-edit`, and `/api/v1/admin/memory/overview`.

### Chat Injection Point
`POST /api/v1/chat/stream` currently builds `augmentedMessage` inline (line 256 of chat.ts). Memory injection should be added here: call `buildMemoryContext({ agentId, projectId, tokenBudget: 2000 })` before the `backend.stream()` call and prepend the result to `augmentedMessage`.

### Learner Service (stop auto-extraction)
`learner.ts` writes new concepts to the `concepts` table with `scope='agent'` and `scope_id=templateId`. The phase goal says "stop auto-extraction from chat responses" — this means no change to `learner.ts` (learning sessions are explicit, not response-extraction). Auto-extraction is not currently implemented in `chat.ts` anyway (no extraction call found). The stop-extraction goal is confirming that no extraction should be added in V3.

### Admin Index
`admin/index.ts` (line 24) has a commented `TODO` about admin routes import path fixes. The new `/api/v1/admin/memory/overview` endpoint should be added to `admin/` routes directory as `admin/memory.ts` and registered in `admin/index.ts` once the admin routes are enabled. Until then, it can live temporarily in `memory.ts` with a manual role check.
