# Phase 48.2: Transcript Capture — Research

**Researched:** 2026-05-11
**Domain:** Memory layer — Claude Code hook capture, transcript persistence, PII redaction, retention sweep
**Confidence:** HIGH (all findings sourced from live codebase, live DB, Claude Code official hooks docs, and an actual transcript JSONL inspected on disk)

---

## Summary

Phase 48.2 captures the raw turns that Phase 48.3's Software Dream Worker will consume. Each user prompt and each assistant reply is persisted to a new `session_transcript_turns` table, tagged with the **silo that was active when the turn happened** (forward-only — turns prior to a `/silo <name>` switch keep their original tag). Capture happens through two Claude Code hooks:

- **`UserPromptSubmit`** (existing hook, extend `~/.claude/hooks/porter-user-prompt.js`) — writes one row per user turn, fire-and-forget.
- **`Stop`** (NEW hook — register in `~/.claude/settings.json`) — fires when Claude finishes an assistant turn. Reads the `transcript_path` field from the payload (a complete JSONL of the conversation at `~/.claude/projects/<slug>/<session>.jsonl`), parses any assistant turns not yet captured, and persists them. This is the canonical capture path for assistant content because (a) the JSONL contains the final assembled message, (b) Stop fires once per assistant turn so the cost is bounded, and (c) it gives us idempotent recovery if a turn was missed.

PII is filtered at write-time using the regex set already in `backend/src/services/learner.ts` (`scrubPII` — emails, @-handles, phone numbers). Retention is a daily sweep registered as a workflow (`every_24h` schedule tag) using the existing Intellect workflow engine — no new scheduler infrastructure needed.

**Primary recommendation:** Capture user turns via `UserPromptSubmit` (cheap, one row per submit). Capture assistant turns via `Stop` hook by parsing `transcript_path` JSONL. Tag silo at the moment of capture (call `detectSilos` from `silo-detector.ts`). Run PII filter inline at INSERT time. Add a `transcript_retention_sweep` workflow row that runs `every_24h` and hard-deletes turns older than 30 days. Privacy kill-switch: a single config flag `intellect.transcript_capture_enabled` (default `true`) plus a per-session opt-out via `/silo none`.

### Pattern 1: Single-Writer Capture Endpoint (NOT direct DB from hook)
**What:** All hook writes go through `POST /api/v1/intellect/transcript/turn`. Hooks never touch the DB directly.
**When to use:** Anywhere a hook writes to the DB.
**Why:** Hooks run as detached child processes; they can't safely share the Fastify pg pool. Hooks should be HTTP clients. This is the existing pattern (`/intellect/correction`, `/intellect/session-end`, `/intellect/silo-command`).

**Example:**
```typescript
// Source: backend/src/routes/v1/intellect.ts (mirrors existing /correction route)
app.post('/transcript/turn', async (req, reply) => {
  const { session_id, cwd, role, content, turn_index, captured_at } = req.body as {
    session_id: string;
    cwd: string | null;
    role: 'user' | 'assistant';
    content: string;
    turn_index: number;
    captured_at?: string;
  };

  // Privacy gate
  if (!config.intellect.transcriptCaptureEnabled) return reply.send(ok({ skipped: 'disabled' }));

  // Silo lookup (same path the context endpoint uses)
  const silos = await detectSilos({ sessionId: session_id, cwd }, pool);
  const siloId = silos.length === 1 ? silos[0].id : silos[0]?.id ?? null;
  // Multi-silo: take first match. Dream Worker re-filters at synthesis time.

  // If an explicit /silo none override is active, skip capture
  const overrideRow = await pool.query(
    `SELECT silo_id FROM session_silo_overrides
     WHERE session_id = $1 AND set_at > NOW() - INTERVAL '24 hours'`,
    [session_id],
  );
  if (overrideRow.rowCount && overrideRow.rows[0].silo_id === null) {
    return reply.send(ok({ skipped: 'silo_none' }));
  }

  const clean = scrubPII(content);

  await pool.query(
    `INSERT INTO session_transcript_turns
       (session_id, turn_index, role, silo_id, content, captured_at)
     VALUES ($1, $2, $3, $4, $5, COALESCE($6::timestamptz, NOW()))
     ON CONFLICT (session_id, turn_index) DO NOTHING`,
    [session_id, turn_index, role, siloId, clean, captured_at ?? null],
  );

  return reply.send(ok({ inserted: true, silo: siloId }));
});
```

### Pattern 2: Stop-Hook Tail Parser (JSONL bookmark)
**What:** The Stop hook does NOT re-parse the entire `transcript_path` JSONL on every fire. It tracks a per-session byte offset bookmark (in a tiny sidecar file at `/tmp/porter-transcript-bookmark/<session_id>.offset`), opens the JSONL from that offset, parses new lines, and updates the bookmark on successful POST.
**When to use:** Every Stop hook invocation.
**Why:** JSONL files grow monotonically. Re-parsing 50MB on every assistant turn is wasteful. Offset bookmark is O(new bytes only).

**Example:**
```javascript
// Source: ~/.claude/hooks/porter-stop.js (new file)
const fs = require('node:fs');
const path = require('node:path');
const readline = require('node:readline');

const BOOKMARK_DIR = '/tmp/porter-transcript-bookmark';

async function parseAndPostTurns(sessionId, transcriptPath, cwd) {
  fs.mkdirSync(BOOKMARK_DIR, { recursive: true });
  const bookmarkPath = path.join(BOOKMARK_DIR, `${sessionId}.offset`);
  let offset = 0;
  try { offset = parseInt(fs.readFileSync(bookmarkPath, 'utf8'), 10) || 0; } catch {}

  const stat = fs.statSync(transcriptPath);
  if (stat.size <= offset) return; // nothing new

  const stream = fs.createReadStream(transcriptPath, { start: offset });
  const rl = readline.createInterface({ input: stream });
  let turnIndex = await getNextTurnIndex(sessionId); // GET /transcript/turn-count?session_id=...
  for await (const line of rl) {
    if (!line.trim()) continue;
    let row;
    try { row = JSON.parse(line); } catch { continue; }
    if (row.type !== 'assistant') continue;
    const blocks = row.message?.content || [];
    const text = blocks.filter(b => b.type === 'text').map(b => b.text).join('\n').trim();
    if (!text) continue;
    await postTurn({
      session_id: sessionId,
      cwd,
      role: 'assistant',
      content: text,
      turn_index: turnIndex++,
      captured_at: row.timestamp,
    });
  }
  fs.writeFileSync(bookmarkPath, String(stat.size));
}
```

### Pattern 3: Workflow-Driven Retention (no new scheduler)
**What:** Register a single `workflows` row with `trigger_type='schedule'`, `trigger_value='every_24h'`, `action_type='transcript_retain'`. The existing `runScheduledWorkflows` loop (`scheduler.ts:414`) picks it up.
**When to use:** Any background maintenance job that fits 30m / 6h / 24h cadence.
**Why:** Adding `setInterval` calls to `scheduler.ts` for every new job creates code soup. Workflows are data, not code. `memory_prune` is the analogous existing workflow.

**Example:**
```typescript
// Source: backend/src/services/intellect/workflow-engine.ts (extension)
const actionHandlers = {
  // …existing…
  transcript_retain: async () => runTranscriptRetention(),
};

const BUILTIN_WORKFLOWS = [
  // …existing…
  {
    name: 'Prune transcripts older than 30 days',
    trigger_type: 'schedule',
    trigger_value: 'every_24h',
    action_type: 'transcript_retain',
  },
];

// Source: backend/src/services/intellect/transcript-retention.ts (new)
export async function runTranscriptRetention(): Promise<{ deleted: number }> {
  const { rowCount } = await pool.query(
    `DELETE FROM session_transcript_turns WHERE captured_at < NOW() - INTERVAL '30 days'`,
  );
  return { deleted: rowCount ?? 0 };
}
```

### Anti-Patterns to Avoid

- **Don't write to the DB directly from the hook.** Hooks are detached processes that can't share connection pools. They must POST to the Fastify backend.
- **Don't re-parse the entire JSONL on every Stop fire.** Bookmark byte offset per session.
- **Don't synchronously block the hook on capture.** The user prompt hook is on the critical path of every keystroke-submit. Fire-and-forget (`http.request(...)` with no `await`) is the right pattern.
- **Don't store raw tool outputs.** Bloat. Only persist `role: user|assistant` with text content. 48.3 needs the conversational signal, not the tool traces.
- **Don't tag silo retroactively.** A `/silo software` switch tags only subsequent turns. Past turns keep their tag (or null). The Dream Worker's synthesis-time re-filter handles wrong tags.
- **Don't auto-create a session row in `session_registry`.** That table tracks dispatch sessions (UUIDs minted by `openSession`), not Claude CLI sessions. Transcript turns reference the Claude CLI `session_id` directly — same namespace as `session_silo_overrides` from 48.1.
- **Don't add an FK from `session_transcript_turns.silo_id` to `silos.id`.** Silos can be disabled or renamed in the future; orphan-tolerant nullable text is more robust. Same precedent as `session_silo_overrides.silo_id` (nullable text, no FK).

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|

| Daily scheduling | `setInterval` in `scheduler.ts` | `workflows` row + `runScheduledWorkflows('every_24h')` | Existing pattern. Idempotent seed. Live admin toggle without redeploy. |
| Idempotency on Stop re-fire | Application-level dedup check | Postgres `UNIQUE (session_id, turn_index)` + `ON CONFLICT DO NOTHING` | DB does the work atomically. |
| JSONL line parsing | Stream the whole file | `readline.createInterface` with byte-offset start position | Standard library, O(new bytes). |
| Sidecar bookmark storage | A new DB table for byte offsets | Plain file at `/tmp/porter-transcript-bookmark/<session>.offset` | Bookmarks are disposable. /tmp gets cleared on boot; on boot the bookmark resets to 0 and `ON CONFLICT DO NOTHING` makes re-parse safe. |

**Key insight:** This phase is mostly plumbing — the components already exist (PII filter, silo detector, workflow engine, hook HTTP pattern). The work is wiring them together with one new table, one new endpoint, one new hook, and one new workflow row.

## Common Pitfalls

### Pitfall 1: Stop hook fires twice (re-entry)
**What goes wrong:** Claude Code's Stop hook can fire repeatedly during a single assistant turn if `stop_hook_active` triggers a continuation. A naive INSERT would create duplicate turn rows.
**Why it happens:** Documented behavior in Anthropic hooks reference — Stop fires "when Claude finishes responding," but a hook that blocks (decision: "block") forces continuation and the Stop event fires again at the next stop point.
**How to avoid:**
1. `UNIQUE (session_id, turn_index)` + `ON CONFLICT DO NOTHING`.
2. `turn_index` is computed server-side from `MAX(turn_index)+1` in the same transaction as the INSERT, scoped by `session_id`. Hooks send role + content; backend assigns the index.
3. Sidecar byte-offset bookmark prevents re-parsing JSONL content already POSTed.

**Warning signs:** Same turn_index reused, or assistant text duplicated in `session_transcript_turns` for one session.

### Pitfall 2: `transcript_path` is stale (referenced bug)
**What goes wrong:** Anthropic Issue #8564 documents that the Stop hook's `transcript_path` can point to a transcript that doesn't include the most recent turn yet (race between hook fire and JSONL flush).
**Why it happens:** Claude Code writes the JSONL asynchronously; the Stop hook can fire before the assistant turn is flushed.
**How to avoid:**
1. The bookmark+tail-parse design tolerates this: if the new turn isn't in the file yet, we parse zero lines and exit. The NEXT Stop fire (or a periodic Stop-driven re-check) will pick it up.
2. Add a 250ms sleep before reading the file — gives the JSONL flush time to land. Cheap, deterministic.
3. As a backup, on `SessionEnd` (existing hook `porter-session-end.js`) do one final tail-parse — catches anything Stop missed at the very end of a session.

**Warning signs:** Missing final assistant turns in `session_transcript_turns`; smoke test counts off by one.

### Pitfall 3: Concurrent sessions and shared bookmark dir
**What goes wrong:** Multiple Claude CLI sessions running in parallel (Moe's standard multi-session workflow) — if bookmarks aren't keyed per session_id, two sessions could clobber each other's offsets.
**Why it happens:** /tmp is a single namespace.
**How to avoid:** Bookmark filename is `<session_id>.offset`. Each session_id is a UUID. No collision possible.
**Warning signs:** Mid-session truncation in one transcript when another session ends.

### Pitfall 4: Large assistant turns (long-form output)
**What goes wrong:** A single assistant turn can be 50KB+ of text. INSERTing without size limits is a footgun.
**Why it happens:** Some replies are essays.
**How to avoid:**
1. Cap content at 32KB at the capture endpoint — truncate with a `... [truncated: N chars]` suffix. The Dream Worker doesn't need the full text; it needs the signal.
2. Postgres `TEXT` handles any size, but indexes choke. We don't index `content` — only `(session_id, turn_index)` and `(silo_id, captured_at DESC)`.

**Warning signs:** Slow inserts, growing table size beyond expectations (10MB/week for moderate use is the rough ceiling).

### Pitfall 5: Multi-silo session, ambiguous tagging at INSERT
**What goes wrong:** `detectSilos` can return multiple silo IDs (e.g., a session that's both `software` and a future `legal` silo). We have one `silo_id` column.
**Why it happens:** Silo detection rules can overlap (per spec: "Multi-silo sessions… get both sets of directives unioned").
**How to avoid:**
1. Phase 48.2 has only ONE silo (software). The realistic multi-match case is silo:software → silo_id='software'.
2. When detection returns multiple, prefer the explicit override (already first in the detection algorithm), else use the first non-software silo if present, else use software. Document this as "single-silo tagging until 48.3 needs multi-tag arrays."
3. Future-proof: leave the door open to a `silo_ids TEXT[]` column in a later migration. NOT in this phase.

**Warning signs:** Turns tagged inconsistently across a session; Dream Worker complains about ambiguous attribution.

### Pitfall 6: Privacy regression — capture happens before /silo none takes effect
**What goes wrong:** Moe types `/silo none`, then types a sensitive prompt. The /silo command is intercepted and the override row is written, but the next user prompt hook may race against the override read.
**Why it happens:** The /silo command and the next prompt are sequential user actions but the override write to DB is async (fire-and-forget POST in `porter-user-prompt.js`).
**How to avoid:**
1. The `/silo` interception path in `porter-user-prompt.js` is NOT fire-and-forget — it `await`s the fetch with a 3000ms timeout (confirmed in the file). The override is committed before the hook exits.
2. The capture endpoint reads `session_silo_overrides` synchronously at the start of every POST. By the time the NEXT prompt fires, the override row exists.
3. Add a smoke test: `/silo none` → next prompt → no row inserted.

**Warning signs:** Rows captured after a /silo none with no override row in DB.

### Pitfall 7: Hook latency on UserPromptSubmit (critical path)
**What goes wrong:** UserPromptSubmit fires on every keystroke-submit. If the capture POST blocks for 2s, the user feels lag.
**Why it happens:** Synchronous HTTP request to localhost is normally <5ms but can spike.
**How to avoid:**
1. The new capture POST in `porter-user-prompt.js` is fire-and-forget, mirroring the existing `/intellect/correction` call (which uses `http.request` with no `await`, 2000ms timeout).
2. The Stop hook can afford to be synchronous — it fires once per assistant turn, which is already a multi-second operation.

**Warning signs:** Reports of typing lag; hook timeouts in `process.stderr` logs.

### Pitfall 8: JSONL contains hook-injected content (loops)
**What goes wrong:** The JSONL transcript includes `hook_success` attachment rows containing the hook's own injected context (e.g., the entire Porter session-start markdown). If we treat those as assistant content, we'd capture the directives back into transcripts and the Dream Worker would mine its own output.
**Why it happens:** Inspected JSONL on disk shows rows like `{"type":"attachment","attachment":{"type":"hook_success","content":"## Porter Context..."}}`. These are NOT assistant turns.
**How to avoid:**
1. Filter strictly on `row.type === 'assistant'` AND `row.message?.role === 'assistant'` in the Stop hook parser.
2. Skip rows where `row.isSidechain === true` (subagent traces).
3. Skip the queue-operation / last-prompt / attachment metadata rows.

**Warning signs:** Captured turns containing "## Porter Context" or markdown directives — sign of mis-filtering.

## Code Examples

### Drizzle Schema Entry
```typescript
// Source: backend/src/db/schema.ts (new entries, appended after sessionSiloOverrides)

export const sessionTranscriptTurns = pgTable('session_transcript_turns', {
  id: serial('id').primaryKey(),
  sessionId: text('session_id').notNull(),
  turnIndex: integer('turn_index').notNull(),
  role: text('role').notNull(),                // 'user' | 'assistant'
  siloId: text('silo_id'),                     // nullable, no FK (see anti-pattern note)
  content: text('content').notNull(),          // PII-scrubbed, ≤32KB
  capturedAt: timestamp('captured_at', { withTimezone: true }).notNull().defaultNow(),
}, (table) => ({
  uniqSessionTurn: uniqueIndex('transcript_turns_session_turn_uniq')
    .on(table.sessionId, table.turnIndex),
  siloCapturedIdx: index('transcript_turns_silo_captured_idx')
    .on(table.siloId, table.capturedAt),
  capturedIdx: index('transcript_turns_captured_idx')
    .on(table.capturedAt),                     // for retention DELETE
}));

export type SessionTranscriptTurn = typeof sessionTranscriptTurns.$inferSelect;
```

### Migration
```typescript
// Source: backend/src/db/migrate-transcripts-v1.ts (NEW, mirrors migrate-silos-v1 pattern)
import type { Pool } from 'pg';

export async function migrateTranscriptsV1(pool: Pool): Promise<void> {
  const client = await pool.connect();
  try {
    await client.query('BEGIN');
    const { rows } = await client.query(
      `SELECT 1 FROM schema_migrations WHERE id = 'transcripts_v1'`,
    );
    if (rows.length) { await client.query('COMMIT'); return; }

    await client.query(`
      CREATE TABLE IF NOT EXISTS session_transcript_turns (
        id           SERIAL PRIMARY KEY,
        session_id   TEXT NOT NULL,
        turn_index   INTEGER NOT NULL,
        role         TEXT NOT NULL CHECK (role IN ('user','assistant')),
        silo_id      TEXT,
        content      TEXT NOT NULL,
        captured_at  TIMESTAMPTZ NOT NULL DEFAULT NOW()
      );
    `);
    await client.query(`
      CREATE UNIQUE INDEX IF NOT EXISTS transcript_turns_session_turn_uniq
        ON session_transcript_turns(session_id, turn_index);
    `);
    await client.query(`
      CREATE INDEX IF NOT EXISTS transcript_turns_silo_captured_idx
        ON session_transcript_turns(silo_id, captured_at DESC);
    `);
    await client.query(`
      CREATE INDEX IF NOT EXISTS transcript_turns_captured_idx
        ON session_transcript_turns(captured_at);
    `);

    // Seed retention workflow row (idempotent by name)
    await client.query(`
      INSERT INTO workflows (id, name, trigger_type, trigger_value, action_type, action_config, enabled)
      SELECT gen_random_uuid()::text,
             'Prune transcripts older than 30 days',
             'schedule', 'every_24h', 'transcript_retain', '{}'::jsonb, true
      WHERE NOT EXISTS (
        SELECT 1 FROM workflows WHERE name = 'Prune transcripts older than 30 days'
      );
    `);

    await client.query(
      `INSERT INTO schema_migrations (id, applied_at)
       VALUES ('transcripts_v1', EXTRACT(EPOCH FROM NOW()))`,
    );
    await client.query('COMMIT');
    console.log('[migrate-transcripts-v1] complete');
  } catch (err) {
    await client.query('ROLLBACK');
    throw err;
  } finally {
    client.release();
  }
}
```

### Stop Hook (new file)
```javascript
// Source: ~/.claude/hooks/porter-stop.js (NEW)
#!/usr/bin/env node
// Porter Stop hook — captures assistant turns from transcript_path JSONL.

const fs = require('node:fs');
const path = require('node:path');
const http = require('node:http');
const readline = require('node:readline');

const BOOKMARK_DIR = '/tmp/porter-transcript-bookmark';
const PORTER = { host: '127.0.0.1', port: 3001 };

function readStdinJson() {
  return new Promise((resolve) => {
    let buf = '';
    const t = setTimeout(() => resolve(null), 1000);
    process.stdin.setEncoding('utf8');
    process.stdin.on('data', (c) => { buf += c; });
    process.stdin.on('end', () => {
      clearTimeout(t);
      try { resolve(JSON.parse(buf)); } catch { resolve(null); }
    });
    process.stdin.on('error', () => { clearTimeout(t); resolve(null); });
  });
}

function postTurn(payload) {
  return new Promise((resolve) => {
    const body = JSON.stringify(payload);
    const req = http.request({
      hostname: PORTER.host, port: PORTER.port,
      path: '/api/v1/intellect/transcript/turn',
      method: 'POST',
      headers: { 'Content-Type': 'application/json', 'Content-Length': Buffer.byteLength(body) },
      timeout: 2000,
    }, () => resolve());
    req.on('error', () => resolve());
    req.on('timeout', () => { req.destroy(); resolve(); });
    req.write(body); req.end();
  });
}

async function main() {
  const data = await readStdinJson();
  if (!data) process.exit(0);
  const sessionId = data.session_id;
  const transcriptPath = data.transcript_path;
  const cwd = data.cwd || null;
  if (!sessionId || !transcriptPath) process.exit(0);
  if (!fs.existsSync(transcriptPath)) process.exit(0);

  // 250ms breathing room for JSONL flush
  await new Promise((r) => setTimeout(r, 250));

  fs.mkdirSync(BOOKMARK_DIR, { recursive: true });
  const bookmarkPath = path.join(BOOKMARK_DIR, `${sessionId}.offset`);
  let offset = 0;
  try { offset = parseInt(fs.readFileSync(bookmarkPath, 'utf8'), 10) || 0; } catch {}

  const stat = fs.statSync(transcriptPath);
  if (stat.size <= offset) process.exit(0);

  const stream = fs.createReadStream(transcriptPath, { start: offset });
  const rl = readline.createInterface({ input: stream });
  for await (const line of rl) {
    if (!line.trim()) continue;
    let row;
    try { row = JSON.parse(line); } catch { continue; }
    if (row.type !== 'assistant') continue;
    if (row.isSidechain) continue;
    const blocks = row.message?.content || [];
    const text = blocks.filter(b => b.type === 'text').map(b => b.text).join('\n').trim();
    if (!text) continue;
    await postTurn({
      session_id: sessionId,
      cwd,
      role: 'assistant',
      content: text.slice(0, 32 * 1024),
      captured_at: row.timestamp || new Date().toISOString(),
    });
  }
  fs.writeFileSync(bookmarkPath, String(stat.size));
  process.exit(0);
}

main().catch(() => process.exit(0));
```

## Open Questions

1. **Should the capture endpoint also write user prompts that match the /silo prefix?**
   - What we know: `porter-user-prompt.js` blocks /silo commands from reaching the model, but the rawPrompt still exists.
   - What's unclear: do we capture `/silo software` as a user turn?
   - Recommendation: **No.** It's metadata, not conversation. Filter `rawPrompt.startsWith('/silo')` in the hook capture block and skip.

2. **What happens to bookmarks when /tmp is cleared (reboot)?**
   - What we know: /tmp is volatile.
   - What's unclear: after a reboot, the Stop hook will see offset=0 and re-parse the entire JSONL.
   - Recommendation: `ON CONFLICT (session_id, turn_index) DO NOTHING` makes re-parse safe. But re-parsing on reboot for every active session is wasteful. Bookmark `mtime` is older than the JSONL `mtime` is the only signal — accept the one-time re-parse cost; it's an edge case.

3. **Should `turn_index` be server-assigned or hook-assigned?**
   - What we know: hooks could maintain a per-session counter.
   - What's unclear: hooks don't share state across Stop and UserPromptSubmit invocations.
   - Recommendation: **Server-assigned.** Endpoint computes `MAX(turn_index)+1 WHERE session_id=$1` inside a transaction. Hooks never count.

4. **Does the `/silo none` 24h TTL play nicely with retention?**
   - What we know: `session_silo_overrides.set_at` rows are not cleaned up today (per 48.1 research, cleanup is inline-only).
   - What's unclear: do orphan override rows interfere with capture?
   - Recommendation: Capture endpoint already filters `set_at > NOW() - INTERVAL '24h'`. Old override rows are inert. The 48.2 retention workflow can optionally include a `DELETE FROM session_silo_overrides WHERE set_at < NOW() - INTERVAL '7 days'` step. Cheap.

5. **Should we capture `cwd` per turn?**
   - What we know: cwd can change within a session.
   - What's unclear: is the Dream Worker going to want this?
   - Recommendation: **Yes — add a nullable `cwd` column.** Tiny cost, future-proofs the Dream Worker's per-project sub-filtering. Cheap.

6. **What about content longer than 32KB?**
   - What we know: cap with truncation suffix.
   - What's unclear: long replies (data dumps, large code blocks) get cut.
   - Recommendation: 32KB cap with suffix `... [truncated: N chars]`. The Dream Worker pattern-matches on phrasing, not on every byte. If 48.3 needs more, it can re-read the JSONL by session_id.

## Risks and Unknowns

### Risk 1: Bookmark filename collision with reused session_id
Claude CLI session_ids are UUIDs — collision probability is negligible. But if a session_id were reused (e.g., a future feature that resumes sessions), the bookmark would be stale. Mitigation: bookmark filename also includes `mtime` of the transcript: `<session_id>-<mtime>.offset`. Minor refinement; not blocking.

### Risk 2: Stop hook output blocks Claude
Stop hooks have a `decision` field — `decision: "block"` forces Claude to continue. Our hook must NEVER write a JSON `decision` to stdout. Pattern: stdout is empty, `process.exit(0)`. Verified by mirroring `porter-session-end.js`.

### Risk 3: `transcript_path` flush race (Anthropic Issue #8564)
Documented bug: transcript may not contain the latest turn when Stop fires. Mitigation: 250ms sleep, plus a final tail-parse on SessionEnd (free — `porter-session-end.js` already exists and fires once per session). Implementation: extend `porter-session-end.js` to also call a `POST /transcript/flush?session_id=...` that runs the same tail-parse server-side using the bookmark. This is a 48.2-03 or 48.2-04 task.

### Risk 4: Workflow seeding order vs migration order
The retention workflow seed inside `migrate-transcripts-v1.ts` assumes the `workflows` table exists. Verified: `workflows` is created in `migrate-consolidated.ts:201` which runs before any v1 migrations. Safe. Document in the plan.

### Risk 5: 32KB content cap may truncate a long, single answer
If Moe gets a 50KB answer (a full research dump), the Dream Worker reads the truncated text and misses the bottom half. Mitigation: 32KB covers the 95th percentile of replies; the suffix `[truncated: N chars]` is a signal the worker can act on (e.g., re-fetch the JSONL by session_id if needed in 48.3). Not blocking for 48.2.

### Risk 6: Hooks are global — they fire for non-Porter projects too
The Stop hook will fire for sessions in `/home/lobster/projects/Baan Yin Dee/...`, `/home/lobster/projects/Funds/...`, etc. Those cwds won't match software silo cwd_markers, so `silo_id` will be NULL. They still get captured. Is that OK? Per spec: "Default if ambiguous: **none** (no silo directives injected — better miss than mix)." But the spec talks about INJECTION, not capture. Capturing with silo=NULL is fine for 48.3 (it filters `WHERE silo_id='software'`). Future silos that the BYD project would belong to will detect at synthesis time. **Decision: capture everything, silo-tag what's tag-able, NULL otherwise.** Document.

### Risk 7: Disk usage / table growth
Rough estimate: 50 turns/session × 5 sessions/day × 30 days × 4KB/turn average = ~30MB. Bounded by retention. Indexes add ~30%. Negligible for Porter's workload.

### Risk 8: Test isolation — smoke tests insert real rows
Smoke tests should use synthetic session_ids prefixed `smoke-48.2-<timestamp>` and clean up at the end (`DELETE FROM session_transcript_turns WHERE session_id LIKE 'smoke-48.2-%'`). Existing `smoke-48.1.sh` doesn't fully do this; recommend tighter cleanup in 48.2.

```sql
SELECT session_id, turn_index, role, content, captured_at
```
