## Decisions Made

- **Verbatim PII set with provenance comment.** The three regexes in `pii-scrub.ts` are copied byte-for-byte from `learner.ts` lines 193-205. An inline comment `// Verbatim from learner.ts lines 193-205 (extracted to shared helper 2026-05-11)` warns future readers not to "modernize" them without checking the source — guarantees zero-drift behaviour vs the pre-extraction state.
- **Kill switch fires BEFORE silo detection AND before any INSERT.** Two reasons: (a) minimum work done when `/silo none` is active, (b) `detectSilos` returns `[]` for BOTH "explicit `/silo none`" (an override row with `silo_id IS NULL`) AND "no silo detected" (no override row, no cwd marker match). Only a direct query against `session_silo_overrides` disambiguates. The capture endpoint must distinguish because case (a) means "do not capture" while case (b) means "capture with `silo_id=NULL`".
- **PII_PATTERNS imported only when needed.** `learner.ts` imports just `scrubPII` (not `PII_PATTERNS`) to keep `tsc --strict` clean under unused-import detection. The symbol remains exported from `pii-scrub.ts` for future consumers.
- **Server-assigned `turn_index` inside BEGIN/COMMIT.** `SELECT COALESCE(MAX(turn_index), -1) + 1 AS next_index FROM session_transcript_turns WHERE session_id=$1` followed by `INSERT ... ON CONFLICT DO NOTHING` in the same transaction. Hooks never count. The transaction bundles the read+write for one POST; the UNIQUE constraint catches dup races. No `FOR UPDATE` needed because `ON CONFLICT` handles the duplicate atomically.
- **Single retry on race, then drop.** Inside the same transaction: recompute `MAX(turn_index)+1` and retry the INSERT once. If still conflict, COMMIT, warn-log `concurrent_race`, return `{ok:true, inserted:false, reason:'concurrent_race'}`. This converts the drop-on-race from "always" (under heavy contention) to "exceptionally rare". Accepted tradeoff per `must_haves`: duplicates would corrupt the Dream Worker more than drops would.
- **32KB cap applied AFTER PII scrub.** Two reasons: (a) truncation can never reveal cleartext that was redacted, (b) the truncation suffix `... [truncated: N chars]` is a signal the Dream Worker can act on (re-fetch the JSONL by `session_id` if needed in 48.3).
- **Auth posture mirrors `/silo-command`: 127.0.0.1-only via server bind.** No auth middleware on this route group. Inline comment documents the parallel posture and references the same precedent.

## Single-Retry-on-Race Note

The drop-on-race tradeoff is the most subtle decision in this plan and is documented at length in the source. Quick reference:

- **Setup:** Under heavy concurrent Stop+UserPromptSubmit fires, two POSTs for the same `session_id` could `SELECT COALESCE(MAX(turn_index), -1) + 1` and get the same `next_index`. The UNIQUE constraint catches the second one via `ON CONFLICT DO NOTHING`.
- **Without retry:** The losing INSERT becomes a no-op, the turn is silently dropped, and the user-visible response is `{ok:true, inserted:false}` (no `reason`). Bad for the Dream Worker — it has fewer turns to mine.
- **With single retry (this implementation):** On `ON CONFLICT`, the orchestrator recomputes `MAX(turn_index)+1` inside the same transaction and retries the INSERT once. The race window shrinks to the gap between the second SELECT and the second INSERT, which is much narrower than the original race. Under realistic contention this almost always succeeds.
- **If retry also conflicts:** COMMIT, warn-log `concurrent_race`, return `{ok:true, inserted:false, reason:'concurrent_race'}`. Caller knows the drop happened and why. Accepted tradeoff per `must_haves`: duplicates would corrupt the Dream Worker more than rare drops would.
