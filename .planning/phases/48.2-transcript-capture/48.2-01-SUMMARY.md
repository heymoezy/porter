## Decisions Made

- **No FK from silo_id to silos.id.** Mirrors the explicit precedent set by `session_silo_overrides.silo_id` in 48.1-01 — silos can be renamed, disabled, or deleted in the future without orphaning transcript rows. The Dream Worker (48.3) already filters `WHERE silo_id = 'software'` so silo_id semantics are by-value, not by-reference.
- **Belt-and-braces workflow seeding.** Migration seeds the workflow row idempotently by name; `BUILTIN_WORKFLOWS` in workflow-engine.ts also seeds the same row idempotently by name on every startup via `seedBuiltinWorkflows()`. Either path alone is sufficient; both running is redundant and harmless. Defense in depth against DB rebuilds (where the migration runs but builtin-seed might not, or vice versa).
- **Drizzle siloCapturedIdx is ASC at the ORM layer; the raw SQL migration installs DESC at the DB layer.** Two index objects with the same name `transcript_turns_silo_captured_idx` would conflict — but only the raw SQL migration ever creates the index (the Drizzle binding is for `$inferSelect` type-safety only, not for `drizzle-kit push`). The DB-level DESC is what serves 48.3's query planner.
- **cwd column added.** Per RESEARCH Open Question #5: nullable TEXT, future-proofs per-project sub-filtering in 48.3 at near-zero cost.

## Issues Encountered

- **First `systemctl --user restart porter-fastify` did not visibly cycle the process.** The old PID 252036 continued serving traffic with the v6.12.0 build, but `journalctl` showed zero post-restart log entries from the migration. Root cause: when chained with a prior `pkill -9 -f "porter/backend"` (which never matched anything — the systemd cgroup is `porter-fastify.service`, not `porter/backend`), the `pkill` exit code 1 caused the bash chain (`&&`) to skip the restart. Resolution: ran `systemctl --user stop porter-fastify` then `systemctl --user start porter-fastify` separately. New PID 355504 came up cleanly, migration logs flowed, table appeared. Lesson: never chain `pkill` with `&&` when the unit-name pattern is uncertain; either use `|| true` or rely solely on systemctl stop/start.
- **No other issues.** TypeScript was clean throughout. No deviations needed.
