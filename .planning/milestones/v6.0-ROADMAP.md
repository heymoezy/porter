# Milestone v6.0: The Orchestration Platform

**Status:** ✅ SHIPPED 2026-05-13
**Phases:** 40-48.4 (12 phases — 8 sequential + 4 decimal inserted)
**Total Plans:** 41

## Overview

v6.0 transformed Porter from a chat router into a real multi-model orchestration platform. Task decomposition turns complex requests into parallel DAGs. Inter-agent messaging enables structured delegation through Porter as coordinator. Autonomous job queues let Porter self-assign work. Gateway capabilities drive intelligent dispatch. Project monitoring watches the world per project. Project substrate gives every project a structured container.

Mid-milestone the **Dream Silos series (Phases 48.1–48.4)** was inserted to deliver silo-scoped reinforcement learning: silo-aware memory injection, transcript capture, model-driven consolidation via the refine-don't-append doctrine, and a human review surface that closes the loop from CLI activity → directive refinement → next-session injection.

### Phase 42: Task Decomposition Engine

**Goal:** Complex requests are classified, broken into a dependency DAG, executed with parallelism where possible, and synthesized back into a single coherent response — the user sees one answer, Porter ran many agents
**Depends on:** Phase 40
**Requirements:** TDE-01, TDE-02, TDE-03, TDE-04, TDE-05
**Plans:** 4 plans

Plans:

- [x] 42-01-PLAN.md — Migration (task_nodes table) + TDE types + task classifier (TDE-01)
- [x] 42-02-PLAN.md — Task planner (LLM DAG generation + validation) + DAG executor (parallel dispatch) (TDE-02, TDE-03)
- [x] 42-03-PLAN.md — Task joiner (synthesis + failure handling) + decomposition engine entry point + chat.ts integration (TDE-04, TDE-05)
- [x] 42-04-PLAN.md — Admin REST endpoints for DAG inspection (TDE-05)

**Success Criteria:**

1. A single-step request dispatches directly without decomposition; a multi-step request produces a task_nodes DAG before any subtask is executed
2. Two independent subtasks in a DAG execute in parallel — both start before either completes
3. A subtask that fails triggers the joiner to either retry that node, replan it with a different approach, or surface an escalation to the user — not silently drop the result
4. Completing all DAG nodes triggers a synthesis step that combines subtask outputs into one coherent response returned to the caller
5. Admin can inspect any decomposed task: see the full DAG, each node's status, assigned gateway, output, and duration

**Success Criteria:**

The 48.x series was inserted mid-milestone as the silo-scoped reinforcement learning system Moe specified in `feedback_dream_silos`. It is the v6.0 milestone's most architecturally substantial work — 20 plans, 40 of 60 requirements, full closed-loop autonomy.

**Key Decisions:**

- **Bridge consolidated to claude_cli only** — v6.9.0 stripped to single backend; v6.14/v6.15 added isolation + raw passthrough. Cleanup pass v6.0.1 removed openclaw/ollama/codex_cli/gemini_cli residue from admin diagnostics (`prompt-pipeline.ts`, `gateway-versions.ts`). Routing engine + adapters confirmed clean.
- **Three-layer Refinement Doctrine** (Phase 48.3) — prompt instruction + `validateRefinementDoctrine()` worker check + DB `sort_order` column. Refine-before-append enforcement applies at write-time and display-time.
- **Raw passthrough by omission** (Phase 48.3 dream-worker) — instead of a `raw: true` flag, the worker omits `agentId`/`projectId`/`skillsUsed`/`directiveStats`/`dispatchStrategy` from RoutingContext. Memory V3 / skill selector / delegation doctrine only engage when those fields are set. Empirically verified on live Sonnet dispatch (all five columns NULL in `bridge_dispatch_log`).
- **Silo-scoped reinforcement learning** (Phase 48 series) — silos are first-class scope (`scope='silo'`, `scope_id='software'`), parallel to global/project/agent. Cross-silo bleed impossible by construction. Software silo seeded with 4 hand-curated `source_type='moe-direct'` directives sealed via DB trigger.
- **`/silo none` kill switch** — per-session privacy override row (`silo_id IS NULL`) makes transcript capture invisible by construction to the Dream Worker. Belt-and-braces with global config flag `intellect.transcriptCaptureEnabled`.
- **Hooks live outside Porter repo** (Phase 48.1-03, 48.2-03) — `porter-user-prompt.js` + `porter-stop.js` are global Claude Code hooks at `/home/lobster/.claude/hooks/`. Deliberately uncommitted; full contents reproduced in plan SUMMARYs for re-deployment.
- **Wave 0 smoke harness pattern** — Phases 48.1, 48.2, 48.3, 48.4 each shipped tests/smoke-48.X.sh BEFORE implementation plans. Every downstream task got a single-shot `<verify>` command. Smoke harnesses self-degrade with backend/feature gates (warn-skip exit 0) so they run cleanly through every wave.

**Issues Surfaced + Resolved (dormant infrastructure bugs flushed by Dream Silos):**

1. **Bridge circuit breaker `action` no-op since opossum 9 adoption** (fixed 48.3-04, 48.3-05) — chat path bypasses `breaker.fire` via `dispatchStream`, so the bug was dormant repo-wide. Dream-worker was the first non-streaming consumer to `await` the breaker result. Fixed with `runThunk = async (fn) => fn()`; timeout bumped 30s → 180s.
2. **Frontend SSE never received named events since v3.0** (fixed 48.4-03 RVS-11) — `useAdminSSE` used `es.onmessage`, but `sse-hub.ts` writes `event: <topic>\ndata: <json>` (named events fire ONLY on `addEventListener`). All `bridge:*`, `decomposition:*`, `agent:*`, `profile:updated` events silently dropped client-side for ~2 months. Refactor to 20 explicit `addEventListener` calls enables 14+ named topics including 3 new dreams topics. Every existing live-update admin surface (bridge dispatch ticker, capacity bars, gateway cards, msg-bus feed, profile updates) starts working for the first time.
3. **dispatchDream crashed backend on undefined Bridge result** (fixed 48.3-05) — null-guard added.
4. **Worker failure path lost dispatch_id** (fixed 48.3-05) — hoisted to outer scope + COALESCE in catch.
5. **Test environment defects** (fixed v6.0.1 cleanup pass) — Chromium first-time install missing in test env, systemctl restart race needing pkill -9, stale `moe@themozaic.com` credentials in `tests/setup-auth.js` + 8 other test files, stale `#uname/#pw/.login-btn` selectors (admin shell uses `<aside><nav>` + `#email/#password` since v4.x refresh), sonner toast Playwright selector mismatch (`[data-sonner-toast]` not `[data-sonner-toaster]`).

**Technical Debt Deferred to v6.1+:**
