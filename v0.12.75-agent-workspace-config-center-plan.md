# Porter v0.12.75 Draft — Agent Workspace Config Center Plan

Date: 2026-02-26  
Status: Planning approved (implement after review)  
Owner: Lobster

---

## 1) Why this exists

Current Assistants cards are good for status + quick actions, but not for deep agent maintenance.
We need a dedicated **Configure** workspace that lets operators manage agent behavior and key markdown config files directly from Porter in a safe, guided way.

---

## 2) Product concept

Add a **Configure** button per assistant card.
Clicking it opens a full-width right-side workspace (replacing normal module content) designed for agent administration.

### Workspace structure

- **Left column:** Agent config file navigator (allowlisted files only)
- **Center column:** Editor + diff/preview + save
- **Right column:** Control panels (friendly forms + guidance)

---

## 3) Scope (Phase A MVP)

### A. Entry
- Add `Configure` button to each assistant card.
- Opens `Agent Workspace` view with selected assistant context.

### B. File navigator (allowlisted)
Initial file list:
- `SOUL.md`
- `USER.md`
- `AGENTS.md`
- `TOOLS.md`
- `MEMORY.md`
- `HEARTBEAT.md`
- `memory/*.md` (read/write with date grouping)

### C. Editor behavior
- View/edit markdown text
- Save changes
- Unsaved-change warning on navigation
- Basic preview toggle

### D. Right control panel (read/write where applicable)
- Identity & persona summary
- Behavior/risk profile
- Memory mode + visibility summary
- Usage threshold summary
- Safety notices

### E. Safety constraints
- Only allowlisted files editable from this workspace
- Confirmation required for sensitive files (`SOUL.md`, `MEMORY.md`)
- Basic audit entry for each save (file + assistant + timestamp)

---

## 4) UX requirements

1. Must feel calm and non-technical by default.
2. Technical/internal details behind collapsible “Advanced”.
3. Every control needs plain-language helper text.
4. Clear ownership context at top:
   - assistant name
   - role/type
   - last active

---

## 5) Non-goals in Phase A

- No arbitrary filesystem browser in this view
- No shell/command execution integration
- No multi-agent bulk editing yet
- No full VCS UI inside Porter (just basic save + audit)

---

## 6) Phase B (next)

- Structured forms that write safely to markdown sections
- Inline file diff before save
- Restore previous version per file
- Per-assistant policy templates

## 7) Phase C (later)

- Multi-agent compare mode
- Policy inheritance maps
- Memory strata visual map and cross-agent linkage controls

---

## 8) Acceptance criteria (Phase A)

- Configure button exists on every assistant card
- Workspace opens with selected assistant context
- Allowlisted files are editable and save correctly
- Sensitive file confirmation works
- Saves are logged in runtime audit
- No regressions to existing Assistants list actions

---

## 9) Versioning intent

This plan targets **v0.12.75** as next incremental release.
If we bundle this with broader IA/module changes (new nav model + workspace shell + policy form migration), we can promote that bundle to **v0.13.0**.

---

## 10) Execution sequence

1. Lock IA and file allowlist.
2. Implement Configure button + workspace shell.
3. Add editor and save flow for allowlisted files.
4. Add right-side control panel summaries.
5. Add sensitive-file confirm + audit save logging.
6. QA + changelog + release notes.
