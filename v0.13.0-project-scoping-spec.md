# Porter v0.13.0 — Project Scoping Spec

Date: 2026-02-26
Status: Draft for implementation sequencing
Owner: Lobster

---

## 1) Problem

Global markdown/state files are being overloaded with project-specific context (e.g., DFSA/MAS).
This causes memory clutter, cross-context leakage, and weak operator clarity.

---

## 2) Goal

Introduce **Projects** as first-class scope boundaries so memory/settings/agent overrides are isolated per project while preserving global defaults.

---

## 3) Scope model (locked)

### Layer 1 — Global defaults (shared)
- Core identity, safety, system-wide defaults
- Minimal and durable only

### Layer 2 — Project scope (isolated)
- Project memory
- Project settings/policy defaults
- Project-specific files and notes

### Layer 3 — Agent override (within project)
- Per-agent behavior/model/tool overrides for that project only

Resolution order at runtime:
1. Agent override (project)
2. Project setting/memory
3. Global default

---

## 4) Filesystem structure

`~/.openclaw/workspace/projects/<project-id>/`

- `PROJECT.md` — project mission, owner, status
- `MEMORY.md` — project-specific long-term memory
- `settings.json` — project defaults
- `agents/<agent-id>/overrides.md` — per-agent project overrides
- `notes/` — freeform docs
- `artifacts/` — outputs/deliverables

Global remains:
- `workspace/MEMORY.md` (durable cross-project profile only)

---

## 5) UI/UX requirements

### A. Project switcher
- Visible in top-level UI
- Current project always explicit

### B. Configure workspace awareness
- File navigator grouped by (visually separated sections):
  - **Global Shared Files**
  - **Project Files (Current Project)**
  - **Agent-Specific Files (Current Agent + Project)**
- Source badges on every file row: `Global`, `Project`, `Agent Override`
- Distinct color/icon treatment per scope so edit impact is obvious at a glance.
- Sticky scope legend in the workspace header: “You are editing: Global / Project / Agent”.
- Save confirmation copy must echo scope explicitly, e.g.:
  - “Save to Global shared file?”
  - “Save to Project file?”
  - “Save to Agent-specific override?”

### C. Guardrails
- Save operations write to current scope only unless user explicitly promotes
- Scope guardrail: block silent scope changes (no implicit global writes from agent/project views)
- Provide actions:
  - `Promote to global`
  - `Reset to inherited`
  - `Copy to agent override`

---

## 6) Memory policy

- Move project/case-specific context out of global MEMORY.md
- Keep global MEMORY.md concise and stable
- Project memory lives in `projects/<id>/MEMORY.md`

---

## 7) Implementation phases

## Phase A (v0.13.0-alpha)
- Create/list/select project
- Persist active project in preferences
- Add project filesystem scaffold
- Wire Configure workspace to show project files

## Phase B (v0.13.0-beta)
- Scoped memory read/write resolution (project-aware)
- Source badges + inheritance indicators
- Promote/reset controls

## Phase C (v0.13.0)
- Agent override editing in project scope
- Audit logs include project_id
- Migration utility for existing mixed memory content

---

## 8) Migration plan

1. Create default project: `default`
2. Keep current global files intact
3. Add migration helper to move scoped content (e.g., DFSA/MAS) into project memory
4. Leave pointers in global memory where necessary

---

## 9) Acceptance criteria

- User can switch project and see isolated project memory/settings
- Editing in Configure writes to selected scope correctly
- No cross-project bleed for memory/settings unless explicitly promoted
- Runtime resolution follows override order deterministically
- Audit entries include project context

---

## 10) Non-goals (this cycle)

- Full multi-tenant RBAC
- Cross-project automatic sync
- Complex branch/merge semantics for markdown

---

## 11) Release intent

This spec defines the milestone boundary for **v0.13.0**.
Patch releases in v0.12.x should only harden current UI/flows.
